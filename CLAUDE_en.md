# Scaling Up

A 2D management simulation game about running an AI model company, with one turn per week, built with **Godot 4.4**.

> English reference translation of `CLAUDE.md`. The Chinese `CLAUDE.md` remains the source of truth for project-agent instructions.

## Language Conventions

- `CLAUDE.md` and all design documents under `design/` should remain **Chinese** to save context.
- Code, commit messages, and Godot-generated files should remain English.

## Conventions

- Organize by scene first: each feature owns its own folder, including scenes and resources used only by that feature.
- Cross-system communication goes through `EventBus` signals. Do not hold direct node references across systems.
- Persistent game state lives in the `GameState` singleton; turn advancement is centralized through `TurnManager`.
- Prefer `.tres` resource files and avoid `.res`.
- When adding important files, update the directory structure in `CLAUDE.md` and the relevant `index.md` files.
- Code logic should include logging so debugging is easier.
- Code implementation is the source of truth. Design documents are for understanding and high-level system design.
- Game data, including numeric tables, templates, and presets, is defined by `.tres` files under `resources/data/`. Values in design documents are reference-only.
- **Codename rule** (gameplay design §0bis / common enum table §16): all GPU, model, architecture, and NPC company names must use codenames only (plant / animal families). Real brands such as NVIDIA, AMD, GPT, Llama, A100, H100, etc. must not appear in code, `.tres`, or UI copy. Design documents may include "approx. X" comments for comparison only.
- **Real units**: compute, training, and inference values should use real units such as `FLOPs`, `TFLOPs`, `B tokens`, and `tokens/s`, not abstract names like `compute_unit` or `throughput`. The numeric system should stay close to the real world where practical.

## Workflow: Test-Driven Development (TDD)

Any functional change must follow this order. Do not skip steps:

1. **Update the design document** — update or add the corresponding document under `design/`, describing the high-level intent and rules. Large numeric tables belong in data files.
2. **Write or update tests** — add or modify tests under `tests/` to cover the new behavior. At this point the tests should fail.
3. **Write code / resources** — implement the behavior in `scripts/`, `scenes/`, or `resources/` so the tests from the previous step pass.
4. **Run unit tests** — all tests for the individual system or script should pass.
5. **Run system integration tests** — end-to-end tests across systems, turn advancement, and `EventBus` should pass.
6. UI changes require screenshot-based verification and testing.

If a later step reveals a problem in the design or tests, go back to the relevant earlier step and fix it there instead of patching around it later.

**Internationalization (i18n)**: any new user-facing text must go through `tr()`. Do not hard-code Chinese strings. For how to add or modify translations (UI copy in `strings.csv`, game content in `.tres` -> `content.csv`, generation commands, and test constraints), see `design/国际化设计.md §8`.
