# 数据集系统设计 (DatasetSystem)

> **目的**: 管理预训练 / 后训练数据资产。提供三条获取渠道 (开源 / 商业购买 / 自采集), 按真实世界时间轴释放, 区分两种 kind 的消费方式与产出公式。
> **读者**: 数据面板 UI、TaskSystem (`pretrain` / `posttrain` / `data_collection` 子类型)、ResearchSystem (训练绑定 dataset)。

---

## 0. 设计原则

1. **两个象限**: `kind ∈ {pretrain, posttrain}` × `source ∈ {open_source, purchased, collected}`。pretrain = "喂海量原始 token", posttrain = "喂少量高质量 instruction/RLHF"; 训练消费与产出公式完全不同。
2. **资源类型显式拆分**: `PretrainDatasetTemplate` 与 `PosttrainDatasetTemplate` 两个独立 `Resource` 子类; `Dataset` 实例上 `kind` 字段区分。TaskSystem 按 `kind` 校验。
3. **按时间轴释放**: 模板带 `released_at_week`; DatasetSystem 只把 `released_at_week ≤ GameState.turn` 的模板在市场列表露出。每半年 (= 26 周) 推一波解锁信号。
4. **规模随年份指数级放大**: 对照现实 (2017 ~3-10B → 2024 FineWeb 15T tokens)。
5. **价格按"每 token / 每 example"反推**: pretrain token 廉价 (清洗成本主导), posttrain example 昂贵 (标注成本主导)。
6. **化名规范**: 数据集 id 用化名/通用词 (`webgiant_corpus_v1` 替代 CommonCrawl, `coral_code_v1` 替代 The Stack); 注释里写 "≈ 现实对照"。

## 1. 数据切片

owner = `DatasetSystem`。

`GameState.datasets: Array[Dataset]`。

### Dataset 字段 (非显而易见的语义)

完整字段以 `dataset.gd` 为准, 本节只记不直观的点:

- `size` 单位都是 B tokens (posttrain 典型 < 0.5)。
- `coverage_tags` 在 pretrain 时影响 evaluate 各能力轴占比, 在 posttrain 时仅作 UI 分类。
- `business_analysis` 是隐藏的 pretrain 内部标签: UI / 自采选项不展示, 但 evaluate 时会轻微压低 `code` / `reasoning` / `agent` 三轴表现。
- `target_capability` 仅 posttrain 用 (`general / code / reasoning / multimodal / agent`)。
- `source` (open_source / purchased / collected) v9 后降为审计/装饰字段, pretrain 配比公式不再读它。

### Modality (规划中, 尚未落地数据)

`Dataset` 资源类已带 `modality` 字段 (单选 text/image/audio/video/code, 不存在 "multimodal" 模态 — 多模态训练靠玩家组合多份单模态集), `dataset.gd` 与 `dataset_system.gd` 也已实现读取与 `text` 兜底逻辑。但**目前没有任何静态 `.tres` 设置该字段** —— 包括 `image_caption_5b` / `image_corpus_v1` 在内, 全部数据集运行时都解析为 `text`。modality 的能力轴校验 (pretrain 子类型校验每份数据集 modality ∈ 模型 `input_modalities` ∪ `&"text"`, 失败码 `dataset_modality_mismatch`; posttrain 不校验) 已在代码里, 但在 .tres 真正写入 modality 之前不会触发约束。

派生量 `Dataset.is_locked()` / `Dataset.pretrain_quality_multiplier()` 见 `dataset.gd`。

### Pretrain 数据配比 (v9, 2026-05 重做)

旧的 source min 乘子 (open ×0.9 / collected ×1.0 / purchased ×1.05) 已**废弃** — 它有两个问题: (a) 一份 open 拖死全局, mix 反直觉; (b) `ds.quality` 字段 (0..1) 在 pretrain 路径完全没用, 高质量小集 (Phi textbook 0.94) 跟 CommonCrawl (0.30) 等价。

v9 改为 **token × quality 加权**: 数据质量因子按 token 量加权平均 quality, 能力轴占比按"该轴 token×quality 份额"经 log 递减映射。source 字段降为审计/装饰 (UI 仍展示, 公式不再读)。完整公式与常量见 `平衡参数.md §DatasetSystem 公式` 和 `任务系统设计.md §6.7`。

**关键性质**:
- token 加权 → 加 8B math 到 15000B FineWeb 上不再变成"半个 reasoning 轴", 必须真砸 token 才能换专精分。
- quality 加权 → 选 Phi textbook (q=0.94) 1B 跟 CommonCrawl (q=0.30) 3B 在分子里近似等价; 高质量数据有真实回报。
- log 递减 → 5-20% 专精占比就能拿到大部分 ratio, 不必 100% 专精, 配比探索有甜区。
- 通用底盘覆盖 → `web/books/encyclopedia/edu/news/arxiv/multilingual/textbook/chat` 等通用知识 token×quality share 不足时, evaluate 的 `raw` 整体打折; 约 45% 覆盖后不惩罚。现实校准参考: 前沿通用模型仍保留大量 general knowledge 数据, code/reasoning 是高杠杆专精而不是全部底盘。
- 隐藏商业分析惩罚 → 带 `business_analysis` 的预训练数据按 token×quality share 计算商业话术污染度, 只对 `code` / `reasoning` / `agent` 三轴做小倍率折扣。黑色幽默意图: 公司策略 PPT、竞品分析和 TAM/SAM/SOM 表格会让模型更会开会, 但稍微不那么会写代码、推理和执行计划。`general` / `multimodal` 不受影响。
- `Dataset.pretrain_quality_multiplier()` 保留 API 兼容, 一律返 1.0; 不再出现在 `_scaling_law` (训练时长只看 token) / `_compute_capability_measured` (走新公式)。

### Posttrain capability 公式

ResearchSystem `posttrain_apply` 收到 `{model_id, dataset_ids[]}` 后, **对每个 dataset 各自结算**: target_capability 同轴累加 gain, 其余各轴扣 forget。

**公式形选择 (实现见 `research_system.gd` + `平衡参数.md §Posttrain 能力增减系数`)**: gain 用 `quality²` 让低质量塌缩快, 用 `log2` 让叠数据集只换线性回报; forget 随 quality 线性, 使 q < ~0.6 净亏。各 dataset 独立累加。

### Posttrain 档位 (时长 / GPU 需求)

posttrain task 不走 scaling law, 按 `base_model.size_params` 查表 (`POSTTRAIN_TIER_TABLE`, 详见 `平衡参数.md`)。GPU 数是**下限校验**, 算力高于阈值不缩短时间。

## 2. 静态模板 (resources/data/datasets/)

两个独立 Resource 子类, 字段以脚本为准:

- `PretrainDatasetTemplate` — `pretrain/{open_source,purchased}/*.tres`。
- `PosttrainDatasetTemplate` — `posttrain/{open_source,purchased}/*.tres`, 比 pretrain 多一个必填 `target_capability`。

`released_at_week` 按 2017-06-12 锚定周数 (见 `游戏基础架构 §3.4.1`)。`modality` 字段虽在资源类上, 但当前没有 .tres 写入 (见 §1)。

DatasetSystem 启动时分别扫描两个目录入两个独立缓存 (`_pretrain_cache` / `_posttrain_cache`); 目录扫空时走代码内 fallback (`bookcorpus_v1` / `alpaca_52k_v1`)。

## 3. 命令与信号

详见 `命令总线表.md §DatasetSystem` 与 `事件总线信号表.md §DatasetSystem`。要点:

### 命令

- `dataset.acquire_open {template_id}` / `dataset.purchase {template_id}`: 开源免费 / 商业扣 `template.price` 走 `economy.spend`; 错误码含 `unknown_template` / `not_released_yet` / `already_owned`。`kind` 由 DatasetSystem 内部按 id 查模板字典区分。
- `dataset.add {kind, size, quality, coverage_tags, target_capability?, modality?, source, display_name?}`: 仅 TaskSystem 在 `data_collection` 完成时调; UI 不直接调。`modality` (v7 PR-G) 默认 `&"text"`, 由 `data_collection` task payload 透传。
- `dataset.delete {dataset_id}`: 锁定中拒。
- `dataset.lock` / `dataset.release`: TaskSystem 训练独占。
- `dataset.list_market {kind?, source?}`: 返回 `released_at_week ≤ turn` 且未拥有的模板缩略列表。

### 信号

`dataset_added(dataset_id, source)`, `dataset_removed(dataset_id)`, `dataset_locked(dataset_id, task_id)`, `dataset_released(dataset_id)`, `dataset_market_updated(kind)`。

### 读档 ID 一致性 (save_loaded)

自采数据集 ID 走会话内计数器 `_next_dataset_seq` (不入存档); DatasetSystem 监听 `save_loaded` 恢复计数器并修旧档重复 ID, 与 InfraSystem 同病同治 (见 MEMORY「会话计数器不入存档」)。

## 4. 对外依赖

- **读切片**: 无。
- **调命令**: `economy.spend` (商业购买; 采集成本由 TaskSystem 自扣)。
- **订阅信号**: `phase_started(action)` — `turn > 0 and turn % 26 == 0` 时 emit `dataset_market_updated` (pretrain / posttrain 各一次), UI 用作"新到货"提醒。模板真实露出由 `dataset.list_market` 实时按 `released_at_week` 过滤, 不依赖信号。

## 5. DatasetCollectionDialog (自采集)

数据 tab 「自行采集」入口对话框。玩家选 `kind` (pretrain / posttrain), 设 `size` (滑块, 步进/范围 见下表), **选 `modality` (v7 PR-G, text/image/audio/video/code 单选; 默认 text)**, posttrain 还要选 `target_capability`, 可选 lead, 并可勾选「监控员工日常工作数据」。点启动 → `task.start(template_id = data_collection_dynamic)`。modality 与后训练监控选项透传到 task payload; 完成时由 TaskSystem 结算后再调用 `dataset.add`。

`base_cost` 由 `_resolve_base_cost(template, p)` 在 `task.start` 时动态计算 (而非固定在 .tres), `task.preview` 同样调用此函数, 让预览与启动一致。

### 成本与时长 (常量见代码 `TaskSystem._data_collection_pricing`)

自采成本 / 周成本 / 时长按 `kind` 与 `size` 由 `_resolve_base_cost` 动态算 (pretrain token 量大但单价低、基础 quality 0.55; posttrain example 贵、quality 由玩家选档)。pretrain 的 `coverage_tags` 由玩家在对话框单选 1-2 个 (默认 `[web]`), posttrain 自动取 `[target_capability, instruction]`。具体范围 / 步进 / 系数见代码常量。

**Posttrain 自采人力定价 (2026-05 rev, 连续曲线)**: 后训练数据现实里成本由**标注人力**主导 (不是 token 量), 质量越高越贵 (众包 → 领域标注 → 资深专家 → PhD)。`examples ≈ size_B × 1e6` (1000 tokens/example), `base_cost = 30k + examples × 每条单价(rate)`。rate 按**有效质量**取, 是一条**连续分段曲线** (不是离散分档), 锚点 ($/example):

| 锚点有效质量 | 0.65 | 0.80 | 0.85 | 0.90 | 0.95 |
|---|---:|---:|---:|---:|---:|
| rate ($/example) | 1 | 10 | 50 | 60 | 800 |

- 锚点之间线性插值, `> 0.95` (只有 lead 加成能顶到) 走平缓指数尾 (`800 × e^{6(q-0.95)}`, 上限 5000)。**连续** ⇒ 选更高档 (或换更强 lead) 永远更贵, 不会像旧的 4 个离散桶那样「lead 把质量顶过 0.90 后, 资深专家 / PhD 同价」(旧 bug: 选专家级没变贵)。常量见 `TaskSystem._POSTTRAIN_RATE_ANCHORS`。
- **有效质量 (用于定价) = `target_quality`(玩家选档) + data_scientist lead 加成, 不 clamp**; 产出数据集 quality 仍 clamp 到 [0,1] (`_data_collection_quality`)。定价用有效质量 → 防止「选 T1 + 强 lead」用低价拿高质数据。员工日常工作监控属于内部行为数据微调项, 不进入标注人力价格曲线。
- 周成本固定 8k, 时长 `min(20, max(3, ceil(size_B × 0.6)))` 周, 两者不随质量变。
- 对话框 posttrain 多一个**质量档下拉** (T1–T4 → target_quality 0.65/0.80/0.90/0.95), pretrain 无此选项 (爬取数据无人力分档, 恒 0.55)。
- **监控员工日常工作数据 (posttrain-only)**: payload `monitor_employee_work_data = true` 时, 完成产出的 posttrain 数据集 quality 额外 +0.03, clamp 到 [0,1]。这是小幅内部真实任务轨迹加成, 不改变 size / duration / weekly_cost / base_cost; 当前不接员工满意度或事件风险, 后续可扩展。

**员工占用 (v8, 2026-05, 随 size 缩放)**: 自采数据集**硬性要求并锁定**数据工程师 (`data_eng`), 数量随数据集 size 缩放, clamp 到 **[2, 8]** (与营销活动同一 2..8 规则)。`task_system` 是权威方 — 按 size 自算并锁定该数量 (`data_collection_staff_count(kind, size)`), payload 不必指定。kind 分别计 (pretrain 爬取语料 1..100000B → 每 4000B +1 人; posttrain 标注 0.01..0.5B → 每 0.08B +1 人), 典型小数据集只需 2 人, 超大语料才吃满 8。没有足够空闲 `data_eng` → `missing_staff`, 不能启动; 完成 / 取消时释放。`dynamic.tres` 的 `input_schema.needs_staff` 只作"需要 data_eng"标记 (`data_collection_default` 等无此声明的模板不占员工)。对话框展示「需占用 N 名 (空闲 M)」并在不足时禁用。

**Pretrain coverage_tags 单选 (v9, 2026-05)**: 旧版自采固定写 `[web, code, reasoning, books, general]` 五标签全覆盖, 配合 v8 公式按"条数命中比"导致一份自采 = 所有轴 ratio 顶到 1.0, 与购买/混合开源严重失衡。v9 把 tag 选择交给玩家 (在 `DatasetCollectionDialog` 多选控件, 上限 2 个; 不选则默认 `[web]`)。配合 v9 公式 (token×quality 加权 + log 递减), 自采本身 quality 偏低 (0.55) → 想堆专精轴还是得砸 token 量。

**Lead 加成**: data_scientist 的 `data_quality_add × ability/100`, clamped 0..0.20。

**设计意图**:
- pretrain 自采 = 清洗 web 数据 → 便宜 + 玩家自选专精方向, 但 quality 偏低 (0.55) 难冲击 SOTA。
- posttrain 自采 = 标注 SFT/RLHF → 按质量分 4 档 (T1–T4), 高质量 (>0.90, PhD 档) 极贵, 与购买商业集 (q ≤ 0.90) 形成「便利溢价 vs 自建血汗」的取舍; 监控员工日常工作数据给少量质量增益, 强化「公司内部真实工作流」的数据价值。
- 时长 / 周成本已在 2026-05 平衡过一轮 (玩家反馈采集等待过长 → 系数 ÷10); 成本分档于 2026-05 重做 (见上「人力分档」)。

## 5bis. DatasetView 筛选 (数据 tab)

数据 tab 在 `kind` 切换 (预训练 / 后训练) 之外, 顶部带**两条 FilterBar pill 筛选条**, 取交集后同时作用于「市场模板」与「我的数据集」两个网格 (与基建 tab 的双 FilterBar 同构):

- **来源**: 全部 / 开源 (`open_source`) / 商业 (`purchased`) / 自采 (`collected`)。市场只有前两者, 自采只出现在「我的」。
- **规模**: 全部 / ≤10B / ≤1000B / >1000B (B tokens, 阈值见 `dataset_view.gd._size_bucket_of`)。

pill 带计数 (市场 + 我的全部卡按各维度独立计数)。0 卡可见时显示空提示。筛选状态是 view 内部态, 不回写 GameState。
隐藏标签不进筛选、卡片「标签」字段或自采标签选择; 玩家只看到最终能力分的自然结果。

## 6. 时间轴释放

模板 `released_at_week` 锚到真实世界数据集发布日期, 大致每 ~26 周 (半年) 推一波解锁。规模随年份指数级放大 (对照现实 2017 ~3-10B → 2024 FineWeb 15T tokens)。

每个数据集具体的 `released_at_week` 与数值以各 `.tres` 文件为准 (`resources/data/datasets/{pretrain,posttrain}/{open_source,purchased}/*.tres`), 本文档不镜像清单。

> 价格估算依据 (设计指引, 不在 .tres 强制): pretrain 商业集 ≈ `size_B × $10K × quality_multiplier`; posttrain 商业集 ≈ `examples × 单价` (examples ≈ size_B × 1e6, 1000 tokens/example)。posttrain 商业集总价对标**自采同质量档 × ~2.4-2.7** (购买 = 即时 + 不占 lead + 质量有保障的便利溢价, 自采要等 3+ 周且质量受 lead 影响)。注意自采按 size 线性计价, 所以"小而贵"的商业集会与自采价崩开 (agent_traces 原 0.015B@10M = 自采 10× → 已把 size 加大到 0.06B 拉回 ~2.7×); 高买价必须配相称体量。对照现实 (Llama 2 偏好数据 100 万+ 对比 ≈ $8–25M, Surge/Scale 年营收 $1B 级), 这些小体量商业集是「单批高质数据」而非「连续数据项目」。
