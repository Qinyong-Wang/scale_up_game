# 事件系统设计 (EventSystem)

> **目的**: 按概率/条件抽取剧情或随机事件, 弹窗给玩家做选择, 选完通过 CommandBus 派发后果。事件是把**长期玩法**变得有戏剧性、趣味与无厘头的主要手段。
> **读者**: 事件弹窗 UI、所有可能被事件影响的 system (经济/市场/科技/数据中心 等)。
>
> v10 (2026-05) 大改: 新增 `routine` 常规事件类别 (现为每 8 周强制弹一张; v11 从 4 周降频) + 7 个新前置条件字段 + 比例化 effect (按当前现金/订阅数百分比, 解决 6 个数量级跨度下固定金额失衡问题)。
>
> v11 (2026-05) 调整: ① 降频 —— routine 强制间隔 4→8 周、随机概率 0.10→0.07, 整体约每 6-8 周一张; ② 新增 `max_triggers` 字段 + `event_trigger_counts` 计数切片, 让重大 drama 事件"一辈子只来一次"; ③ 新增一批 **真两难** drama 卡 (AI 历史争议 + 硅谷梗), 约定: 这批卡**两个选项都必须挂真生效的 effect** (不再有"一支为空"), 见 §4.2 与 `事件库.md`。
>
> v12 (2026-05) 新增 **历史档案事件**: 取材自 2017 游戏开局后的 AI 发展史无选择教育卡, 使用 `flavor` 类; 玩家只点"知道了", 不产生资源奖励。历史档案 `weight=0` 不进随机池, 由固定顺序的 `HISTORICAL_EVENTS` 按 `min_turn` 锚点触发, 见 §4.4bis 与 `事件库.md §1ter`。
>
> v15 (2026-05) 降低重复打扰: `dc_meltdown` 首年保护 (`min_turn=52`) 且全局最多触发 2 次; `routine_open_source_pr` 全局最多触发 2 次。
>
> v16 (2026-05) 频率审计修复: Monte Carlo 显示成熟局约每 4.7 周一张, 高于"每 6-8 周"目标。调整为 `ROUTINE_INTERVAL=12`、`EVENT_TRIGGER_PROB_PER_WEEK=0.03`; `cooldown_months` 明确按月换算为周 (`* TurnManager.WEEKS_PER_MONTH`); 低门槛 routine 池新增 4 张可重复小事, 避免无产品/无员工局只剩 `routine_lawsuit_spam` 反复刷屏。

## 1. 数据切片

owner = `EventSystem`。

| 字段 | 类型 | 语义 |
|---|---|---|
| `pending_events` | `Array[EventInstance]` | 待玩家处理队列 (FIFO) |
| `event_history` | `Array[EventInstance]` | 已处理记录, 上限 `HISTORY_LIMIT` = 50 |
| `event_cooldowns` | `Dict[template_id, int]` | 抽中后多久不再抽 |
| `event_trigger_counts` | `Dict[template_id, int]` | 每张卡历史累计触发次数; 配合 `EventCard.max_triggers` 实现"最多触发 N 次" (见 §4.7), 入存档 |
| `last_routine_turn` | `int` | 上次弹出 routine 事件的 turn; 每满 `ROUTINE_INTERVAL`(=12) 周触发一张, reset 时归 0 |

### EventCard / EventOption / EventEffect / EventInstance 字段

详见 `scripts/resources/event_*.gd`。要点:

- `EventCard.{id, category, title, body, options, passive_effects, weight, cooldown_months, max_triggers, ...触发条件}`
  - `max_triggers: int = 0` (v11) — 全局最多触发次数, `0` = 不限。重大 drama 卡设为 `1` (一辈子只来一次), 比硬塞 `cooldown_months=9999` 更语义化。计数存 `GameState.event_trigger_counts`, 见 §4.7。
  - `passive_effects: Array[EventEffect] = []` (v12) — 无选择 `flavor` 卡的可选被动后果。`event.dismiss_flavor` 时按顺序结算, 并把 `applied_effects` 透传到 `event_resolved`。普通有选项事件仍只用 `EventOption.effects`。**历史档案事件默认不填 passive effects**, 避免"读新闻发钱"的出戏感。
- `EventCard.category` ∈ `opportunity | crisis | flavor | routine`:
  - `opportunity` — 拿好处 / 拒绝, 不会负面。
  - `crisis` — 通常花钱减伤, 拒绝可能更糟。
  - `flavor` — 叙事/新闻/历史档案, **无 options**, 点"知道了"即关; 可选 `passive_effects` 表示系统自动结算的微量影响。旧时代锚点卡不得再把废弃 `npc_*` effect 藏在 option 里, 否则 UI 不会展示也不会结算, 资源语义会和系统契约脱节。
  - `routine` — **每 12 周强制弹一张** (见 §4.5), 必须选 option, 内容是公司经营日常的无厘头小决策。
- **触发条件 (全部 AND)**: 每张卡都至少满足一个条件 —— 事件要贴合当前局势, 不能凭空乱弹。
  - `min_turn` / `requires_unlocks` / `requires_cash_min` / `requires_revenue_min` / `requires_rank_max` (旧)
  - `requires_datacenter: bool` — 至少拥有 1 个数据中心
  - `requires_product: bool` — 至少有 1 个产品
  - `requires_published_model: bool` — 至少有 1 个 `status==published` 的模型
  - `requires_lead_min: int` — **非 founder** lead (`is_player_scientist=false`) 数量下限。把创始人本人排除掉, 否则单人时期 `lead_poached`("挖你最依赖的那位 lead")会荒诞触发
  - 叙事预设了某项"已研究出的能力"的卡, 必须 gate 对应 `requires_unlocks` —— 例如"编程助手 agent 半夜自己开 GPU"(`rogue_agent`)须 gate `application:fox_code_specialist`(Coding Agent), 否则没研究出智能体也会弹
  - `requires_staff_min: int` — 在岗员工数 (`staff_pool` 各 role 求和) 下限
  - `requires_dataset_min: int` — `datasets` 数量下限
  - `requires_paid_users_min: int` — `paid_users` 下限
- `EventOption.{id, label, effects: Array[EventEffect]}`
- `EventEffect.{kind, params}` — 见 §4.2 effect 分发表

> 模板放 `resources/data/events/*.tres`。具体卡牌内容见 `事件库.md`。

## 2. 命令与信号

详见 `命令总线表.md §EventSystem` / `事件总线信号表.md §EventSystem`。

### 命令

- `event.choose_option {event_id, option_id}` → `{ok, applied_effects: Array}`; 错误 `unknown_event / unknown_option / already_resolved`。
- `event.trigger_card {template_id}`: 强制触发, 仅供测试用 (`EventView` 上已无对应 UI 入口 — 玩法不需要手动加事件)。
- `event.dismiss_flavor {event_id}`: 仅 flavor 类可直接关 (无 options)。

### 信号

`event_pushed(event_id, category, title)` · `event_resolved(event_id, option_id, applied_effects)`。

**约定**: `category == flavor` 不需选项, 点"知道了"即关; `opportunity / crisis / routine` **必须选一个 option** 才会从 pending_events 移除。多个 pending 按 FIFO 排队弹窗。

**选项后果预览**: 事件弹窗的每个选项按钮上, label 后面拼一句**后果预览** —— 玩家选之前就知道会发生什么 (e.g. "请全员喝奶茶 — 支出约 $6,000")。由 `EventSystem.describe_option_consequence(opt)` 生成: 遍历该 option 的 effects, 比例化数值按**当前** GameState 即时估算 (= 此刻选择会落地的实际值), 无 effect 的选项显示"无直接影响"。

**破坏性后果必须直说** (2026-05 文案审计): `dc_terminate` / `dataset_delete` 这类会永久移除资产的 effect, 后果预览不能只写"处理/关停/删除"这种泛词, 必须明确:

- `dc_terminate`: 会移除一座数据中心; 若其中有自购 GPU, 会在关停前按二手价自动出售 (云租用不退款)。
- `dataset_delete`: 会永久删除一份数据集。

事件选项 label / body 也不能暗示尚未实现的资产变化。例如没有 hire/lead 加入 effect 的卡, 不写"研究员加入""人才入伙"; 没有 lead 移除 effect 的卡, 不写"放他走"。这种叙事可以改写成"顾问背书 / 市场信心 / 团队声誉"等已由现有现金或订阅 effect 表达的结果。

### 回合推进门禁

`pending_events` 非空时**不允许进入下一周** —— `TurnManager.can_advance()` 返回 `false`, `main.gd` 据此禁用「推进回合」按钮。`TurnManager.advance()` 本身保持纯粹 (不内嵌校验, 测试可直接调用); 门禁只在 UI 层强制。

**推进后的事件导航**: 玩家点击「推进回合」后, 若 `action` 相位由 EventSystem 推入了新的 `pending_events`, `main.gd` 在本次 `TurnManager.advance()` 同步返回后立即切到侧栏「事件」页, 让玩家直接处理 FIFO 队首事件。若 UI 处理函数被调用时已经有 pending 事件, 不推进回合, 也切到「事件」页作为门禁提示。该行为只属于主 HUD 交互层; `TurnManager.advance()` 与 EventSystem 抽卡逻辑不直接持有 UI。

**事件页一致性兜底 (2026-06)**: `main.gd` 构造事件页数据时必须复用 `EventSystem._load_card()` / `EVENTS` 注册表, 不再自行按 `template_id.tres` 拼路径。若旧存档里出现找不到模板的 pending event, 事件页仍要显示「事件模板缺失」占位卡, 让玩家看到 badge 对应的队列项, 不允许显示成「无」。缺模板占位卡可通过「知道了」走 `event.dismiss_flavor` 清掉, 避免坏档永久卡住推进。

## 4. 核心算法

### 4.1 抽取流程 (每 action 相位)

每周 action 相位按固定优先级尝试抽事件: 先确定性触发时间锚点 paradigm 卡 (§4.4), 再触发历史档案卡 (§4.4bis), 再尝试 routine 强制卡 (§4.5), 最后才按 `EVENT_TRIGGER_PROB_PER_WEEK` 概率从随机池抽一张。任一步把卡推入 `pending_events` 后即停止 (一次一张)。随机池 = 非 paradigm、非 history 且非 routine 的卡, 候选还需满足条件检查 (§4.3)、cooldown 未过、`weight > 0`, 按 `weight` 加权随机挑选。

> **频率取舍** (v16): `EVENT_TRIGGER_PROB_PER_WEEK` = `0.03` (随机池期望每 ~33 周命中一次, 成熟局约每年 1-2 张随机事件), `ROUTINE_INTERVAL` = `12` (每 12 周一张强制 routine)。加上历史档案与 paradigm 锚点后, 成熟局整体节奏约**每 6-8 周**一张事件。演进路径: v9 起步 0.35 (玩家体感"几乎每周弹一次", 信噪比太低) → v10 降到 0.10 + routine 每 4 周 → v11 降到 0.07 + routine 每 8 周 → v16 降到 0.03 + routine 每 12 周, 修复成熟局约每 4.7 周一张的过密问题。`max_triggers` 让重大事件不再无限重复 (§4.7), 配合降频进一步提升信噪比。

**冷却单位**: `EventCard.cooldown_months` 是设计层的"月", 代码落地时换算为周: `cooldown_turns = cooldown_months * TurnManager.WEEKS_PER_MONTH`。字段名沿用以避免批量迁移 `.tres`, 但所有实现/测试必须按月语义处理; 例如 `cooldown_months=24` 表示约 96 周。

### 4.2 EventEffect 分发表 (`_apply_effect`)

**设计原则**: effects **永不直接改 GameState**, 都通过 CommandBus 走对应所有者系统。

> **v11 真两难约定**: 新增的 drama 卡 (见 `事件库.md`) 两个选项都必须挂**真生效**的 effect, 形成对立取舍 (典型: A 拿钱但掉用户 / B 花钱但涨用户)。只允许用下表中**已实现**的 kind (`economy_spend` / `economy_award` / `product_boost_subscribers` / `dc_terminate` / `dataset_delete`), **禁用**已废弃 no-op 的 `fame_add` / `npc_perturb` / `npc_capability_jump` 以及尚未落地的 `flag_set`。`event_system_test.gd` 有 lint 钉住这一点。早期 routine 填充卡仍可保留一个"无直接影响"的轻量选项 (不属 drama 卡)。

| effect.kind | 调用 | 用途 |
|---|---|---|
| `economy_spend` | `economy.spend` | 玩家支付 |
| `economy_award` | `economy.award` | 玩家收钱 |
| `tech_grant` | `tech.unlock_node` | 强解某科技节点 |
| `dc_terminate` | `infra.terminate_dc` | 危机类: dc 故障下线; 无 `dc_id` 时随机挑一个 |
| `model_buff` | `research.evaluate_apply` | 直接覆盖 capability |
| `dataset_delete` | `dataset.delete` | 合规审查类; 无 `dataset_id` 时随机挑一个 |
| `product_boost_subscribers` | `product.update_subscribers` | 订阅数 ±N; 无 `product_id` 时挑订阅最多的产品 |
| `flag_set` | (punt: log + ok) | 待通用 flags slice 落地 |
| `debug_starter_kit` | 内部 helper | 仅 debug 事件用 |
| ~~`fame_add` / `npc_perturb` / `npc_capability_jump`~~ | — | 已废弃 no-op (兼容老 .tres) |

#### 4.2.1 比例化数值 (v10)

游戏现金从 `$80k` 起步、可达 `$80B` (融资 F 轮), 跨 6 个数量级。固定金额必然要么早期破产、要么后期没感觉。因此 effect 数值改为**按当前状态的百分比 + 下限/上限兜底**:

- `economy_spend` / `economy_award` 的 `params` 支持 `pct` (现金比例) + 可选 `floor` / `cap`:
  - `amount = clamp(round(abs(pct) * max(0, cash)), floor, cap)`
  - 仍兼容旧的 `amount` / `cost:{cash:N}` 绝对值写法。
  - 花钱永远花得起 (是现金的百分比); `floor` 保证早期也不是白嫖。
  - **`cap` 必填** (写了 `pct` 的就必须写 `cap`): 后期玩家现金可达 $50B-200B, 没 `cap` 时 7% 的"政府补助"会算成 $14B, 显然脱离事件叙事。 `cap` 数值按事件体量分档 (routine 小事件 ≤ $5M / 危机 ≤ $50M / 大机会 ≤ $200M)。`event_system_test.gd::test_all_proportional_economy_effects_have_cap` 自动 lint。
  - **琐事 routine 卡 cap 更狠 (2026-05)**: 喝奶茶 / 养猫 / 咖啡机这类"无厘头小决策"即便算到 cap 也只该是几万级 — 玩家反馈后期"喝奶茶 100w"严重出戏。故这几张的 `cap` 砍到 ≤ $100k (奶茶/咖啡 100k、养猫 60k); 并给养猫 / 咖啡机 / 搬办公室 (`routine_office_move`) `max_triggers=2`、实习生 demo `max_triggers=1`, 出现 1-2 次即可, 不再反复刷屏 (搬办公室原本每 ~10 个月就再弹一次, 玩家反馈"搬太多次")。`event_system_test.gd::test_routine_chore_cards_have_trigger_caps / test_routine_chore_caps_stay_small_late_game` 钉住。
- `product_boost_subscribers` 的 `params` 支持 `pct` (总订阅数比例) + 可选 `floor` (最小绝对量级) + `cap` (|delta| 上限):
  - `|delta| = clamp(round(|pct| * Σ products.subscribers), floor, cap)`, 再按 pct 正负号决定方向。
  - 仍兼容旧的 `delta` 绝对值写法。
  - 比例化使「爆火」在用户多时依然是按比例的增益, 不会失效。
  - **`cap` 必填** (写了 `pct` 的就必须写 `cap`): 后期总订阅可达 5B+, 一张 22% pct 的 viral_meme 不加 cap 时能算成 +1.1B 用户, 完全脱离事件叙事。`cap` 数值按事件叙事可信度分档 (routine 媒体采访 ≤ 10M / 中型机会 30M-100M / 大型事件 ≤ 200M)。`event_system_test.gd::test_all_proportional_subscriber_effects_have_cap` 自动 lint。
  - **任何含 `product_boost_subscribers` 的卡都必须 `requires_product=true`**。EventSystem 不带 `product_id` 时会选订阅最多的产品; 若公司还没有产品, 下游 `product.update_subscribers` 没有目标, 预览和实际结算就会脱节。`event_system_test.gd::test_subscriber_effect_cards_require_product_gate` 自动 lint。
- `economy_award` 只表示外部现金流入 (补助、合同、投资、转卖等), 或明确由具体工具/外包方案带来的可落地成本节省; 不表示"省下了预算"、"保持诚实"、"占领道德高地"这类抽象姿态。若需要机械取舍, 优先用订阅变化、花费减少为 0, 或未来的声誉/风险系统。

### 4.3 触发条件检查 (`_conditions_met`)

`min_turn` / `requires_cash_min` / `requires_revenue_min` / `requires_rank_max` / `requires_unlocks` 之外, v10 加 7 个状态门禁 (见 §1)。全部 AND。任一不满足 → 该卡不进候选池。

### 4.4 时间锚点 paradigm 事件

`weight=0 不进随机池`, 到 `min_turn` 由 `_maybe_trigger_paradigm_events()` 确定性触发, 一次性 (`cooldown=9999`)。纯叙事卡, 不再触碰 NPC capability (v8 PR-H)。

### 4.4bis 历史档案事件 (v12)

历史档案事件是教育向 `flavor` 卡, 用来把**2017 开局之后** AI 发展史的重要拐点以短新闻/档案方式插入游戏中。更早的图灵测试、达特茅斯会议、专家系统、反向传播、围棋等只作为设计背景或帮助文档素材, 不作为回合内"新闻事件"推给玩家。规则:

- `id` 统一以 `history_` 开头, `category=flavor`, `weight=0`, `max_triggers=1`, `cooldown_months=9999`, 不进随机池。
- `EventSystem.HISTORICAL_EVENTS` 按历史顺序登记; `_maybe_trigger_historical_event()` 在 action 相位中按顺序检查 `min_turn`、条件、cooldown、`max_triggers`, 一次最多推一张。
- 玩家没有选项, 只点"知道了"; 首批历史档案不结算任何资源/订阅影响。`passive_effects` 只保留为其它 flavor 新闻未来扩展点, 不用于历史档案。
- 历史档案正文是主要信息载体, UI 必须完整显示正文, 不能沿用通用资产卡字段的 3 行截断。
- UI 文案必须遵守化名规范: 不在 `.tres`/UI 文案中出现真实 GPU、模型、架构或公司品牌名; 如需现实对照, 只写在设计文档中。

### 4.5 routine 常规事件 (每 12 周强制)

距上次 routine 满 `ROUTINE_INTERVAL` (=12) 周且 `pending_events` 为空时, 从 routine 类卡中按条件检查 + cooldown + `max_triggers` 过滤出候选, 加权随机挑一张强制弹出, 并更新 `last_routine_turn`。本周无合格 routine 则下周再试。

- routine 卡也走 `cooldown_months`, 避免同一张连弹。
- 至少 5 张 routine 卡只 gate `min_turn >= 4` 且 `max_triggers=0` (无其他前置), 任何公司都满足 → routine 池长期不塌缩。它们必须是单人公司也成立的低打扰小事, 例如域名续费、票据整理、密码轮换、椅子维修、无厘头律师函。
- 高频轻量 routine 仍需限次: `routine_open_source_pr` 最多触发 2 次, 避免神秘 PR 变成固定月经梗。
- 涉及"全员"/"工程师"/"实习生"/"工位挤"/"搬大办公室"等多人意象的 routine 卡必须 gate `requires_staff_min >= 1` —— 创始人单人时期不弹"全员大会"/"咖啡机罢工"/"搬更大办公室"(`routine_office_move`)这种叙事不符的卡。
- routine 卡必须有 options (无厘头小决策), 不能是 flavor。

### 4.6 融资 (v9 已下线)

融资完全归 EconomySystem 拥有, 玩家自发 8 轮顺序触发, 不走事件流。详见 `经济系统设计.md §4.6`。

### 4.7 限次事件 (max_triggers, v11)

`EventCard.max_triggers` (默认 `0` = 不限) 表示该卡**全局最多触发几次**。`EventSystem._trigger()` 每推一张卡就把 `GameState.event_trigger_counts[card.id] += 1`; 三处候选过滤 (随机池 / routine 强制 / paradigm 锚点) 都调 `_triggers_exhausted(card)` —— 当 `max_triggers > 0 且 count >= max_triggers` 时该卡不再进候选。

- 用途: 重大 drama 事件 (董事会内乱 / 暂停训练公开信 / 数据诉讼 …) 一局**只来一次**, 来过就不再骚扰。比早期靠 `cooldown_months=9999` 硬凑一次性更语义化, 也和 paradigm 卡 (仍用大 cooldown) 解耦。
- 破坏性危机也可以用 `max_triggers` 控体验强度。`dc_meltdown` 会永久移除一座数据中心, 因此首年不出现 (`min_turn=52`), 且一局最多触发 2 次。
- `event_trigger_counts` 入存档 (owner = EventSystem), 读档后限次状态保持; `GameState.reset()` 清零。
- 与 `cooldown_months` 正交: cooldown 控"两次之间隔多久", max_triggers 控"一共能来几次"。一次性卡设 `max_triggers=1` 后 cooldown 实际不再生效 (只要 ≥ 单次处理周期即可)。

## 6. 未决问题

- [ ] **事件持续效果 (buff)**: 当前 effect 只有瞬时 (含比例化的瞬时)。「全行业泡沫期持续 6 月用户 ×1.5」这类仍需 `active_buffs` 切片或独立 `BuffSystem`。**明显扩展点**。
