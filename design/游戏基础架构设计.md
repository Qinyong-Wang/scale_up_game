# 游戏基础架构设计

> **目的**: 定义代码骨架 — 状态放哪、系统怎么通信、回合怎么推进、存档怎么做。
> **读者**: 要往项目里加新系统、新数据、新机制的人。

---

## 1. 设计目标

- **解耦**: 13 个业务系统互不直接引用, 新增系统不改旧系统代码。
- **可测**: 任意 system 可脱离场景树跑单测。
- **可存**: 任意时刻把游戏状态序列化到 JSON 并完整还原。

---

## 2. 总体结构

```
Scenes / UI         读 GameState · 发命令 · 订通知
   ↕
Systems            处理命令 · 发通知 · 读写自己的状态切片
   ↕
GameState          唯一可变状态 (纯数据)
EventBus           通知信号 (过去式, fan-out)
CommandBus         命令派发 (fan-in, 带返回值)
TurnManager        驱动回合相位
```

三种通信模式:

| 模式 | 用什么 | 例 |
|---|---|---|
| **读状态** | 直接访问 `GameState.xxx` | UI 显示余额; TaskSystem 算速率遍历 lead |
| **通知"X 发生了"** | EventBus 信号 | `lead_hired`, `task_completed` |
| **命令"做 X 并要结果"** | `CommandBus.send` | `economy.spend`, `task.start` |

### 2.1 核心纪律

1. Systems 之间**不持有引用** — 跨系统操作走 CommandBus。
2. Systems **内部不存可序列化状态** — 所有要存档的数据进 GameState。
3. EventBus / CommandBus 只是**契约**, 不含业务逻辑。
4. GameState 不订阅信号, 是**被动**数据容器。
5. **写从总线, 读走 GameState** — 写要可拒绝/可审计走总线; 读没有副作用直接访问。

**例外: 自增 ID 计数器 (`_next_*_seq`)**。各系统用会话变量 `_next_*_seq` 生成
`dc_NNNN` / `ds_collected_NNNN` / `lead_NNNN` / `model_NNNN` / `task_NNNN` /
`product_NNNN` / `campaign_NNNN` / `event_NNNN` 这类 ID。计数器本身**不入存档**
(违反纪律 2 的代价是接受的 — 存它意义不大)。但**每个系统必须监听
`EventBus.save_loaded`**, 用 `GameState.max_seq_for_prefix()` 把计数器恢复到档内
现存最大编号之后, 否则读档后新建对象与档内对象**撞 ID** → `find_*()` 取首个匹配
→ 锁定 / 派单 / 训练解析到错的对象 (典型症状 `already_locked`)。被动型对象
(lead / model / product / campaign / event / dc / dataset) 还应 `GameState.dedup_ids()`
修旧档已有的重复; task 是主动 locker, 只 restore 不 dedup。详见 759612f 与
`数据集系统设计.md §3`。

### 2.2 状态切片所有权

GameState 的每个顶层字段有一个所有者 system, 只有所有者能写, 其他只读。**完整切片所有权表见 `系统耦合矩阵.md §1`** (13 个系统, 30+ 字段)。本节不重列。

## 3. Autoload 四件套

四个全局单例 (`project.godot` 注册, 加载顺序固定):

```
GameState     scripts/autoload/game_state.gd
EventBus      scripts/autoload/event_bus.gd
CommandBus    scripts/autoload/command_bus.gd
TurnManager   scripts/autoload/turn_manager.gd
```

业务 system 也以 autoload 形式注册, 但不属于"四件套"。

### 3.1 GameState

唯一可变状态容器, 纯数据。提供 `to_dict / from_dict / reset / rng()` 与日期 helper; 字段实现以 `game_state.gd` 为准, 完整切片所有权见 `系统耦合矩阵.md §1`。读档 ID 一致性 helper (`max_seq_for_prefix / dedup_ids`) 见本节 §2.1 例外说明。

**禁止**: 订阅信号、含业务逻辑、持有 UI 节点引用。

**RNG 使用约定**: 任何需要随机的系统**只走 `GameState.rng()`**, 禁止 `randi()` / `randf()` / `Math.rand*` 等全局随机。这样存档/读档后随机序列可重放, 测试里 `GameState.rng_seed = 42` 即可固定结果。

### 3.2 EventBus

只声明**通知信号** (过去式), 零或多订阅, emit 完不关心结果。

**命名**: `<domain>_<verb_past_tense>`。`event_bus.gd` 是跨系统信号声明表 (信号通常由其他 system / UI emit 或 connect), 声明区统一 `@warning_ignore("unused_signal")` 抑制 Godot 静态检查。所有信号及其载荷 / 订阅方登记在 `事件总线信号表.md`。

### 3.3 CommandBus

跨系统命令派发 — 调用方只知道命令字符串, 不知道谁处理。`command_bus.gd` 维护 `命令名 → Callable` 单 handler 表; `register` 对重复命令记 `Log.error` 并忽略 (测试可用 `register(..., true)` 显式替换), `send` 对未注册命令记 `Log.warn` 并返回 `{ok = false, error = &"unknown_command"}`。不 assert。

**约定**:
- 命令名 `<domain>.<verb>`, 例 `economy.spend`, `task.start`。
- handler 接收 `Dictionary`, 返回 `Dictionary`, **至少含 `ok: bool`**; 失败时含 `error: StringName`。
- **一个命令只能注册一个 handler**。
- 所有命令登记在 `命令总线表.md`。

> Godot 信号回调是**同步**的, handler 也是普通函数调用, 所以"发命令拿结果"是同帧完成。如未来出现真异步命令 (网络/AI 调用), 再引入命令队列模式。

### 3.4 TurnManager

`GameState.turn` 单位是**周** (`TurnManager.TURN_UNIT = &"week"`, `WEEKS_PER_MONTH = 4`)。各系统字段名 2026-05 后统一 `*_weekly_*` / `*_per_week` / `*_weeks` 命名 (少数 from_dict 路径接受 legacy `*_months` / `monthly_*` 旧键做存档兼容)。

`advance()` 把 `turn += 1`, 按 `PHASES = [upkeep, action, resolve]` 顺序逐相位 emit `phase_started`, 末尾 emit `turn_resolved`, 最后写自动存档 (`autosave_enabled` 时)。另提供:

- `can_advance()` — `pending_events` 为空才允许推进 (回合推进门禁, 见 `事件系统设计.md §2`); `advance()` 本身不带门禁, 门禁在 UI 层强制。
- `is_advancing()` — UI 用来在推进中合并刷新; 业务系统不依赖。
- `turn_label() / duration_label()` — UI 文案 helper。

**纪律**: TurnManager 不直接调用任何 system 方法, 只发 `phase_started`。所有 system 通过订阅这个信号并匹配相位名在自己的相位干活。

3 个相位的具体行为见 `玩法设计.md §1` (跨系统活动总表) 与各系统设计的 §3 "对外依赖"。

> 同相位内多 system 的回调顺序未指定。出现依赖冲突时, 要么拆相位, 要么用信号链显式串起来 (例: `users_resolved → revenue_resolved`)。

**UI 刷新批处理**: `advance()` 是同步调用, 中间信号可能很多。`TurnManager`
暴露只读查询 `is_advancing()` 给 UI 判断当前是否处在推进中; 业务系统不依赖它。
主 HUD 在推进中把多次刷新请求合并, 等 `turn_resolved` 后刷新一次, 避免后期每周
重复重建整个 HUD。

#### 3.4.1 真实世界日期锚点

游戏世界从 **2017-06-12** (Transformer 论文 *Attention Is All You Need* 在 arXiv 发布日) 开始计时, `turn = 0` 即 2017-06-12。所有硬件 GPU 上市、NPC 模型分数刷新、未来事件解锁都按这个真实时间线设定。

`GameState` 提供日期 API:

| 方法 | 返回 |
|---|---|
| `GameState.GAME_START_DATE` | `"2017-06-12"` 常量 |
| `GameState.current_date() -> String` | `"YYYY-MM-DD"` (turn_to_date 快捷) |
| `GameState.turn_to_date(turn) -> String` | `2017-06-12 + turn × 7` 天 |
| `GameState.date_to_turn(date) -> int` | `(date - 2017-06-12) / 7`, 把日期换算成 turn 阈值 |

UI 顶栏同时显示 「第 N 周 · YYYY-MM-DD」, 玩家可以直接看到当前真实日期。

> **未来扩展** (未实现): 新游戏对话框允许玩家选择开局年份 (最早 2017 / 2020 / 2022 / 2025); NPC 自动追到对应年份能力档位。

## 4. System 编写规范

### 4.1 位置与生命周期

- 文件: `scripts/systems/<name>_system.gd`
- 形态: autoload Node, 在 `project.godot` 注册
- 命名: `PascalCase + System` 后缀 (`EconomySystem`)

### 4.2 骨架

system 在 `_ready` 里 `CommandBus.register` 自己的命令、`EventBus.phase_started.connect` 订阅回合相位; 回调按相位名 (`upkeep` / `action` / `resolve`) 分支干活。命令 handler 命名 `_on_<verb>`, 接收并返回 `Dictionary`。

### 4.3 跨切片操作

- **改自己拥有的切片**: 直接写 `GameState.xxx`, 再 emit 对应的 `*_added` / `*_changed` 通知信号。
- **改自己不拥有的切片**: 必须 `CommandBus.send` 对应命令并检查返回的 `ok`, 失败时把错误透传给调用方。

例: TaskSystem 启动任务时, 花钱走 `economy.spend` (不拥有 `cash`), 写 `active_tasks` 则直接操作 (自己拥有)。

### 4.4 禁止事项

- 不持有其他 system 的引用 (包括字段和参数)。
- 不在 `_process` 推游戏逻辑 — 回合驱动来自 `phase_started`。
- 不直接读写 UI 节点。
- 不修改自己不拥有的切片 (走 CommandBus)。
- 不缓存可序列化状态 (存档会丢、读档会陈旧)。

## 5. Resource 规范

数据分两类:

| 类型 | 例子 | 存哪 | 进存档 |
|---|---|---|---|
| **静态模板** (设计期定义, 运行时只读) | `TaskTemplate / EventCard / FacilitySpec / GPUSpec / ProductTypeSpec / TechNode / LeadLevelSpec / EconomyTuning / MarketTuning / UserTuning / MarketingTuning` 等 | `resources/data/<domain>/*.tres` | 否 |
| **运行时实例** (玩家行动产生) | `Lead / Datacenter / Dataset / Model / Product / Campaign / TaskInstance / EventInstance / Loan / DatacenterConstruction / LeaderboardEntry / NpcCompany / GPUBatch` | 内存, 挂在 GameState 字段下 | 是 |

**约定**:
- 静态模板类名后缀 `Template / Card / Spec / Definition / Tuning`, 运行时实例无后缀。
- 静态模板用 `@export` 暴露字段, 文件用 `.tres`, **不允许 `.res`** (二进制不可 diff)。
- 运行时实例必须实现 `to_dict / from_dict`, 字段为纯数据 (数值/字符串/Dictionary/Array/StringName/其他实现 to_dict 的 Resource)。
- 实例持有 `template_id: StringName`, 通过 `_load_template` 类查回模板, **不直接持有模板引用** (避免存档把模板序列化进去)。
- 旧字段 rename 时, `from_dict` 路径用 `d.get("new_name", d.get("old_name", default))` 兼容旧存档。

## 6. 持久化

### 6.1 原则

- 存档 = `GameState.to_dict()` 序列化为 JSON。
- **不做版本迁移**: `SAVE_VERSION` 不一致直接拒绝读取, 提示玩家升级或开新档。
- 路径: `user://saves/<slot>.json`。

### 6.2 Save 模块

`save.gd` 是 autoload Node, UI 直接调, 不走信号。对外: `write / read / list_slots / delete_slot`, slot 目录 `save_dir` 默认 `user://saves` (测试覆写到 `user://test_saves` 保持干净)。

`write` 把 `{version, saved_at, turn, state}` 写到 `<save_dir>/<slot>.json`。`read` 校验版本后 `GameState.reset()` 再 `from_dict(state)`, emit `save_loaded`; 失败返回 `&"not_found"` / `&"corrupted"` / `&"incompatible_version"`。

### 6.3 自动存档

`TurnManager.advance()` 在 `resolve` 相位结束后调 `Save.write(AUTOSAVE_SLOT)` (`autosave_enabled = true` 时; 测试中 `-gtest`/`gut_cmdln.gd` 检测自动 disable)。

游戏内主场景收到窗口关闭 / 应用退出请求时, 也必须同步写一次 `AUTOSAVE_SLOT`。这是一次兜底存档, 不受 `TurnManager.autosave_enabled` 开关影响: 玩家可能在一周内完成建设、招聘、营销等操作后直接关闭应用, 这些操作尚未经过下一次回合推进, 不能只依赖回合末自动存档。

### 6.3.1 SaveLoadDialog (玩家入口)

顶栏「💾」按钮打开 `scenes/ui/save_load_dialog/SaveLoadDialog.tscn` (基于 `Window` 节点的模态对话框)。功能与视觉:

- **slot 列表**: 列出 `user://saves/` 下所有存档, 显示 `slot 名 / 回合 / 保存时间`。`autosave` 始终排在最前 (灰底, 标「自动」)。
- **新建存档**: 顶部输入框 + 「保存」按钮, slot 名走简单 sanitize (`a-z 0-9 _ -`, 截到 24 字符, 空名按 `manual_<turn>` 兜底)。
- **覆盖**: 对已有 slot 点「保存到此」时, 同名直接覆盖, 无二次确认 (单机游戏, 误操作可由 `autosave` 兜底)。
- **读档**: 点「读取」, 失败 (`not_found` / `corrupted` / `incompatible_version`) 在对话框内红字显示, 不关闭。
- **删除**: 点「删除」, slot 行立刻消失。`autosave` 行不显示删除按钮。
- **反馈**: 成功后 status 行显示「已存档 / 已读档 / 已删除」, 1 秒后清空; 失败保留直到下次操作。
- **视觉结构**: 对话框使用分区面板, 不再是裸输入框 + 裸列表。顶部是「手动存档」面板 (`SAVE_SECTION_CREATE`), 内含输入框与主保存按钮; 中部是「存档列表」面板 (`SAVE_SECTION_SLOTS`), 每个 slot 是一张卡片行 (slot 名 / 自动标签 / 元数据 / 读取、覆盖、删除按钮); 无存档时显示正式空状态 (`SAVE_EMPTY_TITLE` / `SAVE_EMPTY_HINT`)。

数据流仍走 `Save.write/read/list_slots/delete_slot`, 对话框只是 UI 包装。读档成功后由 `EventBus.save_loaded` 触发 `_refresh()` 把整个 UI 刷新。

### 6.3.2 Web 导出持久化契约

Web/HTML5 版本继续使用同一套 `Save` API 与 `user://saves/<slot>.json` 路径, 不新增 `localStorage` 分支。Godot Web 导出会把 `user://` 映射到浏览器本地的 IndexedDB 虚拟文件系统; 这样桌面版与 Web 版的存读档代码保持一致, 存档 JSON 仍以 `GameState.to_dict()` 为唯一来源。

Web 版发布契约:

- `Save.write/read/list_slots/delete_slot` 仍只通过 `FileAccess` / `DirAccess` 操作 `user://`, 禁止在游戏逻辑里直接写浏览器 `localStorage`。
- 运行时可用 `OS.is_userfs_persistent()` 做提示, 但不能把它作为唯一安全判定; 隐私模式、清站点数据、第三方 iframe 存储策略都可能让存档不可持久。
- Web 首发必须提供清晰文案: 存档保存在当前浏览器与当前站点下, 换浏览器/设备或清数据会丢。
- 后续若要跨设备或给玩家备份, 增加"导出存档 / 导入存档"按钮, 数据仍是现有 JSON 包装, 不改变 `GameState` schema。

### 6.3.3 Web 发布托管契约

Web 试玩版发布到静态托管即可, 不需要自建后端。首选渠道是 GitHub Pages + GitHub Actions:

- GitHub Actions 在干净 checkout 中生成 CI 专用 Web `export_presets.cfg`, 调用 Godot CLI 导出到 `build/web/index.html`。
- `build/web/` 只包含 Godot Web 静态产物 (`index.html`、`.js`、`.wasm`、`.pck` 等); 不提交 `build/` 到仓库。
- Pages 部署源使用 GitHub Actions, 不使用 `gh-pages` 分支, 避免把 100+ MiB 的导出包写进 git 历史。
- 正式公开 URL 绑定到仓库 Pages 域名: `https://qinyong-wang.github.io/scale_up_game/`。后续若接自定义域名, 只改 Pages 配置, 不改游戏存档 schema。
- GitHub Pages 只能托管公开静态文件, 不承担账号、跨设备存档、排行服务或遥测后端。若未来加入这些在线功能, 另起后端服务并补独立设计。

### 6.4 测试要点

- `to_dict → from_dict` 等价回环。
- 不兼容版本拒绝。
- 损坏的 JSON 归为 `&"corrupted"`, GameState 不被修改。
- 损坏 JSON 是常规失败路径, 不应触发 Godot 引擎级 ERROR; 用解析返回码转成业务错误并记录 `Log.warn`。

## 7. UI ↔ 核心交互

UI 三种交互方式: 读状态直接访问 `GameState.xxx`; 写操作走 `CommandBus.send` 并按 `r.ok` 做 toast 反馈 (成功提示 / 失败把 `r.error` 翻译成人话); 订阅 `EventBus` 信号做被动刷新。

**UI 不能做的事**:
- 不直接修改 GameState。
- 不直接调用 system 方法 (`TaskSystem.start_task(...)` 是反模式)。
- 不假设命令成功 — 必须看 `r.ok`。
- 不在 UI 节点存游戏数据 (状态在 GameState, UI 只持有局部展示状态)。

**派生数据** (如"按 specialty 分组的 lead") 默认每帧从 GameState 重算。回合制下变化不频繁, 性能足够; profiling 证明是瓶颈再加缓存。

**输入门控** (回合推进时不能再点"开始训练") 由 UI 自行 `disabled`, system 仍要做防御性校验。

## 8. 测试规范

呼应 `CLAUDE.md` 的 TDD 工作流第 4、5 步。

### 8.1 单元测试 (`tests/unit/`)

测单个 system 或数据类。直接用真 autoload, `setUp` 时 `GameState.reset()` 隔离状态。需要 mock 某命令时, 用 `CommandBus.register(cmd, stub, true)` 覆盖注册。

### 8.2 集成测试 (`tests/integration/`)

跑完整回合或完整存读档, 断言跨系统行为 (`vertical_slice_test.gd / playthrough_sim_test.gd` 等)。

### 8.3 测试不能依赖

- 屏幕渲染、输入设备、真实时间。
- 资源加载顺序 (必要时显式 seed RNG)。
- 文件系统副作用 (存档测试用 `user://test_saves/`)。

### 8.4 测试框架

GUT (`addons/gut/`, Godot 4 兼容版); 入口 `addons/gut/gut_cmdln.gd`, 详见 `docs/开发环境配置.md §2.3`。

### 8.5 导出包回归测试

编译包暴露的问题优先补成测试, 不能只靠手测。导出包回归测试分三类:

- **源码卫生单测**: 运行时代码不得用 `DirAccess.open("res://resources/data/...")` 或等价目录枚举加载静态数据; 发布包里 `.tres` 可能以 `.tres.remap` 存在, `res://` 枚举也不稳定。新增静态数据系统必须使用显式 `{id: path}` 表, 再加路径表与磁盘 `.tres` 一致性测试。`Save` 枚举 `user://saves` 是例外。
- **静态数据烟雾测**: 数据市场、拍卖行、NPC 时间线、事件库等首屏或回合推进依赖的数据, 必须有集成测试断言冷启动后非空、未来时间线会推进、UI 能展示 pending 状态。
- **存档时机测**: 一周内的玩家操作不能只等回合末 autosave; 返回主菜单、窗口关闭等退出路径要有测试覆盖, 并断言关键运行时切片能从 JSON 往返恢复。

## 9. 命名约定

| 场景 | 约定 |
|---|---|
| 信号 / 方法 / 字段 | `snake_case` |
| 类名 | `PascalCase` |
| 常量 | `SCREAMING_SNAKE_CASE` |
| 命令 ID / 错误码 / 信号载荷的离散字符串 | `StringName` (`&"..."`), 不要 `String` |
| 文件名 | 脚本/场景 `snake_case.gd / .tscn`; 设计文档允许中文 |
| 命令名 | `<domain>.<verb>`, 例 `economy.spend` |
| 通知信号 | `<domain>_<verb_past_tense>`, 例 `task_completed` |
| 日志 | **严禁** `print / printerr / push_error`, 统一走 `Log.*` (详见 `开发调试设计.md`) |

化名规范 (动物族 / 植物族, 不出现真实品牌名) 见 `公共枚举表.md §16` + `玩法设计.md §0bis`。
