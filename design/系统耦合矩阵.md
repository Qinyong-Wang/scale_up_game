# 系统耦合矩阵

> **目的**: 把 13 个 system 各自文档里的"对外依赖"汇总成总览表, 集中审视耦合关系。回答: 跨系统通信是否都走总线? 有没有隐藏耦合? 顺序依赖在哪?
> **读者**: 任何要新增/重构 system 的人, 评审设计时优先看本表。

> 本文档不重复各 system 内部细节, 只整理它们的**外部接口**与**互相调用关系**。系统行为请去对应的 `xxx系统设计.md`。

### 2.1 高入度 (被多方调用)

- **Economy** 被 7 系统调用 — 必然, 经济是花钱中枢。

## 3. 跨切片读 (read 不走命令)

**纪律**: 读切片允许 (走 GameState 直接访问), 但要**只读**。

| 读方 | 读哪些切片 | 用途 |
|---|---|---|
| EconomySystem | `MarketSystem.leaderboard` (通过 `market.get_rank`) / `revenue_history` (内部环形) | 融资估值 + 信用评级 + 融资轮触发条件 (v7 PR-F: 删 fame 依赖, 改 revenue + rank) |
| HiringSystem | `GameState.cash` | 月初候选池 S/A 抽取概率 (v7 PR-F: 旧用 fame, 改用 cash) |
| InfraSystem | `ResearchSystem.models` + `resources/data/models/os/*.tres` | deploy 校验自有模型 (published); 公共开源模板 |
| ResearchSystem | `InfraSystem.datacenters` / `ProductSystem.products` / `InfraSystem.GPU_SPECS+POWER_SPECS` | unpublish 校验 + v6 PR-E 定价基准取 last-year-best GPU |
| TaskSystem | `HiringSystem.leads / staff_pool` / `InfraSystem.datacenters` / `DatasetSystem.datasets` / `TechTreeSystem.unlocks+NODES` / `ResearchSystem.models` | 启动 task `_validate` |
| MarketSystem | `ResearchSystem.models` (published) | 排榜 |
| UserSystem | `MarketSystem.leaderboard` (经 `market.get_rank`) / `MarketingSystem.campaigns` / `ProductSystem.products` / `ResearchSystem.models` | 周度结算 (高频派生, v7 PR-F: rank-driven, 删 fame) |
| ProductSystem | `HiringSystem.leads / staff` / `ResearchSystem.models` / `TechTreeSystem.unlocks` (经 `tech.is_unlocked` 命令) | 创建产品校验 + quality 公式 + auto_track_latest |
| MonetizationSystem | `ResearchSystem.models` / `InfraSystem.datacenters.serving_tokens_per_sec` / `UserSystem.api_token_demand` / `ProductSystem.products` / `HiringSystem.LEAD_BONUS_TABLE` | 周度结算 (高频派生) |
| EventSystem | `turn / unlocks / cash / leaderboard / models / products` + `EconomySystem.quarterly_revenue` | 触发条件检查 + 融资轮 (v7 PR-F: 删 fame, 加 revenue) |

### 3.1 设计抉择: 何时读切片 vs 走命令?

- **读切片**: 高频、纯读、无副作用 — UserSystem 周度结算遍历产品列表。命令开销过大。
- **走命令**: 写操作、需返回成功/失败、需触发副作用 — `hiring.lock_lead`, `economy.spend`。

## 4. 信号订阅矩阵

| 监听方 | 来源信号 | 用途 |
|---|---|---|
| EconomySystem | `phase_started(upkeep/resolve)` / `state_reset` | 利息扣 + 破产判定 |
| HiringSystem | `phase_started(upkeep/action)` / `save_loaded` | 工资 + 月初候选池 (`turn-1) % 4 == 0`) + 读档恢复自增 ID |
| InfraSystem | `phase_started(upkeep/action)` / `save_loaded` / `tech_unlocked` | 周费 + 自建工程 + 重算 serving t/s + engineering 解锁触发重算 |
| DatasetSystem | `phase_started(action)` / `save_loaded` | 半年解锁 (`turn % 26 == 0`) 发 `dataset_market_updated` + 读档恢复自增 ID |
| TaskSystem | `phase_started(upkeep/action)` / `save_loaded` | 周度 `weekly_cost` + 进度推进 + error_rate + 读档恢复自增 ID |
| MarketSystem | `phase_started(action)` / `model_published` / `model_unpublished` / `state_reset` | 周度排榜 + 即时重排 + 重装 NPC |
| UserSystem | `phase_started(action)` | 周度结算 |
| MonetizationSystem | **`users_resolved`** | 信号链锁顺序 (UserSystem → Monetization) |
| ProductSystem | `phase_started(action)` / `model_updated` / `model_published` / `model_unpublished` / `lead_released` / `save_loaded` | 重算 quality + auto_track_latest + 自动建/删 API 产品 + 防御性下架 + 读档恢复自增 ID |
| ResearchSystem | `save_loaded` | 读档恢复自增 ID |
| MarketingSystem | `phase_started(upkeep)` / `save_loaded` | 周度扣预算 + 自然结束 + 读档恢复自增 ID |
| TechTreeSystem | `task_cancelled` | 清理 researching_nodes |
| EventSystem | `phase_started(action)` / `leaderboard_resolved` / `save_loaded` | 抽事件 + 融资轮 offer + 读档恢复自增 ID |

`save_loaded` 由 8 个业务 system (hiring / infra / dataset / task / product / research / marketing / event) 加 UI 订阅 — 各系统借此把会话级 `_next_*_seq` 自增 ID 计数器恢复到档内现存最大编号之后, 见 §2.1 例外说明。`state_reset` 由 EconomySystem (破产判定状态清零) + MarketSystem (重装 NPC 名册) 订阅。

### 4.1 关键信号链 (强制顺序)

```
phase_started(action)
   ├── ProductSystem._on_phase(action)           # 重算 quality
   ├── UserSystem._resolve_action()              # 用 quality 算 subscribers + token_demand
   │     └── emit users_resolved ──────────────►
   │                                              ▼
   │                              MonetizationSystem._on_users_resolved  # 用 token_demand 算营收
   ├── MarketSystem._on_phase(action)            # 排榜 (v7 PR-F: 不再发 fame)
   │     └── emit leaderboard_resolved ─────────►
   │                                              ▼
   │                              EventSystem._on_leaderboard_resolved   # 融资轮 offer
   ├── EventSystem._on_phase(action)             # 抽事件
   └── ...
```

**已显式锁的顺序**:
- `users_resolved → revenue_resolved`: MonetizationSystem 订阅 users_resolved 而不是 phase_started, 锁死顺序。
- `leaderboard_resolved → 融资轮 offer 检查`: EventSystem 订阅。
- `tech_unlocked(engineering) → dc_compute_recomputed`: InfraSystem 订阅, 让 engineering 解锁立即反映到 serving t/s。
- `model_published / model_unpublished → 排榜 + API 产品自动管理`: MarketSystem + ProductSystem 订阅。

**仍依赖 autoload 顺序的隐性约束** (未来若加更多系统要警惕):
- ProductSystem.quality 重算必须**先于** UserSystem 结算 (UserSystem `attract / churn` 用最新 quality)。当前 ProductSystem 在 autoload 顺序里靠前。

### 4.2 已用的"信号链"模式

- **users_resolved → MonetizationSystem**: 顺序锁死。
- **task_completed → 所有者 system 接收产出**: 通过 `completion_command` 派发。
- **model_published → MarketSystem 即时重排 + ProductSystem auto_track / API 自动建**: 单向广播。
- **tech_unlocked(engineering) → InfraSystem 重算 serving t/s**: 反馈链 (玩家研究 FP8 立即看到 dc 卡片 t/s 上升)。

- **arch 字符串**最危险: 4 个系统都依赖。`tech_tree_system.gd::ARCH_ID_ALIAS` 维护 legacy `transformer_v*` 迁移。

- **specialty 与岗位绑定**: ProductSystem 写死 `chief_engineer`; 未来若加新岗位 (agent_lead?) 要改 ProductSystem。可接受 — 业务规则。

## 6. 解耦评估总结

### ✅ 已经做好的

1. **Systems 互不持有引用**: 全部走 CommandBus 与 GameState; 无 `lead_system._tasks_system` 这种字段。
2. **TaskSystem fan-out 模式**: 完成时 `completion_command` 派发, 不直接写 ResearchSystem.models。新增 subtype 不改本系统。
3. **EventSystem effect dispatch**: 中央派发表 + CommandBus, 加新事件不改其他系统。
4. **资源 vs 资产二分** (v7 PR-F): cash / paid_users / token_demand 是资源; 其他都是资产。fame 已废弃。
5. **三态互斥状态机** (lead / dc): 命令对 (lock/release) 严格管理, 不允许"半锁"。
6. **信号链强制顺序** (users_resolved): 把不能并行的步骤显式串起来。
7. **engineering 乘数下沉到 dc.serving_tokens_per_sec** (v4 PR-B): InfraSystem 是单一权威, MonetizationSystem 不再二次乘, 反馈链不断。

### ⚠️ 需要警惕的

1. **同相位内的脆弱顺序**: ProductSystem.quality 重算 → UserSystem 结算; 当前靠 autoload 顺序而非显式信号链。**仍是改进点**。
2. **MonetizationSystem 直接遍历 datacenters** 找 `deployed_model_id == m.id`: 高频派生计算, 但 InfraSystem 改 dc 字段名时要改 Monetization。可接受, 已记录在公共枚举表。
3. **arch 字符串通过 4 个系统**: 见 §5.1。
4. **EventEffect.dc_terminate 在 busy dc 失败**: 危机事件想强制关停 training/serving dc 当前不支持。仍未决, 可能需 `infra.force_terminate_dc` 或 effect 自带 unlock 逻辑。
5. **paid_users 的派生 vs 字段**: GameState.paid_users 是冗余 (= Σ subscribers)。当前只 UserSystem 写 (路径单一), 但**未来加任何修改 subscribers 的路径必须重算 paid_users**。
