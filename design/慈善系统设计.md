# 慈善系统设计

**一句话目的**：给后期溢出的资金一个出口——玩家向若干「公益方向」发起阶梯式捐助；每次捐助是一个跑若干周的 **task**（不是捐完立刻生效），启动时一次性扣钱并全额抵税，**任务完成时**才把捐助额计入该方向、激活一个小而封顶的直接数值加成（S 级人才出现率 / 估值 / 营销转化率）；不引入任何「声誉」中间软指标。

> 本系统分三期落地，本文档先把**一期（核心 sink）**写清楚，二期（办公室荣誉 + 拍卖行）、三期（宇宙模拟 capstone）只在 §9 预告，等一期验收后再各自展开。

## 1. 定位与玩法

- **money sink 本体**：捐助直接扣 `cash`，是后期消化溢出资金的主要出口。
- **是 task，不是即时结算**：每次捐助通过 `TaskSystem` 起一个 `charity` 子类型任务。启动当周一次性扣捐助额（`base_cost`），但**加成与奖杯要等任务跑满 N 周、完成时**才生效——「捐了钱不等于马上拿到回报」。进行中只是「钱已捐、效果在路上」。
- **阶梯性（每档只能捐一次，顺序爬梯）**：每个方向有从小到大的档位（参考 ¥10M → ¥1B+，权威数值见 `.tres`）。档位越高，捐助额越大、**任务周期越长**、完成后激活的加成越高；爬到顶档后加成封顶。**每一档是一次性动作**：必须**按顺序**从最低档往上捐（先完成第 0 档才能捐第 1 档），**每档只能成功执行一次**，已完成或正在进行的档**不能再次捐**——这样每个方向的加成自然有上限（顶档 bonus），玩家不能靠反复点同一档刷钱或越级。
- **全额可抵税**：捐助在任务启动当周作为普通可抵扣支出全额冲减应税利润（见 §5）。卖点：边际税率最高 25%+20%（企业税 + AI/UBI 税）时，捐 X 的净成本 ≈ `X × (1 − 边际税率)`。
- **加成是直接的、小而封顶的**：完成后直接乘/加到目标数值上，中间**没有**「声誉 / prestige / reputation」全局软指标（`fame` 已在 v7 PR-F 删除，本系统不复活它）。加成定位为锦上添花。

## 2. 捐助方向

一期三个方向，正好一一对应三个目标数值；每个方向只驱动**一个**直接加成：

| 方向 id | 显示名 | 叙事 | 直接加成（完成后生效）| 钩子（见 §6） |
|---|---|---|---|---|
| `bio_science` | 生物 / 前沿科学基金 | 资助科学 → 顶尖人才慕名而来 | S 级人才池权重 ↑（加性） | `HiringSystem._draw_level()` |
| `fundamental_compute` | 基础物理 / 超算捐助 | 高调资助"知识边界" → 投资人给溢价 | 估值乘子 ↑ | `EconomySystem._compute_valuation()` |
| `social_welfare` | 失业 / 低收入援助 | AI 抢工作 → 回馈社会 → 公众转化更高 | 营销转化率乘子 ↑ | `UserSystem` 营销拉新项 |

`social_welfare` 与现有 **AI 税（=UBI 税）**主题呼应；二期可考虑追加「减免部分 AI 税」的强联动，一期先只做转化率加成。

## 3. 数据模型

数值放在 `resources/data/charity/causes/*.tres`（`CharityCauseSpec` 资源），由 `CharitySystem` 在 `_ready` 自动加载。本文档只描述字段语义，具体数值以 `.tres` 为准。

`CharityCauseSpec`（`scripts/resources/charity_cause_spec.gd`）字段：

- `id: StringName` — 方向 id（`&"bio_science"` / `&"fundamental_compute"` / `&"social_welfare"`）。
- `display_name: String` / `description: String` — UI 文案（走 i18n，见 §8）。
- `effect_kind: StringName` — 该方向驱动哪种加成：
  - `&"s_tier_weight"` — **加性**叠到 S 档抽取权重（语义同 `FounderOriginSpec.s_tier_weight_bonus`）。
  - `&"valuation_mult"` — 估值乘子，最终乘子 `1.0 + bonus`。
  - `&"conversion_mult"` — 营销转化率乘子，最终乘子 `1.0 + bonus`。
- `tier_amounts: Array[int]` — 升序的**单档捐助额**（第 `i` 档的捐助包金额；不是累计阈值）。
- `tier_weeks: Array[int]` — 与 `tier_amounts` 平行；发起该档捐助时任务跑多少周（档位越高越久）。
- `tier_bonuses: Array[float]` — 与 `tier_amounts` 平行；**已完成到第 `i` 档**时当前 bonus = `tier_bonuses[i]`（顺序爬梯，故就是已完成的最高档），最后一档即封顶。
- `tier_labels: Array[String]` — 各档显示名（如「区域级 / 国家级 / 全球级」），走 i18n。

**取档规则（按已完成档数）**：每个方向记一个**已完成档数** `done`（见 §4）。当前已生效档 index = `done - 1`，bonus = `tier_bonuses[done - 1]`；`done == 0`（未完成任何档）时取中性值（`s_tier_weight` → `0.0`，两个 `*_mult` → 乘子 `1.0`）。下一可捐的档 index = `done`（即必须按顺序爬：第 0 档完成后才解锁第 1 档）。进行中（未完成）的捐助任务**不**增加 `done`，故加成在任务完成那一刻才跳一档。`charity_donated` 累计额仅用于「累计已捐 ¥」展示与账本，不再参与取档。

## 4. GameState 字段

新增切片（owner：`CharitySystem`）：

- `charity_donated: Dictionary` — `{cause_id(StringName): 已完成累计捐助额(int)}`。随存档持久化。仅在慈善任务**完成**时累加。**仅用于展示 / 账本**，不再决定档位。
- `charity_tier_done: Dictionary` — `{cause_id(StringName): 已完成档数(int)}`，0 表示一档未完成。**取档与加成的唯一依据**。每完成一档慈善任务 +1（顺序爬梯，故等于「已完成的最高档 index + 1」）。随存档持久化。

两者 `reset()` 都置空 `{}`；`to_dict()` 用 `_dict_with_sn_keys_to_strings`、`from_dict()` 用 `_strings_to_sn_keys` 序列化（与 `staff_pool` 等同构）。旧档无 `charity_tier_done` key 时回退为空 dict（全中性），不破坏兼容。

> 进行中的捐助任务存在 `GameState.active_tasks`（owner = TaskSystem）里，subtype = `charity`，无需 CharitySystem 另存切片。二期奖杯 / 三期宇宙模拟进度等切片，等对应期再加。

## 5. 任务接入 / 命令 / 抵税

慈善作为 `TaskSystem` 的新子类型 `charity` 接入，复用其「启动扣费 → 周推进 → 完成 fan-out」框架（见 `任务系统设计.md`）。改动面：

- **模板**：`resources/data/tasks/charity/charity_project.tres`（`TaskTemplate`，`subtype = &"charity"`）。`base_cost` / 周期为**动态**：由发起时选的方向 + 档位决定（参照 `data_collection_dynamic` 的动态定价：`base_cost_override` = 捐助额，时长 = 该档 `tier_weeks`）。无 `weekly_cost`、无资源锁（常规慈善不占 lead/staff/dc；三期宇宙模拟例外，见 §9）。
- **启动入口（owner 命令）**：`charity.start_donation {cause_id, tier_index}`（owner：`CharitySystem`，仿 `tech.start_research`）。
  - 校验（顺序爬梯 + 每档一次 + 互斥）：
    - `cause_id` 存在，否则 `cause_unknown`；
    - `tier_index` 在 `[0, tier_amounts.size())` 内，否则 `tier_invalid`；
    - **该方向尚有未完成的档**（`done < tier_amounts.size()`），否则 `all_tiers_done`；
    - **`tier_index` 必须等于下一可捐档** `done`（不能越级、不能重捐已完成档），否则 `tier_out_of_order`；
    - **该方向当前没有正在进行的慈善任务**（一次只捐一档），否则 `already_running`；
    - 捐助额 `amount = spec.tier_amounts[tier_index]` 满足 `amount ≤ GameState.cash`（**慈善是自愿支出，不允许捐成负数 / 触发破产倒计时**），否则 `insufficient_cash`。
  - 通过：调 `task.start({template_id: &"charity_project", cause_id, amount, weeks: spec.tier_weeks[tier_index], base_cost_override: amount})` 起任务。启动当周一次性扣 `amount`。
  - 错误码：`cause_unknown` / `tier_invalid` / `all_tiers_done` / `tier_out_of_order` / `already_running` / `insufficient_cash`。
- **完成回调**：任务完成时 `TaskSystem` 走 `completion_command = &"charity.credit"`，payload `{cause_id, amount}`。`CharitySystem._on_credit` 把 `charity_tier_done[cause_id] += 1`（推进一档）、`charity_donated[cause_id] += amount`（累计展示），重算该方向当前 bonus，emit `charity_completed`（§7）。此刻加成才对外生效。爬到顶档时点亮 `charity_global` 奖杯。
- **取消**：复用 `task.cancel`；按 `任务系统设计 §5.1` 退 `base_cost × (1 − elapsed/total) × REFUND_COEF`。

**抵税（全额可抵）**：慈善任务启动扣的 `base_cost` 走 `economy.spend`，reason 用一个**不在** `EconomySystem.NON_TAXABLE_REASONS` 里的类目，因此当周作为普通可抵扣支出自动冲减应税利润——无需额外税务代码。为了让周账本把慈善单列出来（而非混进 `task_start`），慈善任务的启动扣费 reason 用 `&"charity_donation"`（实现上给 `charity` 子类型的 base_cost 走该 reason；见 `任务系统设计` 启动扣费分支）。文档此处显式声明：`&"charity_donation"` **必须可抵税**，不得划进不可抵名单。

## 6. 系统接入点（加成在任务完成后才生效）

`CharitySystem`（autoload，`scripts/systems/charity_system.gd`）加载所有 `CharityCauseSpec`，按 `GameState.charity_donated`（已完成累计额）聚合当前加成，暴露三个访问器（范式抄 `FounderSystem`，无任何方向 / 未达首档时返回中性值）：

- `s_tier_weight_bonus() -> float` — `effect_kind == &"s_tier_weight"` 方向的当前 tier bonus（中性 `0.0`）。
- `valuation_multiplier() -> float` — `1.0 + (valuation_mult 方向当前 bonus)`（中性 `1.0`）。
- `conversion_multiplier() -> float` — `1.0 + (conversion_mult 方向当前 bonus)`（中性 `1.0`）。

各系统在已有计算点叠加（与 `FounderSystem` 共存、各自独立）：

- **HiringSystem** — `_draw_level()` 调整 S 档权重时，把 `CharitySystem.s_tier_weight_bonus()` 与 `FounderSystem.s_tier_weight_bonus()` **一起**加性叠加后再归一化（两者为 0 时零行为变化）。
- **EconomySystem** — `_compute_valuation()` 结果再乘 `CharitySystem.valuation_multiplier()`（紧挨现有 `FounderSystem.funding_multiplier()` 乘法；中性 1.0 时零行为变化）。
- **UserSystem** — 营销拉新项 `marketing_attract`（`budget × MARKETING_CONVERSION_RATE × lead 系数`）再乘 `CharitySystem.conversion_multiplier()`。该乘子也要计入营销 tab / 新建活动对话框的「预计每周拉新」预览，使显示与实际一致（同 `出身系统设计.md §5` 处理 `user_growth_multiplier`）。

因为加成只读 `charity_donated`（完成时才累加），所以「任务进行中加成不生效、完成后才跳变」是这套读法的自然结果，无需额外状态机。

## 7. 信号

慈善任务的生命周期复用 `TaskSystem` 既有信号（`task_started` / `task_progress` / `task_completed`，subtype = `charity`）；UI 进度条直接订阅它们。

另新增一个 EventBus 信号（**三处同步**：`event_bus.gd` + `event_bus_test.gd` 的 `EXPECTED_SIGNALS` + `事件总线信号表.md`），用于「加成已生效」这一慈善语义事件：

- `charity_completed(cause_id: StringName, amount: int, cumulative: int)` — 慈善任务完成、`charity_donated` 累加后发。

`main.gd._subscribe_events()` 订阅 `charity_completed`（以及已订阅的 `task_*`）→ `_request_refresh()`，否则完成后顶栏 / 慈善 tab / 营销预览不刷新。

## 8. UI（慈善 tab）

- 新视图 `scenes/ui/views/charity/`，挂为 HUD 的一个 tab（沿用卡片墙 / 抽屉风格）。
- 每个方向一张卡：显示名 + 描述 + **当前已生效档与 bonus** + 下一档捐助额 + 各档捐助按钮。
- **按钮态（顺序爬梯）**：三档各一个按钮——已完成档显示「已完成」且禁用；**下一可捐档**可点（点它 → `charity.start_donation`），若现金不足或该方向已有进行中的捐助任务则禁用；尚未解锁的更高档显示「未解锁」且禁用。
- 卡上展示**进行中的慈善任务**：剩余周数进度（读 `active_tasks` 里 subtype=charity 且 cause 匹配的实例），明确「效果将在 N 周后生效」。
- 捐助前显示**净成本提示**：`净成本 ≈ 捐助额 × (1 − 当前边际税率)`。
- 文案走 `tr()`（`strings.csv`），方向 / 档位内容文案走 `.tres → content.csv`。

## 9. 阶梯参考数值（仅参考，权威在 `.tres`）

「小而封顶」对照：创始人 scientist 出身给的 S 权重加成约 `+0.15`，故慈善封顶应明显更小。下表为讨论用参考，实际写进 `.tres`：

| 方向 | 档位（累计额）| `tier_weeks`（参考）| bonus（参考）| 封顶 |
|---|---|---|---|---|
| `bio_science`（S 权重，加性）| 10M / 100M / 1B | 4 / 12 / 26 | +0.02 / +0.05 / +0.08 | +0.08 |
| `fundamental_compute`（估值乘子）| 10M / 100M / 1B | 4 / 12 / 26 | +1% / +3% / +5% | +5% |
| `social_welfare`（转化率乘子）| 10M / 100M / 1B | 4 / 12 / 26 | +2% / +5% / +10% | +10% |

> ¥100B~¥1T 这类超大档不进一期 buff 表（已超 F 轮融资量级），留给二期奖杯与三期宇宙模拟做门槛。任务周期与档位绑定：捐得越大、见效越慢，强化「这是一项长期事业」。

## 10. 二期 / 三期预告（占位，验收后展开）

- **二期 · 办公室荣誉 + 拍卖行**：现有 `office/` 场景加荣誉桌 / 墙，陈列**挣来的**奖杯（慈善里程碑等）；拍卖行作为另一个 sink，竞拍收藏品摆进办公室。
- **三期 · 宇宙模拟 capstone**：玩家先捐建特殊「东西」——一座 ~¥100M 的**太空数据中心**（特殊基建资产）→ 以它为锁定资源起一个 ~52 周的 `charity` 超长任务 → 一年后任务完成，揭晓终极答案 **"42"**，授予终极奖杯 + 结局彩蛋。它正是 §1「捐了钱和东西、要跑很久才完成」的极端形态，与一期同一套 task 框架，只是多锁一个 dc、周期最长。
