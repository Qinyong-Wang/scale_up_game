# 公共枚举表

> **目的**: 集中登记所有跨系统使用的字符串 (`StringName`), 便于审计与防止手抖打错值。
> **读者**: 新增枚举值或新写消费方时, 必须同步更新本文 + 所有 consumer 代码位置。

---

## 1. 系统相位 (TurnManager)

| 值 | 消费方 |
|---|---|
| `&"upkeep"` | EconomySystem (loan) / HiringSystem (salaries) / InfraSystem (rent + gpu runtime + cloud) / TaskSystem (weekly_cost) / MarketingSystem (campaign budget) |
| `&"action"` | HiringSystem (lead_pool) / InfraSystem (construction) / TaskSystem (progress) / MarketSystem (排榜) / UserSystem / ProductSystem (quality) / DatasetSystem (半年解锁信号) / EventSystem (抽事件) |
| `&"resolve"` | EconomySystem (破产判定) |

## 2. lead.specialty (6 种)

| 值 | 主职能 |
|---|---|
| `&"chief_scientist"` | pretrain (主) + research (主) + posttrain (副) |
| `&"ml_research_lead"` | posttrain (主) + evaluate (评分加成) |
| `&"eval_lead"` | evaluate (主) |
| `&"chief_engineer"` | product (lead) + dc (cluster_efficiency 加成) |
| `&"data_scientist"` | data_collection |
| `&"marketing_lead"` | campaign (campaign_efficiency) |

`Lead.is_player_scientist = true` 时 specialty 固定 `&"founder"` (万能匹配但不提供 bonus)。详见 招聘系统 §1.1。

定义方: `HiringSystem.SPECIALTIES`。消费方: TaskSystem (input_schema), ProductSystem (lead_specialty_bonus), InfraSystem (chief_engineer dc 加成), UserSystem (campaign_efficiency)。

## 3. lead.level

`&"S"` / `&"A"` / `&"B"` / `&"C"` / `&"founder"` (player_scientist) — 候选池按 cash 档抽取等级; 数值见 `resources/data/hiring/lead_levels/*.tres` 与 `resources/data/hiring/lead_pool.tres`。

## 4. dc.status

`&"idle"` / `&"training"` / `&"serving"` — 三态互斥。转移规则见 基础设施 §1。

## 5. dc.ownership

`&"rented"` / `&"owned"` / `&"cloud"` — 决定周费组成。详见 基础设施系统设计 §1.4。

## 6. model.status

`&"pretrained"` / `&"posttrained"` / `&"evaluated"` / `&"published"` — 模型生命周期。见 研究系统 §4.1。

只有 `evaluated` 才能 publish; 只有 `published` 才能 deploy 与 serve。`pretrained` 的 capability **不可见** (`capability_revealed = false`); `posttrained` 直接结算 capability (v2)。

## 6bis. model.provenance

`&"trained"` (玩家走 pretrain task 出来) / `&"downloaded_os"` (玩家走 `research.download_open_source`, 或基建公共开源部署经 `research.ensure_open_source_release_published`, 从 OS NPC 的 pretrain release 实例化, v9 PR-I)。后者走开源 API 指导价与价格弹性路径。

## 6ter. serving_target.kind (Datacenter)

`&"owned_model"` / `&"open_source_model"` — 新部署统一写 `owned_model`, id 指向 `GameState.models`。`open_source_model` 是旧存档兼容值, id 指向 OS NPC 的 pretrain release (`resources/data/npcs/*.tres → NpcCompany.model_releases[*].id`, v9 PR-I); 读档时 InfraSystem 会尽量修复/回退。v9 PR-I 之前 (`open_source_model` id 曾指向 `OpenSourceModelTemplate` 模板) 的存档由 InfraSystem 在 save_loaded 时迁移失败 (旧 template_id 找不到对应 release), 服务自动转 idle, 不影响其他状态。

## 7bis. attention.id (B 轴) + loss.id (C 轴)

PretrainDialog 的 B / C 轴, 对应 `TechNode.tree = &"attention"` / `&"loss"`。权威源 `resources/data/tech/attention/*.tres` 与 `resources/data/tech/loss/*.tres`。

- **attention.id**: `&"mha_baseline"` (默认解锁) / `&"gqa"` / `&"mqa"` / `&"mla"` / `&"hybrid_attention"` / `&"infinite_attn"`。通过 `tech.get_attention_coefs` 查 train_coef / inference_coef。
- **loss.id**: `&"ce_baseline"` (默认解锁) / `&"zloss"` / `&"mtp"` / `&"dpo"` / `&"rlvr"` / `&"o1_rl"`。后三者是 reasoning RL 链 (reasoning 维度 capability 加成递增)。通过 `tech.get_loss_coefs` 查 train_coef / capability_coef。

## 8. dataset.source / kind / modality

- source: `&"open_source"` / `&"purchased"` / `&"collected"`
- kind: `&"pretrain"` / `&"posttrain"` (v2)
- modality (v7 PR-G, 单选): `&"text"` (默认) / `&"image"` / `&"audio"` / `&"video"` / `&"code"`. pretrain 训练时校验 `ds.modality ∈ model.input_modalities ∪ {text}`; posttrain 不校验. 旧存档默认 `text`. 详见 `数据集系统设计.md §1`.

## 10. campaign.target_product_types

`Array[StringName]` 字段, 元素从 `&"chatbot"` / `&"agent"` / `&"coding_agent"` / `&"multimodal_assistant"` 选; 空数组 = 全平台。

Legacy `campaign.target_segment`: `&"all"` / `&"chatbot_users"` / `&"agent_users"` / `&"open_source_devs"` 在 `Campaign.from_dict` 路径映射到新字段, 新代码不写。

## 11. modality

模型 / 产品的输入输出模态。当前: `&"text"` / `&"image"` / `&"audio"` / `&"video"`。模型默认 `[&"text"]` 进 / `[&"text"]` 出。

**与 dataset.modality 的区别 (v7 PR-G)**: dataset 是**单选** modality (含 `&"code"`); model.input_modalities 是 **Array** (text + 一个或多个非 text); product 维度同 model。pretrain 校验 `ds.modality ∈ model.input_modalities ∪ {text}` (text 总是允许 — 任何模型都需要文本骨架, 代码数据集是 text modality 的 subset, 在 coverage_tags 上区分)。

## 11bis. multimodal_method (v7 PR-G, PretrainDialog E 轴)

| `&"pixel_ar"` | `pixel_ar` | 1.15 | 1.10 |

### 16.3 供电方式

`&"grid"` / `&"solar"` / `&"nuclear"` / `&"wind"` / `&"coal"`。影响 `weekly_cost_per_card`, `efficiency_modifier`, `build_cost_modifier`。详见 PowerSupplySpec (基础设施 §1.5)。

### 16.4 模型 / 架构命名

| 范畴 | 命名 |
|---|---|
| 玩家自训模型 id | 玩家在 PretrainDialog 手动输入 (即 `model.display_name`, 同字符串); 校验 1-40 字符, 中英文 / 数字 / `-_.` 空格 |
| Legacy / 测试 fallback | `model_NNNN` 序号 (`_gen_id` 自动); 仅 TaskSystem 不传 display_name 时使用 |
| 架构族 (A 轴) | 动物化名: `ant_v* / bee_v* / octopus_v*/sparse/super_sparse / spider_* / crab_* / whale_*` |
| 注意力 (B 轴) | 学术缩写, 不走动物族: `mha_baseline / gqa / mqa / mla / hybrid_attention / infinite_attn` |
| Loss (C 轴) | 学术缩写: `ce_baseline / zloss / mtp / dpo / rlvr / o1_rl` (后三者 reasoning RL 链) |
| 公共开源模型模板 | `&"wolf_os" / &"hare_os" / &"finch_os" / &"lynx_code_os"` 等 (放 `resources/data/models/os/*.tres`) |
| NPC 公司 / 模型 | 全动物族 (与 NPC配置.md 同步) |

**为什么 attention / loss / engineering 量化节点 (`firefly_fp8` / `glowworm_fp4`) 例外**: 化名规范主要避免真实**商业品牌**; 学术机制名 (GQA / MLA / Z-loss / FP8) 与硬件相关精度缀 (`fp8`/`fp4`) 是技术行话, 玩家 (尤其 AI 圈) 通常知道, 套动物名反增认知负担。

## 风险点 (变更时同步检查)

- **`arch`**: 4 个系统 (Task / Research / Market / Monetization) 依赖; legacy `transformer_v*` alias 在 `tech_tree_system.gd::ARCH_ID_ALIAS`。
- **`target_product_types` vs legacy `target_segment`**: UserSystem `_campaign_targets_product` + Campaign.from_dict 同步映射。
- **`effect.kind`**: 新增事件 effect 时改 EventSystem `_apply_effect` 派发表。
- **codename 词库**: 新增模型 / GPU / 公司化名前先扫一遍 `resources/data/` 防止重名。
