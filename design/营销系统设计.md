# 营销系统设计 (MarketingSystem)

> **目的**: 管理"花钱推一个具体产品"的活动 (campaign)。每个 campaign 持续 N 周, 周度扣预算, 给 UserSystem 注入**指定 product** 的 `subscribers` 增长。
> **读者**: 营销面板 UI、UserSystem (消费 active campaigns 加成)。

---

## 1. 数据切片

owner = `MarketingSystem`。`GameState.campaigns: Array[Campaign]` (只有 active 的, finished / 孤儿自动删除)。

### Campaign 字段

| 字段 | 类型 | 语义 |
|---|---|---|
| `id`, `display_name` | StringName / String | 标识 |
| `weekly_budget` | int | 每周扣 cash 数 |
| `remaining_weeks`, `total_weeks` | int | 倒计时 + UI 进度条 |
| `target_product_id` | StringName | **v7 PR-F3 (2026-05)**: campaign 锁定的具体产品 (Product.id)。必填。换模型 = 把 product `bound_model_id` 改掉, campaign 仍指向同一 product, subscribers 池保留。 |
| `lead_id` | StringName | 可选 `marketing_lead`; UserSystem 读取 campaign_efficiency 加成。**v8 (2026-05)**: 启动时 `hiring.lock_lead`, 活动期间 lead 被独占 (一个 lead 同时只能带一个项目), 结束释放。 |
| `fake_score_level` | StringName | **真实性策略**: `none / low / medium / high`。表示营销文案对模型性能分数的夸大程度。默认 `none`, 旧存档缺字段也按 `none`。UserSystem 消费该字段: 获客倍率略升, 但目标产品每周留存率下降。 |
| `locked_staff` | Dictionary | **v8 (2026-05)**: 活动占用的营销员工 (role → count, `{marketing: N}`, N=2..8 随周预算)。结束时按此释放。 |
| `started_at_turn` | int | 审计 |
| ~~`target_product_types`~~ | **v7 PR-F3 弃用** | 旧存档仍可加载 (Campaign.from_dict 保留字段); 但新建 campaign 必须用 `target_product_id`, type 多选 UI 删 |

**已删字段**: ❌ `fame_boost` (v7 PR-F)。

**孤儿处理**: 玩家删了 product 后, 下一次 upkeep 相位扫 campaign, 发现 `target_product_id` 找不到对应 product → 直接 terminate (理由 `target_product_gone`), 当周**不扣预算**, 避免「花钱打空气」。

**旧字段兼容** (`Campaign.from_dict`): `monthly_budget → weekly_budget`, `total_months → total_weeks`; 旧 `target_segment` 还能加载, 但运行时 `target_product_id == &""` 的旧 campaign 在 UserSystem 里被当孤儿处理 (终止自己)。

**读档 ID 修复**: `_on_save_loaded()` 若发现重复 `campaign.id` 并重发 ID, 必须同步迁移该 campaign 所占 `lead_id` 的 `locked_by_task_id` (旧 ID → 新 ID)。否则玩家终止活动时 `hiring.release_lead` 会因 holder 不匹配失败, lead 继续处于 locked, 新建活动下拉框会把营销主管过滤掉。

## 2. 命令与信号

详见 `命令总线表.md §MarketingSystem` / `事件总线信号表.md §MarketingSystem`。要点:

- `marketing.start_campaign {display_name, weekly_budget, total_weeks, target_product_id, lead_id?, fake_score_level?}`: 错误码 `too_many_campaigns / unknown_lead / lead_specialty_mismatch / lead_busy / insufficient_staff / unknown_product / product_required / invalid_fake_score_level`。**并发上限 5** (`MAX_CONCURRENT_CAMPAIGNS`)。`target_product_id` 必须指向 `GameState.products` 里现存的 `Product.id`; `fake_score_level` 缺省为 `none`。
- **v8 员工占用 (硬性要求 + 锁定, 随预算缩放)**: 每个活动占用 `marketing` 员工, 数量 = `clamp(min_staff_per_campaign + floor(weekly_budget / budget_per_extra_staff), min, max)` (默认 2..8, 每 +¥10万/周 多 1 人, ¥60万/周+ 吃满 8 人; 见 `marketing/tuning.tres` 与 `MarketingSystem.required_staff_for_budget`)。没有足够空闲营销员工 → `insufficient_staff`, 不能开活动 (并发活动数被营销人头自然限制)。指定 lead 时 lead 必须 idle, 否则 `lead_busy`。lead + 员工在活动期间锁定, 终止 / 自然结束 / 孤儿 三条路径都释放。
- `marketing.terminate_campaign {campaign_id}`: 立即停止, 当周已扣不退。
- 信号: `campaign_started(id)` / `campaign_terminated(id, reason: player_terminated | finished_naturally | target_product_gone)` / `campaign_progress(id, remaining, total)`。

**约定**: start 不立即扣钱; 第一次扣在最近一次 `phase_started(upkeep)`。terminate 立即停止。孤儿 campaign (target product 找不到) 在 upkeep 扫描时**先 terminate 再扣预算**, 故当周也不扣。

## 4. NewCampaignDialog 字段与预览

`scenes/ui/new_campaign_dialog/new_campaign_dialog.gd`, 参照 `NewDatacenterDialog` 模式 (ConfirmationDialog 子类)。

| 字段 | 控件 | 校验 |
|---|---|---|
| 名称 | LineEdit | 可空, 空则用 `"Campaign #N"` |
| 周预算 | SpinBox | > 0, cash ≥ 1 周预算 |
| 周数 | SpinBox, 默认 13 | ≥ 1 |
| 目标产品 | **OptionButton** (v7 PR-F3) | 必选, 来自 `GameState.products`; 显示 "产品名 (type · bound_model)" |
| 性能分数表述 | OptionButton | `真实 / 轻微夸大 / 中度夸大 / 高度夸大`; 写入 `fake_score_level`。 |
| Lead | OptionButton | 可选; 非 `marketing_lead` → `lead_specialty_mismatch` |

**预览** (根据目标 product 的 type 切换显示): 展示总投入、预计 CAC、预计每周拉新量、真实性策略带来的获客倍率与留存惩罚。订阅产品的拉新单位是付费用户数, API 产品的拉新单位是 API 需求 (tokens/周)。S 级 `marketing_lead` 可显著降低 CAC。具体折算系数 (转化率 / lead 加成 / API token 换算) 见 `user_system.gd`。

**创始人出身加成**: 预计每周拉新量 = `budget × 转化率 × lead_mult × founder_mult`, 其中 `founder_mult = FounderSystem.user_growth_multiplier()` (网红出身 ×1.3, 其余 1.0)。该系数实际由 `UserSystem` 施加到本周正向用户净增, 营销拉新是其一部分, 故计入预览使显示与实际一致; 系数 `≠ 1` 时预览/活动卡片显示「创始人 ×N」后缀, 营销 tab 顶部显示加成提示 (见 `出身系统设计.md §5`)。

**确认按钮 disable 条件**: `weekly_budget ≤ 0` / `total_weeks ≤ 0` / `cash < weekly_budget` / `campaigns.size() ≥ MAX_CONCURRENT_CAMPAIGNS` / 玩家没产品 (列表空)。

## 5. 关键算法

### 5.1 周度推进 (upkeep 相位)

每周扫一遍 `campaigns`: 先做孤儿检查 (target product 已删 → 先 terminate, 当周不扣预算), 否则 `economy.spend(weekly_budget)`、`remaining_weeks -= 1` 并 emit `campaign_progress`; 倒计时归零则自然结束并 emit `campaign_terminated`。

### 5.2 与 UserSystem 的契约

UserSystem `_marketing_attract(p)` **直接读** `GameState.campaigns` 切片 (高频派生计算, 走命令开销过大), 把所有 `target_product_id == p.id` 的 campaign 周预算按转化率折成拉新量, 在场 `marketing_lead` 再乘一个 ability 加成, 结果直接加到 `p.subscribers`。订阅产品 1 单位 = 1 个付费用户; API 产品 1 单位 = `API_TOKENS_PER_SUB_PER_WEEK` tokens/周。

**CAC 在所有 budget 量级下恒定**: 多花钱拉更多池单位, 不存在"规模溢价"。基础 CAC = `1 / MARKETING_CONVERSION_RATE`, 转化率 `MARKETING_CONVERSION_RATE` 在 `user_system.gd` (由 `marketing/tuning.tres` 装配); S 级 marketing_lead 进一步降低有效 CAC。

**虚假性能分数策略**: `fake_score_level` 在获客侧乘以小幅倍率, 在留存侧作为目标产品 weekly_rate 的负项。活动结束后两者都消失。

| `fake_score_level` | UI 文案 | 获客倍率 | 留存惩罚 |
|---|---|---:|---:|
| `none` | 真实表述 | ×1.00 | 0.00 |
| `low` | 轻微夸大 | ×1.05 | -0.10 |
| `medium` | 中度夸大 | ×1.12 | -0.40 |
| `high` | 高度夸大 | ×1.25 | -1.00 |

多条活动同时命中同一 product 时, 获客倍率随各自活动单独计算; 留存惩罚按活动累加后夹到 `[-1.0, 0.0]`, 避免多条高度夸大活动把 rate 推到小于 -100%/周。`none` 不改变旧行为。

**v7 PR-F3 关键语义**: campaign 一对一锁定 product。换模型 = `product.bound_model_id` 改掉, campaign 不动, 池保留 — demand 自动指向新 model (UserSystem `_recompute_token_demand` 按 `product.bound_model_id` 汇总到 model)。要并行推 N 个产品就开 N 个 campaign (上限 5)。

## 7. 营销 tab UI (marketing_view, v7 PR-F3)

`scenes/ui/views/marketing_view/marketing_view.gd`, 仿 `product_view` / `event_view` 模式。main.gd 把 `_render_marketing_tab` 改成挂载这个 view 并调 `refresh(data)`。

界面分两块: 顶部「启动新活动」入口 (展示并发上限, cash 不足时禁用), 下方「进行中」campaign 卡片墙 (每卡含进度、目标产品、lead、预计每周拉新量与终止按钮)。进度文本使用跨平台稳定的 ASCII 形式 `38% (5 / 13 周)`, 不用 `█` / `░` 等块状字符模拟进度条, 避免 Windows / macOS 字体 fallback 时出现方块或乱码。卡片墙用 **`HFlowContainer`** 横向流式排布 (与 hiring / tasks 视图一致), 不用 `VBoxContainer` — 否则卡片会竖成一条。每张 campaign 卡使用 `assets/sprites/ui/marketing/campaign.png` 作为通用营销活动缩略图, 经 `IconRegistry.marketing_icon(&"campaign")` 接入; 缺图或加载失败时仍回退到 `Avatar` 的 seed/glyph 占位, 不阻塞 UI。view 自身不读 GameState, 由调用方把所需数据转成 dict 传入 `refresh(data)`; 字段结构以 `marketing_view.gd` 为准。新建活动对话框展示「需占用 N 名营销员工 (空闲 M)」, 不足时禁用启动。

信号: `new_campaign_pressed()` / `terminate_campaign_pressed(campaign_id)`。
