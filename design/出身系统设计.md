# 出身系统设计

**一句话目的**：游戏从一个起始页进入；新游戏时玩家给自己和公司取名，并选择一种「出身」，出身在整局游戏里持续提供一组优势与劣势。

## 1. 起始页 (StartScreen)

`scenes/start_screen/start_screen.tscn` 是工程的 `run/main_scene`（取代原来直接进 `main.tscn`）。

菜单按钮：

- **新游戏** — 打开 `NewGameDialog`（见 §2），确认后进入 `main.tscn`。
- **继续游戏** — 读取最近一次存档（优先 `autosave`，否则按 `saved_at` 取最新），直接进入游戏；无任何存档时按钮禁用。
- **读取存档** — 打开既有的 `SaveLoadDialog`，读档成功后进入游戏。
- **设置** — 语言切换 + 自动存档开关。
- **退出游戏** — `get_tree().quit()`。

进入游戏统一走 `_enter_game()` → `change_scene_to_file("res://scenes/main/main.tscn")`，并用一个幂等标志位避免重复切换。

**视觉（hero 欢迎页）**：起始页不是信息密集控制台，而是一张品牌欢迎页，目标"专业 + 吸引人"，所以用比控制台正文更大的 hero 字号档（见 `UI视觉系统设计.md §2`）。布局：

- **背景**：浅灰底（`BG_BASE`）上绘制一层抽象办公室 / 数据中心夜景：底部是低对比的楼体、窗格与服务器灯带，中景有细线电路轨迹，远景保留大面积留白。装饰只用 `UITheme` token 的黑灰白与低透明语义色，不出现真实品牌或写实模型名。
- **主操作面板**：左侧一张半透明白色大圆角面板（`BG_SURFACE` 高透明度 + `R_LG` + 柔和投影），不再是孤立小白卡。面板内自上而下：品牌标记（统一 `UITheme.draw_brand_mark`）→ 双色字标 "AGI **Inc**"（`FS_HERO`，"Inc" 取次级灰）→ 副标题（`FS_HERO_SUB`）→ 菜单按钮 → 状态行 → 版本号。
- **右侧仪表预览**：主面板右侧放一块纯装饰的产品仪表预览（小型排行榜 / 算力曲线 / 任务轨道的抽象线框），用于让首屏立刻传达"经营 AI 模型公司"的题材。该预览不承载真实游戏状态，不可点击，不参与存档或 CommandBus。
- **按钮层级**：主操作「新游戏」用实心炭黑主按钮（filled），「继续 / 读取」用描边次按钮（outlined），「设置 / 退出」用幽灵按钮（ghost，并排一行），靠视觉权重区分主次。按钮高度由 stylebox 的内容边距 + hero 字号自然撑出，不再用控制台的矮按钮。

按钮文案、可用性逻辑（继续禁用 / 最近存档优选）与下方各节一致，hero 化只改观感不改行为。

**返回主菜单**：游戏内顶栏「设置」弹窗里有「返回主菜单」入口（起始页打开同一弹窗时该入口隐藏，见 `国际化设计.md §11.0`）。点击先弹确认框，确认后 `main.gd` 的 `_on_return_to_main_menu()` 先 `Save.write(Save.AUTOSAVE_SLOT)` 把当前进度写入 autosave（玩家可从「继续游戏」恢复，不会误丢进度），再 `change_scene_to_file("res://scenes/start_screen/start_screen.tscn")` 切回起始页。切回后不 `reset` `GameState`（下次「新游戏」会自行重置）。测试运行（`_is_test_run()`）下跳过真正切场景，避免卸载测试 runner。

**自动化兼容**：当 `AGI_AUTOPLAY` / `AGI_SCREENSHOT` 环境变量存在时，起始页不显示菜单，直接用默认名字（创始人 / Scaling Up）、无出身开一局并切场景，使端到端截图与自动演练流程不受影响。

## 2. 新游戏对话框 (NewGameDialog)

`scenes/ui/new_game_dialog/new_game_dialog.gd`（脚本型 `AcceptDialog`，与 `SaveLoadDialog` 同构）。

- 两个输入框：**你的名字**、**公司名字**，均为必填。任一输入框为空或只有空白时，「开始游戏」禁用；即使代码路径直接触发确认，也不得发出 `start_requested`。
- 输入框占位只提示输入内容（如「请输入创始人名称」「请输入公司名称」），不再把「创始人」「Scaling Up」当作可自动套用的默认开局名。自动化路径（`AGI_AUTOPLAY` / `AGI_SCREENSHOT`）仍由起始页单独使用默认名字，避免影响截图与端到端脚本。
- **出身选择**：三选一，列出每种出身的人设、优势文案、劣势文案。
- **公司标志选择**：从一组**确定性生成的实心品牌标记**（`assets/sprites/ui/brand/brand-NN.png`，抽象 logo 符号，符合化名规范）里点选一个，作为顶栏品牌标记。这批图由 `tools/art/build_deterministic_ui_icons.py` 绘制，不走 AI 抠图，保证新游戏网格与顶栏小尺寸下不会出现透明缺块。第一个选项是默认的抽象「A」标记（id 为 `&""`），不选等于沿用它。可选 key 由 `IconRegistry.company_logo_keys()` 给出；贴图由 `IconRegistry.company_logo_texture()` 取，`UITheme.draw_company_logo` 把它叠在**浅灰圆角底**（`BG_ELEVATED`，配浅色 UI；非炭黑）上绘制。缺图 / 旧档的程序化形状 id / 未知 id 都回退到默认「上升 A」（深色 A + 绿色横杠，浅底上可读）。起始页 hero 的 `draw_brand_mark` 仍是炭黑底 + 白 A（app 图标），两者共用 `_draw_a_glyph` 字形。
- **个人头像选择**：从一组**创始人专属头像**（`assets/sprites/ui/founder/avatar-NN.png`，写实半身**透明抠图**，与招聘 lead 肖像同一画风、中性调色）里点选一个，作为「玩家自己」这位 lead 的头像。美术未就位时头像组件回退到按名字哈希的色块 + 首字母，不阻塞选择。
- **布局（三栏，做大）**：对话框放大到约 1080×620（仍不超过 1280×720 最小窗口），内容分三栏并排：
  - **左栏**：你的名字 / 公司名字两个输入框 + 出身选择（三张出身卡，竖排，**整卡点选**——与标志/头像瓦片一致，不再每卡一个按钮；过高时自滚）。
  - **中栏**：头像选择网格。
  - **右栏**：公司标志选择网格 + **实时预览卡**（按当前选择绘制：标志 + 公司名 + 创始人名/出身 + 头像），改名 / 换标志 / 换头像即时反映。
  - 「开始游戏」按钮始终可见（出身卡过高时只滚左栏，不滚整窗）。
- **视觉与可读性（2026-05 polish）**：三栏不再裸排在弹窗里，而是放进一张浅色 shell 面板；每栏用独立 `PanelContainer` 分区，标题使用 bold + `FS_LG`，区段副标题/说明不低于 `FS_BASE`。出身卡使用更明确的选中态（2px 深色描边 + 浅灰底），优势/劣势用语义色但保持足够对比；头像区做画廊式大瓦片，Logo 区做紧凑品牌格；右栏预览卡放大头像与标志，作为“确认身份”的视觉锚点。目标是在 1280×720 最小窗口下，中文字体不糊、不灰、不卡边。
- 「开始游戏」确认后发出 `start_requested(player_name, company_name, origin_id, company_logo, founder_avatar)`，由起始页写入 `GameState` 并进入游戏。

## 3. GameState 字段

新增以下随存档持久化的字段（owner：无系统，开局一次性写入）：

- `player_name: String` — 玩家名字。
- `company_name: String` — 公司名字，顶栏标题显示它。
- `founder_origin: StringName` — 出身 id（`&"scientist"` / `&"entrepreneur"` / `&"influencer"`，旧档或默认开局为 `&""`）。
- `company_logo: StringName` — 公司标志 id（`IconRegistry.company_logo_keys()` 里的某个 `brand-NN`，旧档或默认开局为 `&""` = 抽象「A」标记）。顶栏品牌标记按它绘制；旧档里的程序化形状 id（`node_teal` 等）仍可序列化，但绘制时回退到默认 A。
- `founder_avatar: StringName` — 创始人头像 key（`avatar-NN`，旧档或默认开局为 `&""`）。创建「玩家自己」这位 lead 时写到 `Lead.avatar_id`。

`reset()` 清空为默认值；`to_dict()` / `from_dict()` 序列化。旧存档缺这些 key 时回退为空，等价于「无出身 / 默认标志 / 哈希头像」，不破坏兼容。

`Lead` 资源新增 `avatar_id: StringName`（默认 `&""`）：非空时头像取 `assets/sprites/ui/founder/<avatar_id>.png`，空时沿用按 `lead.id` 哈希分配的多元肖像池（见 `招聘系统设计.md` / `IconRegistry.lead_texture`）。目前只有「玩家自己」会带 `avatar_id`，其余 lead 仍走肖像池。

## 4. 三种出身

数值放在 `resources/data/founders/*.tres`（`FounderOriginSpec` 资源），由 `FounderSystem` 自动加载并暴露访问器。每种出身**有得有失**：

| 出身 | 优势 | 劣势 |
|---|---|---|
| **ML 科学家** scientist | 人脉广，人才池每 4 周刷新时 S 级科学家出现率大幅提升 | 不擅长讲商业故事，投资人给的估值与融资额打折 |
| **连续创业者** entrepreneur | 与投资人关系好，开局即解锁 seed 轮，每次融资额与估值翻倍 | 顶尖研究者对「商人主导」的公司兴趣低，S 级人才更难刷出 |
| **知名网红** influencer | 自带流量，发布的产品用户增长更快 | 投资人质疑「流量泡沫」，估值与融资额打折（折扣最重） |

### 4.1 加成口径

`FounderOriginSpec` 字段（中性默认值表示「无影响」）：

- `s_tier_weight_bonus: float`（默认 `0`）— **加性**叠加到人才池抽取权重里的 S 档，再对四档重新归一化。用加性而非乘性，因为早期 cash 段 S 基础权重为 0，乘法无效。
- `funding_multiplier: float`（默认 `1.0`）— 乘到估值与（随机滚出的）融资额上。
- `user_growth_multiplier: float`（默认 `1.0`）— 乘到产品每周**正向**用户净增（负增长不放大，劣势只体现在估值）。
- `seed_round_unlocked: bool`（默认 `false`）— 为真时 seed 轮融资开局即满足条件。

每种出身的具体数值见 `resources/data/founders/*.tres`，本文档只描述字段语义。

## 5. 系统接入点

`FounderSystem`（autoload，`scripts/systems/founder_system.gd`）加载三份 spec，按 `GameState.founder_origin` 返回当前 spec；未知 / 空 id 返回全中性 spec。

- **HiringSystem** — `_draw_level()` 选定 cash 段后，对该段 `weights` 做 `s_tier_weight_bonus` 加性调整并归一化（`bonus == 0` 时原样返回，零行为变化）。
- **EconomySystem** — `_compute_valuation()` 结果乘 `funding_multiplier`；`_on_start_funding_round()` 对**随机滚出**的融资额乘 `funding_multiplier`（显式传入的 amount 不乘，保留测试/CLI 语义）；`_funding_round_conditions_met(&"seed")` 在 `seed_round_unlocked()` 为真时直接通过。
- **UserSystem** — `_resolve_per_product()` 算出本周净增 `delta` 后，若 `delta > 0` 乘 `user_growth_multiplier`。
- **UI 显示**（网红等加成 `≠ 1` 时）：营销 tab 顶部显示「创始人加成 · 新增用户 ×N」提示；新建活动对话框预览与营销活动卡片的「预计每周拉新」都把该系数计入，使显示人数与实际增长一致（营销拉新是本周正向 `delta` 的一部分，照样被放大）。见 `营销系统设计.md §4`。
