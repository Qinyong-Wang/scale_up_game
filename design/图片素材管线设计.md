# 图片素材管线设计

> **目的**: 定义 *Scaling Up* 图片素材从生成、清理、验收到接入 Godot 的 harness 契约, 让 AI 生成图变成可追溯、可测试、可复用的游戏资产。
> **读者**: 生成 UI 图标 / 卡片缩略图的人; 维护 `tools/art/` 与 `assets/sprites/ui/` 的人。
> **状态**: 🟡 v0 先落地本地处理 harness; UI 绑定留后续迭代。

---

## 1. 目标与非目标

### 1.1 目标

- 建立一条稳定链路: **人工/agent 写 prompt → 生成 raw 图 → 本地确定性后处理 → QA meta → 接受后的 PNG 进 Godot**。
- 第一阶段服务现有浅色 Dashboard UI: 模型、数据集、产品、基建、事件等卡片头像 / 缩略图 / 小图标。
- 所有接收进工程的图片都能追溯到 prompt、原图、处理参数和 QA 结果。
- 图片资产继续遵守化名规范: 文件名、manifest、UI 文案不出现真实商业品牌或真实模型名。
- Godot 运行时只依赖接受后的 `assets/sprites/**.png`; 不读取工具 run 目录。

### 1.2 非目标

- 本阶段不把 UI 改成像素风; 仍遵守 [UI视觉系统设计.md](UI视觉系统设计.md) 的现代扁平浅色科技感。
- 不做任何非 UI 平面素材, 包括背景、角色动画和可交互画面素材。
- 本阶段不在游戏内调用图片生成模型; harness 是开发期工具。
- 本阶段不做大规模素材库管理器, 只定义文件契约与处理脚本。

## 2. 风格与命名

### 2.1 风格

第一批素材使用 `clean_hd_ui`:

- 2D clean HD / editorial tech icon / product thumbnail 风格。
- 背景生成时使用纯 `#FF00FF` 洋红, 方便本地去背。
- 接受后的 PNG 必须透明背景、边缘干净、主体居中。
- 避免复杂阴影、文字、logo、真实品牌暗示、低清噪点。
- 图标在 40×40 到 96×96 的 Avatar / Card header 中仍要可读。

### 2.2 命名

- 文件名只用英文小写、数字、连字符: `model-sparrow`, `gpu-cypress`, `event-funding`。
- 模型 / 架构 / NPC 公司沿用动物族; GPU / 供电 / 机房沿用植物族。
- 现实品牌、真实模型名、真实 GPU 型号不进入文件名、prompt 存档、manifest、UI 文案。
- 设计文档可在注释里解释现实类比, 但素材文件和运行时数据不可出现。

## 3. 目录契约

```
assets/
  sprites/
    ui/
      models/          接受后的模型 / 架构图标
      datasets/        数据集来源 / 模态图标
      infra/           GPU / 机房 / 供电图标
      products/        产品类型图标
      events/          事件卡图标
      marketing/        营销活动通用缩略图

tools/
  art/
    process_asset.py   本地确定性处理 harness
    index.md           工具说明与命令索引
    runs/              生成 run 工作区, 不作为运行时资源
      .gdignore        禁止 Godot 导入中间文件
```

`tools/art/runs/` 推荐结构:

```
tools/art/runs/<asset_id>/
  raw.png
  prompt-used.txt
  pipeline-meta.json
  <asset_id>.png       单图处理结果, 或 sheet 模式下为透明合图
  frames/              可选: sheet 切出的帧 / 图标
```

接受流程是人工确认后把最终 PNG 放进 `assets/sprites/**`; raw、prompt、meta 可保留在 run 目录或随提交附带, 但运行时不依赖它们。

## 4. Harness 行为

`tools/art/process_asset.py` 只做确定性像素处理, 不写 creative prompt, 不生成最终美术。

### 4.1 输入

- `single`: 单张图标 / 缩略图。
- `sheet`: 固定行列的 UI 图标包, 例如 `2x2` / `3x3` icon pack。

共同输入:

- `--input`: raw PNG, 背景应为纯 `#FF00FF`。
- `--output-dir`: 输出目录。
- `--size`: 单个输出 PNG 的目标方形尺寸。
- `--fit-scale`: 主体占目标尺寸比例。
- `--component-mode`: `largest` 只保留最大连通主体; `all` 保留所有非透明像素。
- `--reject-edge-touch`: 任意主体触碰源格边缘时失败。
- `--prompt` / `--prompt-file`: 可选, 写入 `prompt-used.txt` 并登记到 meta。

### 4.2 输出

`single` 输出:

- `<asset_id>.png`: 透明背景、居中缩放后的最终图。
- `clean.png`: 去背后的整张清理图。
- `pipeline-meta.json`: 输入、参数、bbox、edge-touch、输出路径。
- `prompt-used.txt`: 有 prompt 时写入。

`sheet` 输出:

- `<label>.png`: 每格切出的透明图。
- `sheet-transparent.png`: 按行列重新拼回的透明合图。
- `pipeline-meta.json`: 每格 source_box、bbox、component_count、edge_touch、输出路径。
- `prompt-used.txt`: 有 prompt 时写入。

### 4.3 QA 标准

一次 run 被认为可接受, 必须满足:

- 输出 PNG 可被 Godot `Image.load` 正常读取。
- 四角 alpha 为 0, 背景透明。
- 主体没有触碰源图或源格边缘。
- 主体 alpha 不能被背景抠像误伤: 头像脸 / 身体、奖杯把手 / 细柱、收藏品边框等连续结构不得出现明显透明裂缝、缺块或被削断。
- 接受前必须用 checkerboard contact sheet 看整批透明 PNG; 若出现残留背景色块、大片投影色块、主体内部缺块, 视为语义 QA 不通过。
- `pipeline-meta.json` 可解析, `ok=true`, 且记录了全部处理参数。
- sheet 模式下 label 数量不超过格子数, 空格子显式 skip。
- prompt 存档存在, 或 manifest 明确 `prompt=""`。

### 4.4 安全去背模式

AI 生成器经常把 prompt 中的纯洋红背景画成粉色、灰色、浅蓝或带渐变的棚拍底。旧流程用高阈值全局抠色可以清掉背景, 但会把与背景相近的主体高光也吃掉, 造成头像 / 奖杯 / 收藏品"缺一块"。

因此所有带人物、奖杯、奖章、收藏品、标志、公益 / 模拟图标等容易有细结构或高光的批次, 后处理应优先使用 **sampled_bg**:

- 从画面边缘采样背景参考色。
- 只移除与边缘连通、且接近背景参考色的像素。
- 不再对整张图做高阈值全局 chroma key, 避免主体内部的粉色 / 紫色 / 高光被误删。
- `defringe` 只允许小半径使用; 头像和奖杯默认 `0`, 标志 / 公益图标可用 `1`, 避免细把手、绶带和头发被腐蚀。

只在 raw 确认为干净纯洋红背景、主体颜色远离洋红时, 才使用旧 `chroma` 模式。

### 4.5 缺块修复策略

2026-05 回归发现: 即使使用 `sampled_bg`, 如果 `background_tolerance` 设到 `75–85`, 紫 / 蓝 / 灰棚拍底会把浅色主体也纳入边缘连通背景。典型坏例是白色聊天气泡被抠空, 只剩描边和内部小点。

因此后续 accepted PNG 的修复默认遵守:

- **优先复用 raw 重处理**, 不直接手修 PNG; 只有 raw 本身主体生成错误时才换 seed / 重生成。
- `sampled` 的默认容差上限为 `45`; 需要更高容差必须先在 checkerboard contact sheet 里确认不会吃掉主体。
- 修复脚本必须把使用的容差、defringe、输出路径写进 report, 方便回溯。
- 接受前看 contact sheet 时, 除 §4.3 标准外, 额外重点看**浅色大面主体** (气泡、纸张、奖章绶带、画框内高光、人物衣服) 是否被抠成透明裂缝或空洞。

### 4.6 精确 UI 标志的确定性生成

公司标志 (`assets/sprites/ui/brand/*.png`) 与任务图 (`assets/sprites/ui/task/*.png`) 会在新游戏选择网格、顶栏 28px monogram、任务卡 Avatar 里反复缩放显示。它们不需要复杂 AI 插画细节, 反而最怕线稿太细、组件太碎或抠图误删, 因此这两类改走**确定性 raster 生成**:

- 用 `tools/art/build_deterministic_ui_icons.py` 直接绘制 128×128 RGBA 透明 PNG, 不经过 AI raw 图、洋红底或去背。
- 图形必须是实心剪影 / 厚描边, 不接受只靠细线、碎片、高光表达的标志。
- 视觉上保留浅色 UI 可读的多色科技感, 但所有主体必须是连续、不贴边、可在 28–48px 识别的图形。
- 机械 QA 增加 alpha 填充率门槛: `brand` 与 `task` 图标的有效 alpha 像素面积不少于 bbox 面积的 45%, 且有效 alpha 像素数不少于 3600。低于该值通常意味着小尺寸下会像"缺块"或"只剩线稿"。
- 若未来重新启用 AI 生成公司标志或任务图, 接受前也必须通过同一填充率门槛; 不通过则回到确定性生成。

## 5. 接入 UI 的路径

现有 `Avatar` 已支持 `Texture2D`。后续接入顺序:

1. 给资源 / view model 增加可选 `icon_path` 或 `thumbnail_path` 字段。
2. 调用方 `load(icon_path)` 后传给 `Card.avatar.texture`。
3. 缺图时继续走 seed 颜色 + glyph fallback, 不影响老存档。
4. 新图标接入必须补 UI 组件或 view 测试, 覆盖有 texture / 无 texture 两条路径。

第一批建议:

- `assets/sprites/ui/models/`: 架构族 / 模型卡缩略图。
- `assets/sprites/ui/infra/`: `cypress` / `maple` / `bamboo` 族图标, 供电与机房图标。
- `assets/sprites/ui/products/`: chatbot / agent / multimodal / coding agent。
- `assets/sprites/ui/datasets/`: text / image / code / audio / video 模态。
- `assets/sprites/ui/marketing/`: campaign 等营销界面通用缩略图。
