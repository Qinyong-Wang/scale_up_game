# 竞争对手系统设计 (MarketSystem)

> 系统模块名仍为 `MarketSystem` (代码/类名不动), 设计文档对外语义改为"竞争对手"。
> **目的**: 维护 NPC 竞争对手 (23 家) 与排行榜 (`total` 主榜 + `closed_source` / `open_source` 双展示榜 + 5 个细分榜)。每周结算榜单, 名次靠前的玩家模型驱动 UserSystem 的需求池演化。
> **v8 PR-H (2026-05) 重大变更**: **NPC 改为"产品时间线"驱动**, 每家 NPC 在固定 turn 发布一个具名模型 (含 `.1 / .5 / .7` 中间版本), NPC 当前能力 = "已发布的最新模型"的 capability, 而不是随时间漂移的浮点数。配套删除: ① `step_size / step_period / growth_curve / perturbation / next_step_turn` 阶跃增长机制 ② `distillation_timers` 与所有蒸馏追赶逻辑 ③ `market.boost_npc / market.perturb_npc` 这两个时间锚点事件用的命令。
> **v7 PR-F (2026-05)**: fame 字段已完全删除, 排名通过 `market.get_rank` 暴露给下游 (UserSystem / EventSystem) 直接消费。
> **读者**: 排行榜面板 UI、UserSystem (消费 rank 驱动需求池)、EventSystem (融资 offer 等触发条件)、HiringSystem (rank 影响候选池)。

---

## 0. 关键设计点

1. **8 棵榜单**: 
   - **`total`**: 用户视角的主榜, 合并所有 published 玩家模型 + 所有总榜 NPC, 按 5 轴 capability 总分排名。NPC 是否属于总榜仍由 `board_membership` 中的 `closed_source` / `open_source` 标记决定, 以免专精公司挤满主榜。**唯一驱动用户需求的总榜**。
   - `closed_source` / `open_source`: 仅做"展示榜单" (开源 / 闭源生态影响力), **按 `is_open_source` 自动收录所有已首发 NPC**, 不再要求 `board_membership` 显式写入 source 榜标签; **不进入用户决策**。
   - 5 个细分榜 (`sub_general / sub_code / sub_reasoning / sub_multimodal / sub_agent`)。
2. **NPC 时间线驱动 (v8 PR-H)**: 每家 NPC 在 .tres 里预先写好一整条"产品时间线" — 大版本 (`X / X+1` 全量 pretrain) + 中间版 (`X.1 / X.5 / X.7` posttrain / RLHF / multimodal 后训 / reasoning RL), 平均 ≥ 2 release/年。每周 action 相位, 系统把 turn ≥ release_turn 的最新一个 release 标记为该 NPC 的 `current_release`; 排榜时以 current_release 的 capability 入榜。NPC release 之间能力**纹丝不动** — "今年谁发了什么"由玩家肉眼可见的事件 (`npc_released`) 推动叙事感。读档后必须重新从 `resources/data/npcs/*.tres` 装载 NPC roster, 按存档 turn 静默推进 current_release 并重排当前排行榜, 避免旧存档里的过期榜单快照 / 缺 NPC roster 覆盖新资源表。
3. **集群与训练参数真实化**: 每条 release 在 .tres 里写明使用的 GPU 化名 (`cypress_t0/t1/t2/t3 / bamboo_t1/t2/t3/t4 / 未来代`) + 卡数 + 训练周数 + params(B) + dataset_tokens(B), 锚定到现实里"那一年那个量级的集群训练出 SOTA"。后期上探到 6M+ 卡。详见 `NPC配置.md` 与 `resources/data/npcs/*.tres`。
4. **NPC 总分上限 (v13, 2026-06)**: 竞争对手 release 的 5 轴合计不超过 `NPC_TOTAL_CAP = 1100`。生成脚本 `tools/build_npc_timelines.py` 在写 `.tres` 前按比例缩放超过上限的五轴能力, 保留强项/弱项相对结构。玩家模型不使用这个上限; 这是 NPC 时间线的平衡护栏, 防止后期榜单跑到 1500+ 而玩家合法上限只有约 1000-1100。
5. **排行榜 entry 显示**: NPC 条目显示**"模型名 — 公司名"** (例 "Orca-5 — OrcaLab"), 玩家条目显示"模型名" (公司名留空)。`LeaderboardEntry` 新增 `company_name` 字段。
6. **开源滞后**: 开源 NPC 的 release 时间线相比同代闭源 SOTA 落后 20-40 周; 在 .tres 时间线里直接体现 (不需运行时计算)。
7. **删除蒸馏机制**: 玩家成为某榜第一不再触发"NPC 强行追上"。原因: NPC 自带固定时间线, 玩家阶段性领先是设计目标 (例如 2025-2027 玩家挤进 top 3); 强制蒸馏会与 NPC 时间线节奏冲突, 也让玩家有"被惩罚领先"感。
8. **downloaded_os 不重复入公开榜**: 玩家部署 / 下载公共开源 release 后, 系统会物化一个 `provenance=downloaded_os` 的本地 Model 用于 API、产品、定价和营收, 但它只是 NPC release 的运营副本, **不作为玩家原创模型进入公开排行榜**。公开榜只显示原 NPC release 一条, 避免 "Wolf-5" 同时以玩家模型和 NPC 模型出现。UserSystem 的产品增长仍可通过 `get_rank_vs_npcs` 把 downloaded_os 模型按 capability 与 NPC 比名次。

### NpcModelRelease (v8 新增)

`resources/NpcModelRelease.gd`, 一条 release 一个 Resource 实例 (在 NPC .tres 里以 sub_resource 形式内联):

| 字段 | 类型 | 语义 |
|---|---|---|
| `id` | StringName | 例 `release_orca_5` |
| `display_name` | String | 例 "Orca-5" |
| `release_turn` | int | 何时发布 (= 何时这个 release 开始计入排行榜) |
| `capability` | Dictionary | 5 轴 {general, code, reasoning, multimodal, agent} |
| `release_kind` | StringName | `&"pretrain"` (大版本) / `&"posttrain"` / `&"rlhf"` / `&"multimodal_posttrain"` / `&"reasoning_rl"` / `&"tool_use_posttrain"`。仅作为叙事/UI 标签, 不影响排行榜 |
| `cluster_gpu_id` | StringName | 训练用 GPU 化名, 例 `cypress_t2` |
| `cluster_gpu_count` | int | 卡数, 例 16000 |
| `training_weeks` | int | 训练周数, 例 25 |
| `params_b` | float | 参数量, B 为单位, 例 1500.0 (1.5T) |
| `active_params_b` | float | MoE 激活参数, dense 模型 = params_b |
| `dataset_tokens_b` | float | 训练 token, B 为单位, 例 13000.0 (13T) |
| `arch_codename` | StringName | 架构, 例 `octopus_v2` (MoE) / `ant_v4` (dense) |

> `release_kind = pretrain` 必须填齐 cluster / params / tokens; 中间版 (`posttrain / rlhf / ...`) 这些字段可不填或填零, UI 显示"基于上一版的后训"。设计意图: `.tres` 既是榜单数据又是 lore — 玩家点 NPC 详情看到的"这家公司的发展线"。

## 2. 命令与信号

### 命令

- `market.preview_rank {model_id}`: 纯读, 扫描所有榜单, 返回该模型**命中的第一个榜单**上的名次 — `{ok, predicted_rank, board}` (不在任何榜上 → `predicted_rank = -1, board = &""`), UI 用。
- `market.get_rank {model_id, board_id}` → `{ok, rank}`: 玩家模型在指定榜的 rank (1-based), 不在榜上 → `rank = 0`。

### 跨系统公开 API (非命令总线, 直接函数调用)

- `MarketSystem.get_rank_for_model(model_id, board_id) -> int`: `market.get_rank` 的便捷只读版 (供 HiringSystem / UserSystem 热路径直接函数调用), 不在榜上返回 0。
- `MarketSystem.get_rank_vs_npcs(model_id, board_id) -> int` (v10): 把玩家模型**只与 NPC 竞争对手**比名次, 玩家自己其他已发布模型不计入。UserSystem 的产品用户增长用它, 保证公司不与自己竞争 (R&D 中保留一个更强但更贵的模型, 不应压低产品实际服务的便宜模型的名次); 公开排行榜 (`get_rank_for_model`) 仍把所有人一起排。不在榜上返回 0。`downloaded_os` 模型虽然不显示在公开榜, 但本函数仍按其 capability 计算相对 NPC 名次, 供产品增长使用。
- `MarketSystem.find_release(release_id: StringName) -> Dictionary`: 在 `npc_companies[*].model_releases` 里反查指定 release。返回 `{ok = true, npc = NpcCompany, release = NpcModelRelease}` 或 `{ok = false}`。`ResearchSystem` 的 `research.download_open_source` 与 `InfraSystem` 的 `infra.deploy_open_source_model` / `infra.preview_deploy_capacity` 都用它把"可下载/可部署的 OS 模型"映射到 NPC release。
- `MarketSystem.list_downloadable_releases(turn: int) -> Array[Dictionary]` (UI 用): 返回所有 `npc.is_open_source == true && release.release_kind == &"pretrain" && release.release_turn <= turn` 的 release 列表, 每条 `{npc_id, npc_display_name, release}`, 按 release_turn 升序。turn 0-214 时返空数组。

**v8 PR-H 已删命令**:
- ❌ `market.perturb_npc` (perturbation 字段废弃)
- ❌ `market.boost_npc` (NPC 由 .tres 时间线驱动, 永久加成机制改为"在时间线里多写一个 release")

**v7 PR-F 已删命令**:
- ❌ `market.add_fame` (fame 字段废弃)

### 信号

`leaderboard_resolved(turn)`, `player_rank_changed(board, old_rank, new_rank)`, **`npc_released(npc_id, release_id, release_turn)`** (v8 新增, 用于事件叙事 / 通知 UI / 触发 paradigm event 卡)。

**v8 PR-H 已删信号**:
- ❌ `npc_distilled(npc_id, board_id)` (蒸馏逻辑废弃)
- ❌ `npc_boosted(npc_ids, delta)` (boost_npc 命令已删)

## 4. 周度结算流程 (action 相位, v8 PR-H)

每周 action 相位顺序触发三步:

1. **推进 NPC current_release**: 遍历每家 NPC, 找出 `release_turn ≤ 当前 turn` 的最末一条 release, 标记为 `current_release`。若与上周不同 (即"本周有新 release") 则发 `npc_released(npc_id, release_id, release_turn)` 信号; NPC 的 5 轴能力同步刷新成该 release 的 capability。NPC 没有首发 release 时 (例如 OrcaLab 在 turn 70 之前) 不上榜。
2. **重排 8 棵 board**: 把所有 published 玩家原创模型与所有有 current_release 的 NPC 装入对应 board, 按 capability_score 降序排, 写回 `GameState.leaderboard`; 玩家在某 board 的 rank 变化时 emit `player_rank_changed`。`provenance=downloaded_os && source_release_id != ""` 的模型不入公开榜, 由对应 NPC release 代表。
3. **追加 history snapshot**, 截断到 `HISTORY_LIMIT` (= 36) 周。

**v8 PR-H 已删步骤**: ❌ NPC step jump、❌ 蒸馏倒计时与触发、❌ perturbation 衰减。

UserSystem 在自己的 `_resolve_action` 里 (顺序晚于 MarketSystem) 调 `market.get_rank` 拿名次, 把 rank → demand rate。

## 5. 核心逻辑

### 5.1 排榜评分

| board | NPC 入围条件 | 排名公式 |
|---|---|---|
| `total` | 在 `closed_source` 或 `open_source` ∈ board_membership 的 NPC | 5 轴 capability 求和 |
| `closed_source` | `is_open_source = false` 的已首发 NPC | 5 轴求和 |
| `open_source` | `is_open_source = true` 的已首发 NPC | 5 轴求和 |
| `sub_general` 等 5 个细分榜 | 该榜 ∈ board_membership (开/闭源都收) | 单轴 (general / code / reasoning / multimodal / agent) |

NPC `.tres` 数据由 `tools/build_npc_timelines.py` 生成。生成期会对超过 `NPC_TOTAL_CAP = 1100` 的 release 做五轴等比例缩放; MarketSystem 排榜时不再额外 clamp, 直接使用资源里的能力值求和 / 取单轴。这样排行榜、NPC 详情与可下载开源 release 的本地副本保持同一标尺。

玩家模型入围: `total / closed_source / open_source` 按模型的 `is_open_source` 分流, 细分榜任意开闭源都进。例外: `downloaded_os` 物化模型不入公开榜, 避免与来源 NPC release 重复显示。

### 5.2 LeaderboardEntry 装配规则

- 玩家条目: `entity_id = model.id`, `entity_type = &"player_model"`, `display_name = model.display_name`, `company_name = ""` (玩家公司名留空)。
- NPC 条目: `entity_id = current_release.id`, `entity_type = &"npc"`, `display_name = current_release.display_name`, `company_name = npc.display_name`, `capability_score` 来自 current_release.capability 按 §5.1 公式取。
- UI 渲染: 若 `company_name != ""`, 双行显示 — 模型名大字、公司名小字灰色; 否则只显示 `display_name`。

## 7. 与原 paradigm event 的关系

`paradigm_rlhf / paradigm_moe / paradigm_long_ctx / paradigm_reasoning_rl` 这 4 张时间锚点事件卡 (`资源/data/events/`) 原来通过 `market.boost_npc` 给 top NPC 加 capability。v8 PR-H 后:

- NPC 已经在 .tres 时间线里**直接发布**对应的范式跃迁模型 (例: OrcaLab 在 turn 300 发 Orca-4, 已经把 GPT-4 等价能力写入), 不再需要事件去 "补刀"。
- 4 张 paradigm 卡**改为叙事卡** (纯 UI flavor + 行业新闻 + 可选给玩家一次性提示 / 资源), 不再触碰 NPC 数据。
- `npc_capability_jump` effect kind **废弃**, 在 `事件库.md` 中标 deprecated。

详见 `事件库.md §技术突破`。

## 8. 排行榜 UI — 荣耀榜单 (LeaderboardView)

> 目标: 把"竞争对手" tab 从纯文本逐行 `Label` 升级成**有仪式感 / 成功感**的荣耀榜单。
> 实现: `scenes/ui/views/leaderboard_view/` (`leaderboard_view.gd` + `.tscn` + `leaderboard_row.gd`),
> 取代 `main.gd` 里旧的 `_render_market_tab` 文本渲染。和其它 tab 一致走"独立 view"模式。

### 8.1 视觉规则

- **荣耀榜单 (强化行)**: 每条 `LeaderboardEntry` 渲染成一行 `LeaderboardRow`:
  `[名次奖章] [头像] [名称 / 公司]  …  [得分] [奖励]`。
- **前 3 名奖章**: #1 金 (`RANK_GOLD`) / #2 银 (`RANK_SILVER`) / #3 铜 (`RANK_BRONZE`) 圆形奖章 + 名次数字;
  第 4 名起用中性灰底 (`BG_ELEVATED`) 数字, 不抢镜。**不用 emoji** (字体不保证, 见 UI视觉系统设计.md §10bis.4), 用色块奖章。
- **玩家行高亮 (成功感)**: 玩家自己的模型行 (`entity_type == &"player_model"`) 用浅蓝高亮底
  (`ACCENT_INFO_SUBTLE`) + 左侧蓝条 (`ACCENT_INFO`) + 名称旁「你」徽章 (`MARKET_YOU_BADGE`, badge kind=info);
  一眼能在榜单里找到自己。
- **顶部 hero 标题**: 复用 `SECTION_MARKET_LEADERBOARD` (含玩家总榜最佳名次), 把"你现在第几"放最显眼处。
- **榜单切换**: 8 棵榜走 toggle 按钮组 (HFlowContainer), 当前榜按钮按下态; 切换 emit `board_selected(board_id)`。
- **规则说明 / 空榜**: 复用 `MARKET_RULE_*` / `MARKET_EMPTY`。

### 8.2 数据契约 (main → view)

`LeaderboardView.refresh(data)` 是**纯渲染器**, 不调 `tr()` / 不持有业务系统引用; 文案与名次奖励
全部由 `main.gd::_build_leaderboard_view_data()` 翻译 / 计算好后传入 (与 ProductView 同 pull 模型):

```
{
  "header_title": String,                 # tr(SECTION_MARKET_LEADERBOARD) % best_rank
  "boards": [{"id": StringName, "title": String}, ...],  # picker, title 已 tr()
  "active_board": StringName,
  "rule_text": String,                    # tr(MARKET_RULE_PREFIX) + 该榜规则
  "empty_hint": String,                   # tr(MARKET_EMPTY)
  "entries": [{
     "rank": int, "display_name": String, "company_name": String,  # 玩家 company 留空
     "score_text": String, "reward_text": String,   # reward 来自 _rank_bonus_for_board, 展示榜为空
     "is_player": bool, "you_label": String, "seed_id": StringName,
  }, ...],
}
```
