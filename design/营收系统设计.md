# 营收系统设计 (MonetizationSystem)

> **目的**: 每周结算 API 营收 + 订阅营收, 把"用户需求"与"算力供给"相乘写入 cash。**纯结算器**, 无切片, 无玩家操作。
> **读者**: 财务面板 UI (营收来源拆解); 其他系统不读本系统状态。

## 3.1 UI 调用纪律 (capacity 单一来源)

`MonetizationSystem.compute_capacity_for_model(m)` 是 capacity 的**单一来源**。所有 UI / 提示 / 警告 (e.g. 产品 tab 算力池行 util_pct、`_any_model_capacity_short` 提示) 都必须直接调它, 不允许在 UI 层重写 capacity 公式 —— 否则结算和显示永远会对不上 (历史事故: v3 UI 用 SECONDS_PER_MONTH + 又乘了一次 engineering 树乘数, 导致饱和时玩家看到的 util 只有真实值的 ~23%, "显示没满但实际没营收")。

顶栏「算力」chip 同理: 走 `MonetizationSystem.total_effective_serving_tps()`, 它把"每个 published 模型的 `compute_capacity_for_model(m)` 折回 t/s 后求和 + OS / 未识别 deployed 的 dc 用原始 `dc.serving_tokens_per_sec` 兜底"。这样顶栏数字 = 算力池所有行的 t/s 之和, 玩家不会再看到顶栏与池子打架。**禁止**在 UI 直接 `Σ dc.serving_tokens_per_sec` —— 那会漏掉 arch.inference_coef 与 chief_engineer.product_throughput 两个乘数。

## 4. 时间单位约定 (2026-05 重要变更)

**1 turn = 1 week**, 所有"_per_month"字段在 weekly tick 下**隐式重解读为 "_per_week"** (字段名保留兼容旧存档与外部文案, 实际语义为 per-turn 量):

- `subscription_price` 隐式 ¥/周
- `tokens_per_user_per_week` 是订阅产品 token 需求权威字段; `tokens_per_user_per_month` 仅作为旧资源/旧存档 fallback, 虽然字段名保留但也隐式 tokens/周
- API demand 由 `api_product.subscribers × api_tokens_per_sub_per_week` 派生, 旧 `TOKEN_BASE_PER_FAME × fame × cap/100` 路径已删除
- `dc_capacity = dc.serving_tokens_per_sec × SECONDS_PER_WEEK` (= 604_800)

之前 v3 用 `SECONDS_PER_MONTH = 2_592_000` 每周结算月度数字, 导致 4× 营收 inflation; 2026-05 修正为 SECONDS_PER_WEEK + 隐式周价语义。

## 5. 核心公式 (`_settle_api` + `_settle_subscription`)

### 5.1 API 营收 (算力池 + subscription-priority 分配)

每个 published 模型 = 独立算力池。**订阅产品 demand 与 API 产品 demand 都按 capacity 截断 — 算力短缺时两边按比例缩减**。订阅 demand 先占, 剩余给 API; 但如果订阅本身就超过容量, 订阅营收按 `ratio = capacity / subscription_demand` 缩减, API 则归零。

v9 后 (2026-05): 把"订阅不受算力影响"改成"两边都按算力 cap 截断" — 原因是旧设计让玩家拿一张桌面 GPU 服务上千订阅用户也照拿营收, 失去基建/订阅平衡的核心循环。现在算力短缺直接砍订阅营收, 玩家必须扩容才能放大订阅规模。

按 published 模型逐个结算: 每个模型一个算力池 (= 部署它的数据中心算力之和, 折进 arch inference 系数与 chief_engineer 加成)。订阅 demand 先占算力, 剩余算力供 API; API demand 超出剩余算力时按比例 `ratio` 缩减, 服务不到的部分计入 `api_demand_lost`。营收 = 实际服务 token 量 × 模型单价。具体计算见 `monetization_system.gd` 的 `_settle_api`。

**关键纪律 (v7 PR-F, 2026-05)**:
- 玩家**开源发布**的自训模型 (`is_open_source = true`) 仍参与 API 结算; 价格不再硬钳位 (v6 PR-E 删 `max_os_api_price`), v7 PR-F 进一步: 整个 API 需求**改由 `api_product.subscribers` 池**驱动 (UserSystem 每周演化, 见 用户系统 §5)。本系统读 `GameState.api_token_demand[m.id]` = `Σ api_product.subscribers × API_TOKENS_PER_SUB_PER_WEEK`, UserSystem 已经把 rank/价格/营销/cliff 全折进去, 这里只看 capacity vs demand。
- ❌ **`Model.demand_multiplier` 字段已删** (v7 PR-F): 它的作用合并到 `api_product.subscribers` 同一池。
- 公共开源模型经 `research.ensure_open_source_release_published` 物化为 `provenance == downloaded_os` 的 published 模型后, 与玩家自训开源模型同样参与 API 结算; 是否有营收只取决于是否有 API 产品、API 需求、部署 capacity 与 `per_token_price`。
- `api_demand_lost` 仅算 API 那部分; 订阅 demand 即使超容量也不记 lost。

### 5.1bis API 营收硬上限 (TAM 天花板, 2026-05)

每个 **API 产品 (= 每个 published 模型, 一对一)** 的当周营收硬封顶:

```
rev = min(served × per_token_price, revenue_cap_per_week)
```

- 上限值放数据: `resources/data/products/types/api.tres` 的 `revenue_cap_per_week` (当前 `10_000_000_000`)。`0` = 无上限 (其余产品类型默认值, 当前只有 api 配了非零上限)。
- **per-product (= per-model) 粒度**: 公司总 API 营收仍随 published 模型数量线性叠加 (2 个模型最多 2×cap), 只是单个 API 产品不会无限膨胀。
- 这是**营收侧的市场饱和 / TAM 天花板**, 不是算力短缺: 封顶后超出的算力 / 需求白白浪费 (鼓励玩家把资源投向别处), **不**计入 `api_demand_lost` —— 那个桶只反映"算力不够导致服务不出去的 token"。
- 背景: 旧版 `rev = served × price` 在玩家把算力 + 需求池 + 单价同时拉满后能涨到上千亿/周, 失去后期张力。封顶给后期一个明确的"单模型已吃满市场"信号。

### 5.2 订阅营收

按 `bound_model` 分组: 每个 model 一个 capacity, 其下所有订阅产品共享同一个 `ratio = min(1, capacity / sub_demand)`。每个订阅产品营收 = `subscribers × subscription_price × ratio`, 即算力短缺时按比例打折。`bound_model` 不在 published 集合的孤儿订阅产品 `ratio = 0`, 营收清零。具体计算见 `monetization_system.gd` 的 `_settle_subscription`。

**算力短缺直接打折订阅营收 (v9, 2026-05)**: 旧版"订阅与算力无关"让玩家可以无限规模化订阅而不扩容, 失去资源管理乐趣。改成线性 ratio 截断后, 玩家扩 1 张卡 → 订阅营收能放大相应倍数, 而不是免费白嫖。`api_demand_lost` 仅记 API 那部分丢失; 订阅缩减不进 lost 桶 (它是营收侧的现象, 不是需求侧丢失)。

### 5.2bis 算力池饼图 (产品 tab UI)

产品 tab "算力池 (按模型)" section 每行旁边渲染一个小饼图 (`scenes/ui/components/capacity_pie/`), 把同一模型的算力池**直接可视化**: 订阅占用 (绿) / API 占用 (蓝) / 空闲 (灰)。需求超过 capacity 时整圈描红边并把订阅 / API 切片按 `ratio = capacity / total_demand` 缩放 (= 实际能被服务的比例), 把"算力已满"这件事在玩家视线里一眼看出来 —— 单靠数字 util_pct 玩家不一定会盯。

输入 = `{capacity, sub_demand, api_demand}` (tokens/周), 与表格行同源。capacity = 0 时画灰圆 + 中心 "未部署"。

### 5.3 `_product_throughput_multiplier(model_id)` (chief_engineer lead 加成)

绑定该 model 的 product 中, 若某 product 的 lead 是 `chief_engineer`, 取 `1 + ability/100 × LEAD_BONUS_TABLE.chief_engineer.product_throughput`; 多个 product 取最大值避免重复相乘。

## 6. v4 (PR-B) engineering 乘数下沉

`throughput_multiplier` 与 `flops_per_token_reduction` 改由 `InfraSystem._recompute_compute` 在算 `dc.serving_tokens_per_sec` 时直接叠乘 (见基础设施 §4.1), MonetizationSystem **不再二次乘**, `_engineering_throughput_multiplier` helper 已删除。

变更原因: (1) 反馈链不断 — 玩家研究 FP8/KV cache 后 dc 卡片显示的 t/s 立即上升; (2) 避免二次乘风险; (3) `flops_per_token_reduction` 在语义上属于 token-cost (与模型 fpt 同侧累乘), 不该出现在算账侧。

## 6bis. 营收 tab 视图 (RevenueView)

营收 tab 不再用 main.gd 内联拼一摞纯文本行, 改为独立视图 `scenes/ui/views/revenue_view/`, 与产品页同款"`refresh(data)` 拉取模型": 纯展示, 不发业务信号, 数据由 `main._build_revenue_view_data()` 从 `GameState.last_revenue_breakdown` 拉好喂进来。设计目标是**简约 + 可一眼对比 + 轻交互**, 不引入历史/涨跌 (后端只存当周快照, 不碰数据层)。

布局 (从上到下):

1. **三块 StatChip** — 总营收 / API / 订阅, 与顶栏同一视觉语言。
2. **来源占比条** — 一根横向占比条 (`ShareBar`, 见 §6ter) 把 API (炭黑) / 订阅 (绿) 两段按金额铺满, 右侧附 "API x% · 订阅 y%"。让玩家不用读数字就看出营收结构。
3. **营收明细 (按模型) — 可折叠分组**: 每个**有营收**的 published 模型一个分组。
   - 分组头 (整行可点, hover 浅灰底, 手型光标): `[▼/▶ 折叠箭头] 模型名 ……… $总额 [占比条] x%`。占比条按 grand_total 为分母, 内部再分 API/订阅两段 — 所以一根条同时表达"该模型占公司营收多大"和"它的营收里 API vs 订阅构成"。
   - 分组体 (展开时可见): 该模型下每个产品一行 `[· 类型标签] 产品名 …… $额 [占比条]`, 占比条同样以 grand_total 为分母, 颜色按类型 (API 炭黑 / 订阅 绿)。产品行不再重复百分比数字, 减噪。
   - **交互 = 点击分组头切换展开/收起**; 展开态在 `refresh()` 之间记忆 (回合推进重渲不丢状态)。模型分组**默认展开**。
4. **算力需求 (按模型) — 可折叠 section**, **默认收起** (二级细节): 展开后每模型一行 `· 模型名 … t/s (周量)` + dim 细分 "订阅 X · API Y"。

未结算时 (`last_revenue_breakdown` 为空) 只显示 "(尚未结算)" 提示, 其余区块隐藏。所有金额走紧凑格式 ($640k / $1.2M), token 量纲走 §4 的 t/s + 周量双显示。

**显示契约 (沿用)**: 分组头用 `model.display_name`、产品行用 `product.display_name`, 不裸露 raw id (集成测试 `main_hud_layout_test.test_revenue_tab_shows_api_per_product_breakdown` 守护)。分组体节点常驻 (折叠只切 `visible`), 所以"展开到 api_per_product"这条契约即使默认折叠也成立。

## 6ter. ShareBar 占比条组件

`scenes/ui/components/share_bar/` — 通用横向占比条。`set_segments(segments, total)`: 在浅灰 track 上从左到右按 `value/total` 铺彩色段 (方角, 8px 高)。`total <= 段和` 时回退到段和防溢出; 空数据只画 track。营收页三处复用它: 顶部来源条 (API/订阅两段)、分组头 (API/订阅两段, 分母 grand_total)、产品行 (单段, 分母 grand_total)。
