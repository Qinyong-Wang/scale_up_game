# 模型训练系统设计 (ResearchSystem)

> **目的**: 管理玩家自训模型这一核心资产, 覆盖 `pretrain → posttrain → evaluate → publish` 生命周期、开/闭源发布、API 单价、改名/删除等操作。主界面入口是「模型」tab。
> **读者**: 模型 tab UI、PretrainDialog / PosttrainDialog / EvaluateDialog、TaskSystem 完成回调、MarketSystem、MonetizationSystem、ProductSystem、InfraSystem。
> **代码类名**: `ResearchSystem` (文件名保留, 设计语义已是"模型训练系统")。

---

## 0. 职责边界

模型训练系统拥有 **`GameState.models` 切片** (玩家自训模型 + 通过 `research.download_open_source` / 公共开源部署自动物化的下载模型):

- 训练: 通过 `task.start` 创建 pretrain / posttrain / evaluate 任务。
- 模型库: 展示模型的状态 / 能力 / 发布 / 部署绑定关系。
- 发布: 自训模型评估后发布为闭源 API 或开源权重 (`is_open_source` flag)。
- 下载 / 公共开源物化: `research.download_open_source` 把 OS NPC pretrain release 一键实例化为 `evaluated + provenance=downloaded_os` 的 Model; `research.ensure_open_source_release_published` 复用同一物化逻辑并在需要时自动发布, 供基建 tab 的公共开源部署入口调用。

公共开源模型 Serving (基础设施 tab 的 Serving 入口直接选 `Wolf-OS` 等) 会先物化 / 复用一个 downloaded_os Model, 再走 published model serving 管道。这样部署后产品页能看到自动 API 产品, 玩家也能调 `Model.per_token_price`。

| 玩家想做什么 | tab | 入口 |
|---|---|---|
| 训练 / 后训练 / 评估 / 发布自己的模型 | 模型 | 模型 tab |
| 跳过训练得到一个已知能力的开源模型 (作为起点) | 模型 | "下载开源模型..." 按钮 → `download_open_source` |
| 把自有 published 模型或公共开源模型跑推理 | 基础设施 | 机房卡片的 Serving 按钮; 公共开源 release 会自动物化并发布 |

## 1. 数据切片

owner = `ResearchSystem`。`GameState.models: Array[Model]`。

### Model 字段

| 字段 | 类型 | 语义 |
|---|---|---|
| `id`, `display_name` | StringName / String | 玩家命名时 id == display_name; 自动 codename 时 id = `model_NNNN` |
| `arch` | StringName | 架构 id (动物族化名, e.g. `ant_v2`, `octopus_v1`) |
| `size_params` | float | 总参数 (百万) |
| `active_param_ratio` | float (0..1) | MoE 激活参数比例; dense=1.0; 见 `model.gd::ACTIVE_PARAM_RATIO_BY_ARCH` |
| `attention_id` | StringName | B 轴 (v5 PR-C): `mha_baseline / gqa / mqa / mla / hybrid_attention` |
| `loss_id` | StringName | C 轴: `ce_baseline / zloss / mtp` |
| `context_length_tokens` | int | D 轴: 4096 / 32768 / 200000 / 1000000 |
| `flops_per_token` | float | 推理一 token FLOPs; `2 × size_params × active_ratio × 1e6 / attention.inference_coef` |
| `input_modalities` / `output_modalities` | `Array[StringName]` | `&"text"` / `&"image"` / `&"audio"` |
| `capability` | `Dict[axis: StringName, float]` | 5 轴 (general/code/reasoning/multimodal/agent), ≥0 无上限; 100 = SOTA 基准 |
| `capability_revealed`, `capability_stale` | bool | reveal 在 evaluate / posttrain 后置 true; stale 在 evaluate 过后又走兼容路径 posttrain 时置 true |
| `posttrain_delta` | `Dict[axis, float]` | 累计 clamp 后 posttrain delta (Bug B fix), `evaluate_apply` 叠加 |
| `posttrain_count` | int | 累计 posttrain 次数 (审计) |
| `trained_at_turn`, `dataset_ids` | | 审计字段 |
| `status` | StringName | `pretrained` / `posttrained` / `evaluated` / `published` |
| `provenance` | StringName | `trained` (自训) / `downloaded_os` (`download_open_source` 或公共开源部署物化路径) |
| `source_release_id` | StringName | downloaded_os 来源的 NPC release id; 空表示自训或旧档未知来源。用于 `ensure_open_source_release_published` 幂等复用, 防止同一公共开源模型多次部署时重复建模型 / API 产品 |
| `is_open_source`, `per_token_price`, `unpublished_at_turn` | | 发布字段 |

`active_param_ratio` (MoE 激活参数比例, dense=1.0) 的权威静态表在 `model.gd::ACTIVE_PARAM_RATIO_BY_ARCH`。
- **影响**: 同时折进训练 compute 与推理 flops_per_token; Chinchilla optimal tokens 仍按**总参数**算 (MoE 数据饥渴是合理代价)。

派生量:
- `Model.displayable_capability()`: 未 reveal 返 `{}`。
- `Model.is_publishable()`: `status == evaluated and !capability_stale`。
- `Model.infer_flops_per_token(size, active_ratio)` / `normalize_flops_per_token(raw, size, active_ratio)`: 静态助手, 兼容旧存档 (`raw < 1e6` 视为旧口径, 自动重算)。

## 4. 核心算法

### 4.1 状态机

```
pretrained ─posttrain→ posttrained ─evaluate→ evaluated ─publish→ published
                            ▲                     │ ▲                │
                            └ posttrain (stale 路径) │ │   unpublish     │
                                                    │ └─────────────────┘
                                                    │ evaluate
                            evaluate / download_open_source 直接到 evaluated
```

`published` 模型不能 posttrain / evaluate / delete; 须先 unpublish。

### 4.2 posttrain_apply

**v12 (2026-05) 防刷分重写**: 旧版对每份数据集**逐份相加** `gain`、目标轴无上限, 导致两个漏洞 —— (1) 堆份数纯线性刷分、能力可刷到无穷; (2) 单份 `log2(1+size×1000)` 是凹函数且地板很高, 把同样 token 预算**拆成很多小份**反而收益翻几十倍。v12 改为「**按轴聚合 + 朝软天花板饱和**」, 两个漏洞同时堵死。

结算分三步:

1. **按 `target_capability` 聚合** (杀拆碎): 把所有有效 posttrain 数据集按目标轴分组, 每组算总 token `T = Σ sizeᵢ` 与 **token 加权质量** `q̄ = Σ(qᵢ·sizeᵢ) / T`。→ 一份 1B 与十份 0.1B 聚合后 `T`/`q̄` 完全相同, 拆碎不再有任何收益; 往里掺低质量大份会把 `q̄` 拉低。

2. **目标轴朝软天花板饱和** (杀无限堆): 
   - `raw_gain = K_gain · q̄² · log2(1 + T·1000)` —— log **只对聚合总量取一次**。
   - `ceiling = base_power · POSTTRAIN_CEILING_MULT · score_mult` —— 单模型的软天花板。`score_mult` 是**后训练科学家加分** (`ml_research_lead.posttrain_score_bonus`, `= 1 + ability/100 × coef`, 无 lead / 万能 lead → 1.0): 强后训练科学家把天花板抬高, 同样的数据能把目标轴推得更高。仍受 ceiling 限制 (不会无限堆), 且 `base_power` 不含 posttrain 贡献 → 不会 ratchet。PosttrainDialog 在预览显示该加成, 预览与 apply 共用同一 `score_mult` (v2.1 一致性契约)。
   - `realized = max(0, ceiling − c0) · (1 − exp(−raw_gain / SATURATION_SCALE))`, 新值 `c0 + realized`。**永不超过 ceiling**, 且越接近天花板边际越小; 多次 posttrain 跨 task 也朝同一天花板收敛。

3. **其余轴遗忘**: 每个有数据的轴组对其它轴施加 `forget = K_forget · max(0, 1 − q̄)`, `max(0, before − forget)` clamp 到 0; 轴组按固定轴序结算 (确定性)。

**base_power (软天花板基准, 绑 pretrain 投入)**: `base_power = max(size 头, 模型 pretrained 最强轴)`, 其中 size 头 = `clamp(20 + 12·log10(size_params_M/100), 10, 95)` (与 evaluate `base` 同曲线), pretrained 最强轴 = `max_ax(capability[ax] − posttrain_delta[ax])` (减掉 posttrain 自身贡献, 防止天花板被反复 posttrain 一点点抬高自己 ratchet)。设计意图: posttrain 是「**释放预训练潜能**」, 天花板由模型规模 + 预训练质量决定; 想刷更高分只能去训更大 / 更好的基座, 而不是堆后训练数据。`base_power` 由 `ResearchSystem.posttrain_base_power(model)` 计算, apply 与 PosttrainDialog 预览共用同一函数 (v2.1 preview-apply 一致性契约)。

公式常量见 `平衡参数.md` 与 `research_system.gd` (`POSTTRAIN_GAIN_K` / `POSTTRAIN_FORGET_K` / `POSTTRAIN_CEILING_MULT` / `POSTTRAIN_SATURATION_SCALE`)。

`delta` 取 clamp / 饱和后的真实差值 (零基线模型被扣 forget 时记 0 而非负值; 目标轴记实际 realized 而非名义 raw_gain)。累计到 `model.posttrain_delta[axis]` 供 evaluate 叠加。

posttrain_apply 完成后置 `status = posttrained`; 若任一 dataset 是有效 posttrain → `capability_revealed = true, capability_stale = false` (能力当场可见, 不再要 evaluate); 否则走 v1 兼容路径 (`capability_stale = true`)。

**对开源公共模型不适用**: 公共开源模板不在 `GameState.models`, 没法 posttrain。

### 4.3 evaluate_apply

最终 capability 为 TaskSystem 产出的 `capability_measured` 叠加 `model.posttrain_delta`, 每轴 clamp 到 ≥0。

**Bug B fix**: posttrain 的轴向 +target / -forget 通过 `posttrain_delta` 累加, 在 evaluate 时叠加回去, 不再被覆盖。`capability_measured` 反映"裸基线" (size / arch / data_quality / data_efficiency / lead), 由 TaskSystem evaluate 公式产出。

### 4.4 publish / set_api_price 与 v6 PR-E 定价系统

publish / set_api_price **不再做硬价格上限 clamp**, 只 floor 到 ≥0 (`maxf(0, requested)`)。

价格上限改由**市场需求弹性**软约束 — 玩家可定任意非负价, 但超过指导价会让对应 **API 产品的 `subscribers` 池**每周衰减 (远超指导价时当周需求直接归零, 跌价后恢复)。v7 PR-F: 原 `Model.demand_multiplier` 字段已删, 合并到 `Product.subscribers` 同一池。详见 `平衡参数.md §UserSystem` 与 `用户系统设计.md §5`。

**三个定价工具方法** (`ResearchSystem.base_price_per_token` / `guidance_price_per_token` / `weekly_growth_rate`), 设计意图:

- **基准价**: 每年重算, 锚定上一年发布的最新 GPU (按单卡推理算力取最优, 空集时回退「截止当前已发布所有 GPU」); 基准价 = 该 GPU 单位算力**全周成本** × 模型**有效推理 flops_per_token**, 其中:
  - **全周成本 = 维护 + 电费 + 购卡价按 3 年 (156 周) 摊销** (2026-05 加入 CapEx; 旧版只算 OpEx, 指导价比真实云价低 ~6×)。摊销周期固定 156 周 (`GPU_AMORTIZATION_WEEKS`)。
  - **有效推理 flops_per_token 上 MoE 稀疏度 8× 上限**: octopus_sparse (1/20) / super_sparse (1/40) 在算 cost 时按 `active_param_ratio = 1/8 = 0.125` 算 (其他口径如训练成本、serving capacity 仍用原始 active_param_ratio)。常量 `MOE_COST_SPARSITY_FLOOR = 0.125`。原因: 现实里 MoE serving 因为路由/KV/小 batch 浪费, 实际成本下降远不到论文宣称的 1/20, 8× 是工业常见 sweet spot; 不加这层 cap, 玩家研究到 super_sparse 后 base 直接塌一个数量级。
- **指导价**: 基准价乘以一个开/闭源差异化倍率 (开源远低于闭源)。当前 OS 5×, 闭源 100× (2026-05 调整: 旧版 OS 2× / 闭源 40× 太薄, 玩家定一个合理 $/M tok 就触 cliff; 100× 闭源对齐真实云 API 毛利)。
- **周需求增长率**: 由"实际定价 / 指导价"的比值映射到一段分段曲线 — 低于指导价给正增长, 接近指导价趋零, 高于指导价转为负增长, 远超指导价触发当周需求归零的 cliff。

所有阈值与倍率常量见 `平衡参数.md` 与 `research_system.gd`。

**MoE / 量化自动便宜**: `m.flops_per_token` 已经折进 active_param_ratio + attention.inference_coef; engineering 树 `flops_per_token_reduction` 下沉到 dc 后, 玩家"同样卡服务更多 token"等价单 token 成本下降。

**downloaded_os 模型**: provenance 走开源价路径 (`guidance = 5 × base`)。只要处于 `published` 且存在 type=`api` 产品, 就与玩家自训开源模型一样参与 API 需求、产品页展示、调价和营收结算。公共开源部署入口会自动完成这一步, 避免"部署了但没有 API / 不能调价"的断链。例外: downloaded_os 是 NPC release 的运营副本, 不作为玩家原创模型进入公开排行榜; 排行榜只显示来源 NPC release, 产品增长需要的相对名次由 MarketSystem 按该模型 capability 与 NPC 比较得出。

open / closed 选择**永久**: 一旦 published 不能再切换 `is_open_source` (`_on_set_open_source` 在 published 时拒)。

**正在 posttrain 拒发布** (bug fix 2026-05): publish 必须额外检查 `GameState.active_tasks` 里没有针对该模型的 `subtype = posttrain` 任务在跑。否则旧版本里玩家可以在 posttrain 进行中点 publish, 任务完成时 `posttrain_apply` 会发现模型已 published 而拒 (`already_published`), 玩家白花了几周的算力+周费。错误码 `posttrain_in_progress`。

### 4.5 download_open_source

**v9 PR-I (2026-05)**: OS 模板表 (`OpenSourceModelTemplate` + `resources/data/models/os/*.tres`) 完全删除。OS 模型的唯一来源是 **7 家开源 NPC 公司 (`is_open_source = true`) 的 `model_releases` 数组中 `release_kind = pretrain` 的条目**。详见 `NPC配置.md §2.6`。

`research.download_open_source {release_id}`:
1. 调 `MarketSystem.find_release(release_id) -> {ok, npc, release}` 反查。
2. 校验 `release.release_kind == &"pretrain"` (非 pretrain 没有 cluster/params 信息, 不能下载) 与 `npc.is_open_source == true` 与 `release.release_turn <= GameState.turn`。
3. 错误码: `unknown_release / not_pretrain / not_open_source / not_released_yet`。
4. 从 release 派生 Model 字段:
   - `size_params (百万) = release.params_b × 1000.0`
   - `active_param_ratio = release.active_params_b / release.params_b` (dense → 1.0)
   - `arch = release.arch_codename`
   - `flops_per_token = Model.infer_flops_per_token(size_params, active_param_ratio)` = `2 × size_params × active_ratio × 1e6` (与玩家自训模型同公式)
   - `capability = release.capability.duplicate()` (5 轴公开值)
   - `input_modalities / output_modalities` 当前一律硬编码为纯文本 `[&"text"]` (OS NPC release 尚未声明多模态轴; 未来 release_kind=`multimodal_posttrain` 不可下载所以无影响)
5. 设 `status = evaluated, capability_revealed = true, provenance = downloaded_os, source_release_id = release_id, dataset_ids = []`; emit `model_added` + `model_evaluated` + `model_updated`。

**冷启动语义**: turn 0-214 时所有 OS NPC 都未首发, 该命令对任意 release_id 返回 `not_released_yet` (除非该 release_id 的 release_turn 已到, 但首个 OS release Wolf-1 在 turn 215)。UI (main.gd serving 抽屉 + 下载按钮) 据此置灰或显示「暂无可用开源模型」。

下载模型的 API 价格仍通过 `provenance == downloaded_os` 在 v6 PR-E 软弹性里走「开源指导价 = 5 × base」路径。

### 4.5bis ensure_open_source_release_published

`research.ensure_open_source_release_published {release_id, per_token_price?}` 是 `infra.deploy_open_source_model` 的桥接命令:

1. 先按 §4.5 同样校验并解析 release。
2. 查找已有 `Model.source_release_id == release_id` 的模型; 旧档没有该字段时, 可退化匹配 `provenance == downloaded_os && display_name == release.display_name`。
3. 没有则创建 downloaded_os 模型, 并写 `source_release_id`。
4. 若该模型不是 `published`, 自动调用发布逻辑: `is_open_source = true`; 若入参没给 `per_token_price` 且模型当前价为 0, 默认用 `guidance_price_per_token(model)`。
5. 返回 `{ok, model_id, created, published, applied_price}`。幂等调用不会重复创建模型或重复创建 API 产品。

### 4.6 PretrainDialog 4 轴 (v5 PR-C)

预训练时 4 个独立轴同时选择, 全部写入 `task.start` payload 与 Model:

| 轴 | 字段 | 解锁来源 | 影响 |
|---|---|---|---|
| A 架构族 | `arch` | `unlocks.arch` | active_param_ratio (训练 + 推理), train_coef, capability_coef |
| B 注意力 | `attention_id` | `unlocks.attention` | attention_train_coef (训练), attention_inference_coef (一次性折进 model.flops_per_token) |
| C Loss | `loss_id` | `unlocks.loss` | loss_train_coef, loss_capability_coef |
| D 上下文 | `context_length_tokens` | 不需解锁 | 训练时长乘子 (4k=1.0 / 32k=1.10 / 200k=1.30 / 1M=1.60) |

`task.preview` payload 见 `命令总线表.md §TaskSystem`。返回 `modifier_breakdown: Array` 是训练前 buff / debuff 的权威 — 对话框必须把所有当前生效项明显显示, 不只放 tooltip。

### 4.7 unpublish / delete 顺序

- `unpublish_model`: 若被订阅类产品绑定 (type != api), 报 `in_use_by_product`; 否则自动 `infra.undeploy_model`; 回 `evaluated`, capability 保留。
- `delete_model`: published 不能直接删; serving 中或被产品绑定不能删 (报 `in_use`)。

### 4.8 定价信息可见性契约 (UI)

§4.4 的三件套数值 (`base_price / guidance_price / weekly_growth_rate`) 不是内部算账, **必须直接给玩家看**。所有相关 UI 表面要把"价格往哪儿动"的因果链摊开:

| 表面 | 必显字段 | 备注 |
|---|---|---|
| **ModelCard (常显)** | `推理成本` (= base_price, $/M tok) · `指导价` (= guidance_price, $/M tok; 评估前模型按尚未决定开/闭源, 标为 `开源 $X / 闭源 $Y`) | 让玩家在 evaluated 阶段就知道发布后大致定价区间 |
| **ModelCard (published 额外)** | `定价比` (= per_token_price / guidance, %) · `周需求增长` (= weekly_growth_rate, %; -100% 标为「⚠ 0 需求」cliff) | 一眼看出当前是否在惩罚区 |
| **PriceEditDialog** | 当前价 / 指导价 / 推理成本 / 输入新价 → 实时算「新定价比」「新周增长率」并高亮 cliff (≥2.5×) 与「低于指导价的增益区」(≤0.6×) | 取代旧的 `+25% / -20%` 两个按钮; 输入框单位 `$/M tok` |
| **ProductCard (API)** | 同 ModelCard 的 base/guidance (取绑定 model) | 玩家从产品视角同样能看到定价空间 |
| **ProductCard (订阅)** | `参考订阅价` (= `ProductTypeSpec.subscription_price_guidance`) | 仅作参照, 不强制 |
| **PretrainDialog 预览** | `flops/token` · `推理成本` · `指导价` (开源 / 闭源各一档) | 调 attention / loss / context / MoE 时实时随动, 让架构决策与发布定价空间挂钩 |

实现要求:

- `task.preview` 返回 dict 必须额外含 `flops_per_token: float`, 公式与完成路径完全一致 (`Model.infer_flops_per_token(size, active_ratio) / attn_inference_coef`)。
- 命令 `research.preview_pricing {flops_per_token}` → `{ok, base_price, guidance_open, guidance_closed}`, 让 PretrainDialog 不必构造 stub Model。
- 命令 `research.preview_growth_rate {flops_per_token, per_token_price, is_open_source}` → 返回该价对应的周需求增长率, 供 PriceEditDialog / PretrainDialog 实时预览。
- 以上命令详见 `命令总线表.md §ResearchSystem`。
- ModelCard / ProductCard 是纯静态翻译器, 价格 / 比值 / 增长率由 view 层在调 `populate(...)` 前从 `ResearchSystem` 取好作参数传入, 翻译器不持有系统引用。
