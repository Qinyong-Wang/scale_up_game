# 产品系统设计 (ProductSystem)

> **目的**: 管理对外销售的产品 (chatbot / agent / multimodal_assistant / coding_agent / api), 每个产品**绑定一个 published 模型**, 持续吸引/流失 subscribers, 或对外开放 API。
> **读者**: 产品面板 UI、UserSystem (写回 subscribers)、MonetizationSystem (订阅营收 + API 算力池)、HiringSystem (锁 lead/staff)、ResearchSystem (校验模型)、TechTreeSystem (application 节点校验)。

---

## 0. 关键设计点

1. **类型由能力门控解锁**: 类型 X 需任意 published 模型在 `unlock_thresholds` 各维度 ≥ 阈值, 且 `application_node_required` (若有) 已在 application 树解锁。
2. **默认绑定最新模型** (`auto_track_latest`): 创建时绑 `latest_published_model_for_type(type)`; 玩家可手动 override。每次有新 published 模型且满足阈值 → 自动重绑 (同 open/closed source 才跟随)。
3. **每用户固定 token/周**: token_demand = `subscribers × ProductTypeSpec.tokens_per_user_per_week` (UserSystem 算; `tokens_per_user_per_month` 是旧字段, 仅作 fallback)。产品卡片的 `t/s 占用`、新建产品预览、订阅营收结算也必须同样优先读 `tokens_per_user_per_week` / `ProductTypeSpec.tokens_per_week()`，避免旧别名滞后时出现产品卡片 2.3G、算力池/营收页 9.3G 的口径分裂。
4. **lead / staff 可选**: 不雇仍能上线 (基础值); chief_engineer + ml_eng 给 quality 加成。
5. **API 产品 = published 模型的 API 销售单元** (v7 PR-F): 与订阅产品**共用 subscribers 池**机制 — 每周按 rank + 价格 + 营销 加法分量演化。`api_product.subscribers` 是"API 需求池单位", 转 token 需求乘 `api_tokens_per_sub_per_week` (默认 2M tokens/sub/周)。价格仍走 `model.per_token_price`, 玩家用 `research.set_api_price` 调; 产品页 API 卡片的"编辑"动作打开 PriceEditDialog, 直接改绑定模型单价。

### 算力池 + 订阅/API 二分 (MonetizationSystem 实现)

每个 published 模型 = 独立算力池, 供给来自所有 `serving_target = model_id` 的 dc, 需求来自所有 `bound_model_id == model_id` 的产品。月度结算:

1. 订阅产品 demand **全额扣除**算力, 即使超过订阅照收 (用户已付订阅费, 算力不足公司亏 API 那头)。
2. 剩余算力切给 API 产品 (多 API 产品按 demand 比例分; 单 model 单 API 是软约束)。
3. API 产品 `revenue = served_tokens × model.per_token_price`。

详见 `营收系统设计.md §6.2`。

## 1. 数据切片

owner = `ProductSystem`。`GameState.products: Array[Product]`。

### Product 字段

| 字段 | 类型 | 语义 |
|---|---|---|
| `id`, `display_name` | StringName / String | 标识 |
| `type` | StringName | `chatbot / agent / multimodal_assistant / coding_agent / api` |
| `bound_model_id` | StringName | 绑定的 published 模型 |
| `is_open_source` | bool | 从 `bound_model.is_open_source` 拷贝; `auto_track_latest` 时只跟随同 open/closed 模型 |
| `auto_track_latest` | bool | true = 跟随新 published; api 强制 false |
| `subscription_price` | int | ¥/周 (v7 PR-F: 隐式per_week, 沿用 v6 命名); api 强制 0 (价格走 model.per_token_price) |
| `lead_id` | StringName | 可空; 非空必须是 `chief_engineer` (或 player_scientist) |
| `assigned_staff` | `Dict[role, int]` | 可空; 锁定的 staff |
| `subscribers` | int | 当前订阅 (v7 PR-F: 池, 每周 UserSystem 演化; api 产品也用此字段作 API 需求池单位, 不再恒 0) |
| `quality` | float | 0..1.5, 仅影响 UserSystem 用户增长率, **不影响** token_demand |
| `launched_at_turn` | int | 审计 |

### ProductTypeSpec 字段 (`resources/data/products/types/*.tres`)

| 字段 | 含义 |
|---|---|
| `id`, `display_name` | 标识 |
| `unlock_thresholds: Dict[axis, float]` | 能力门 (例 `{reasoning: 50}`); 模型每轴必须 ≥ 阈值 |
| `application_node_required: StringName` | tech 节点门 (例 `&"tool_use"`); 必须在 application 树解锁; 空 = 无门 |
| `required_modalities: Array[StringName]` | 模型必须支持的 input_modalities (例 `[&"image"]`) |
| `tokens_per_user_per_week: int` | (v7 PR-F 重命名) 用户每周 token 用量 (UserSystem 用); 数值不变, 仅字段名清晰化 |
| `subscription_price_guidance: int` | (v7 PR-F 新增) ¥/周 指导价 — UserSystem 价格弹性公式分母 |
| `default_subscription_price: int` | ¥/周 创建时 UI 默认价 (与 guidance 通常一致, 玩家可改) |
| `capability_penalty_axis: StringName` | (v10 新增) 能力门槛看哪条轴; 空 = 无门槛。见 用户系统设计.md §5.2bis |
| `capability_penalty_tiers: Array[Dict]` | (v10 新增) `[{below, rate}, ...]`; 绑定模型对应轴低于阈值时, UserSystem 每周扣订阅池 |
| `max_subscribers: int` | 单产品 subscribers 池硬上限。`0` = 无上限 (api 用); 非 0 时 ProductSystem 在 `update_subscribers` 写回时夹住。订阅产品按市场体量定 (chatbot 2B / agent 1B / multimodal 1B / coding 200M) |
| `lead_specialty_bonus: StringName` | 适配 lead 的 specialty (默认 `chief_engineer`); 不匹配则 lead_factor = 1.0 |

5 个 type 的具体数值见 `平衡参数.md §Product Token 默认值`, 表权威源是 `.tres`。

### ProductCard 显示契约 (v8 PR-I)

ProductCard 是纯翻译器, 由 ProductView 在调 `populate(...)` 时把定价上下文喂进来:

- **API 产品**: 除现有「单价 / 需求 / 上周营收」, 还要显示 `推理成本` (= `ResearchSystem.base_price_per_token(bound_model)`, $/M tok) 与 `指导价` (= `guidance_price_per_token(bound_model)`, $/M tok)。让玩家从产品视角也能看到当前 `per_token_price` 在指导价之下还是之上。
- **订阅产品**: 除现有「订阅价 / 订阅数 / 质量 / t/s 占用」, 还要显示 `参考订阅价` (= `ProductTypeSpec.subscription_price_guidance`, ¥/周)。仅作参照, 玩家可任意定价 (`product.update {price}`)。

价格 / 推理成本计算永远不在 ProductCard 里调用 ResearchSystem; view 层拿好再传, 保持翻译器无副作用。

## 2. 命令与信号

详见 `命令总线表.md §ProductSystem` / `事件总线信号表.md §ProductSystem`。

### 命令

| 命令 | 说明 |
|---|---|
| `product.create {type, display_name?, bound_model_id?, subscription_price?, lead_id?, staff?, auto_track_latest?}` | 错误码: `unknown_type / unknown_model / model_not_published / unknown_lead / lead_specialty_mismatch / capability_below_threshold / application_node_locked / insufficient_staff / duplicate_api_product` |
| `product.update {product_id, fields: {price?, staff?, bound_model_id?, display_name?, auto_track_latest?}}` | 改 staff 走 "release 旧 + lock 新", 失败回滚 |
| `product.delete {product_id}` | 释放 lead + staff; 已收订阅费不退 |
| `product.update_subscribers {product_id, delta}` | UserSystem 私有; 写回订阅 delta。下限 0, 上限按 `ProductTypeSpec.max_subscribers` 夹住 (0 = 无上限, api 类型用); 写入后 `subscribers_changed` 信号里的 delta 是夹住后的实际 delta |
| `product.recompute_quality {product_id}` | debug / 测试用 |
| `product.list_unlocked_types` | UI 列已解锁 type (有 published 模型满足阈值 + application 节点解锁) |

**API 产品自动化**:
- `research.publish_model` 成功 → `_auto_create_api_product_for(model_id)` 自动建一个 type=api 产品。
- `infra.deploy_open_source_model` 成功前会经 `research.ensure_open_source_release_published` 物化并发布公共开源 release, 因而也走上一条自动建 API 产品路径。
- `research.unpublish_model` → 静默删 api 产品 (订阅产品仍阻止 unpublish, ResearchSystem `_has_product_binding` 跳过 type=api)。
- 同 model 多个 API: `duplicate_api_product` 错误。

### 信号

`product_created(id)` · `product_updated(id, changed_fields)` · `product_deleted(id)` · `subscribers_changed(id, delta, new_total)` · `quality_recomputed(id, new_quality)`。

## 3. 对外依赖

- **读切片**: `HiringSystem.leads / staff_pool / staff_busy`, `ResearchSystem.models`, `TechTreeSystem.unlocks.application` (经 `tech.is_unlocked` 命令)。
- **调命令**: `hiring.assign_lead / unassign_lead / lock_staff / release_staff`, `tech.is_unlocked`。
- **订阅信号**:
  - `phase_started(action)` — 重算所有产品 quality。
  - `model_updated(id)` — 同 (绑定模型可能 posttrain 升级了)。
  - `model_published(id, is_open_source)` — auto_track_latest 重绑 + 自动建 API 产品。
  - `model_unpublished(id)` — 删 API 产品。
  - `lead_released(id)` — 防御: lead 莫名释放 → 删对应产品。
  - `save_loaded` — 恢复 `_next_product_seq` 到现有 `product_####` 最大序号 + 1, 修复旧档重复 product id, 避免读档后新建 / 删除 / 营收解析到错误产品; 同时重算一次 quality。

## 4. 核心公式

### 4.1 quality (仅影响用户增长率)

`quality` 由三个因子相乘并 clamp 到上限: 模型能力因子 (绑定模型 5 轴能力和) × lead 因子 (specialty 匹配的 chief_engineer 按 ability 放大) × staff 因子 (按 ml_eng 数量放大)。具体系数与上限以 `product_system.gd _compute_quality` 为准。

`lead_factor` 仅当 lead.specialty == `spec.lead_specialty_bonus` (默认 chief_engineer); 否则为 1.0。player_scientist 在 `lead_matches_specialty` 通行但 `lead_factor` 通过 specialty 实际匹配判断, 玩家本人不给 bonus (与 招聘系统 §1.2 一致)。

### 4.2 关键耦合: quality → UserSystem subscribers

`phase_started(action)` 内 ProductSystem 重算 quality **必须先**于 UserSystem 结算 (UserSystem `attract / churn` 用最新 quality)。

依赖 autoload 顺序 + `users_resolved` 信号链 (UserSystem 等 quality 重算后再结算)。

### 4.3 capability 阈值校验

`_meets_thresholds(model, spec)`: 模型在 `unlock_thresholds` 声明的每条能力轴上都必须 ≥ 对应阈值, 才算满足。UI `list_unlocked_types` 把"任一 published 模型满足全部阈值 + application 节点解锁"的 type 返回给玩家。

## 5. 数据表

5 个产品类型 (chatbot / agent / multimodal_assistant / coding_agent / api) 由 `resources/data/products/types/*.tres` 定义 (`ProductTypeSpec`)。每个类型的 token 用量、指导价、解锁阈值、所需 application 节点、对应子榜、能力门槛等数值均以 `.tres` 为权威源。
