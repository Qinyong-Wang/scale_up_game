# 用户系统设计 (UserSystem)

> **目的**: 把「排行榜 rank + base 需求曲线 + 价格 + 营销」折算成每个 **Product**.`subscribers` 的演化, 进一步派生 `paid_users` (订阅总数)。**无自身资产**, 是一个派生计算器。
> **v7 PR-F (2026-05)**: **完全废弃 `fame` 字段**, 改成排行榜驱动 + 时间曲线 + 加法分量。API 产品也走 `subscribers` 同一管道 (= 旧 `api_token_demand` 池等价物)。
> **读者**: 用户面板 UI、MonetizationSystem (消费 `Product.subscribers` 算 API + 订阅营收)、ProductSystem (本系统写回 `product.subscribers`)、EconomySystem (派生 paid_users 算订阅营收)。

---

## 1. 数据切片

owner = `UserSystem`。两个资源型字段, 没有"实体资产":

| 字段 | 类型 | 语义 |
|---|---|---|
| `paid_users` | int | 派生 = `Σ products[type!=api].subscribers`, 每周由本系统重算 (api 产品 subscribers 是 token 需求池, 不是用户数, 不入 paid_users) |
| `token_demand` | `Dict[model_id, int]` | 每周派生需求量 (sub + api 合计), UI 显示 |
| `api_token_demand` | `Dict[model_id, int]` | 每周派生 api 需求量 = `api_product.subscribers × API_TOKENS_PER_SUB_PER_WEEK`; MonetizationSystem 直接读 |
| `last_user_resolved_turn` | int | 上次结算回合 |

**v7 PR-F 关键变更**:
- **`subscribers` 是一个池 (Pool)**: 每周乘性增长 / 衰减 + 加法基础引流, **持久化在 Product 上**。
- **API 产品的 `subscribers`** 是「API 需求池单位」, 不是真实用户数; 转化为 token 需求时乘 `API_TOKENS_PER_SUB_PER_WEEK`。
- **节奏**: subscribers / paid_users 每周更新; token_demand 同周覆写 (snapshot)。
- **`api_token_demand` 计算方式变了** (来源: api 产品.subscribers, 不再是 `fame × cap`).

## 2. 命令与信号

详见 `命令总线表.md §UserSystem` / `事件总线信号表.md §UserSystem`。要点:

- `user.preview_demand {model_id}`: UI 预览, 返 `{ok, predicted}`。
- `user.recompute_now {}`: 测试 / debug 触发立即结算。

信号: `users_resolved(turn, paid_users_delta)` (每周 action 结算后) / `token_demand_changed(model_id, new_value)` / `paid_users_changed(delta, new_total)`。

**无玩家直接调用的命令** — 用户增长完全派生, 玩家通过其他系统间接影响 (营销预算 / 提升模型能力 / 改产品价格)。

## 4. 关键耦合: 相位顺序

UserSystem 必须**先**于 MonetizationSystem 同相位执行 (Monetization 算 API 营收时要读最新 `token_demand` / `api_token_demand`)。

实现: UserSystem `_resolve_action` 结束 emit `users_resolved` → MonetizationSystem **订阅该信号**而不是直接订阅 `phase_started(action)`, 由信号驱动顺序。

## 5. 核心公式 (v7 PR-F)

> 本节是 v7 重写。**每个产品 (含 api) 都是一个池**: 每周按「加法 rate 之和」做乘性演化, 再叠加来自排行榜可见度的基础引流 (additive)。具体公式实现以 `user_system.gd` 为准, 本节只记意图与规则。

### 5.1 weekly_rate 加法分量

每个 product 的周演化 rate = rank rate + 价格弹性 rate + 能力门槛惩罚 三项相加。营销活动 **不进 rate**, 走绝对引流 (§5.4)。

### 5.2 总榜 / 子榜 rank rate (取较优, **不叠加**)

rank rate 取**总榜 rate 与子榜 rate 的较优 (max)**, 模型在哪个榜表现最好就吃哪个榜的 rate; 没有对应子榜的产品类型只看总榜。

**v10 (2026-05) 变更①**: 旧版把总榜 rate 与子榜 rate **相加**, 结果一个在某细分榜登顶但总榜垫底的模型会被算成负 rate — 明明是某领域第一却在掉用户, 反直觉。改成取较优后不再正负相抵。

**v10 (2026-05) 变更②: rank 只与「其他公司」比**。用户增长侧的 rank 由 `MarketSystem.get_rank_vs_npcs(m.id, board)` 算 — **只数比该模型强的 NPC 条目, 玩家自己其它已发布模型一概不计入**。动机: 推理成本随模型规模暴涨, 玩家常常合理地让产品服务一个便宜的小模型, 同时实验室里有更强但更贵的前沿模型。若按全局榜算, 玩家自己的前沿模型会把产品在用的模型挤下一名 → 产品掉用户, 等于「公司自己跟自己竞争」。改成只跟竞品比后, 产品增长只反映「相对其他公司的竞争力」。
> 公开排行榜 UI (`get_rank_for_model`, 科技/市场页) 仍展示所有模型的全局名次, 不受影响 — 只有用户增长这一侧用 `get_rank_vs_npcs`。

产品类型 → 子榜的映射 (`api` 默认走 `sub_general`)、各档排名的 rate 取值见 §6 与 `tuning.tres`。

### 5.2bis 能力门槛惩罚 capability_penalty (v10)

**动机**: 旧版只要产品有用户就能产生营收, 玩家可以训一个极小、通用能力很低的弱模型, 然后疯狂砸营销预算把订阅池堆起来 — 前期营收失控暴涨。加入「产品类型对绑定模型的能力门槛」: 绑定模型对应能力轴低于门槛时, 每周对订阅池施加一个**固定负 rate** (落在多档时取最重的一档), 逼玩家先把模型练强再规模化。

数据驱动, 写在 `ProductTypeSpec` 上 (`resources/data/products/types/*.tres`):

| 字段 | 类型 | 语义 |
|---|---|---|
| `capability_penalty_axis` | `StringName` | 看哪条能力轴 (`general` / `code` / ...); `&""` 表示该产品类型无门槛 |
| `capability_penalty_tiers` | `Array[Dictionary]` | 形如 `[{below: 50.0, rate: -0.025}, {below: 70.0, rate: -0.015}]` (v11: ×0.5) |

各产品类型的门槛轴与分档数值见对应 `.tres`。`api` 暂不设能力门槛 (`axis = ""`): API 产品已有价格弹性 + 价格悬崖 (§5.3) 两道闸, 且其 `subscribers` 是需求池单位而非真实用户。

### 5.3 价格弹性 rate

价格弹性 rate 是一条以「价格 / 指导价」之比为自变量的**分段曲线**: 低于指导价小幅正引流, 接近指导价归零, 超过指导价线性转负, 超过封顶倍数后硬封到 -25%/周 (v11 平衡: 各档百分比相对 v10 全部 ×0.5, 让排行榜 / 价格优势的周变化不再过激)。

- **订阅产品** (`p.type != api`): 用 `p.subscription_price` 与 `ProductTypeSpec.subscription_price_guidance` 之比。
- **API 产品** (`p.type == api`): 用 `Model.per_token_price` 与 `ResearchSystem.guidance_price_per_token(m)` 之比, 走同一曲线; 额外有**价格悬崖**: 当价比 `≥ API_PRICE_CLIFF_RATIO` (= 2.5) 时, 当周 `subscribers` 直接强制清零, 不做乘性演化 (见 §5.6)。

`guidance_price_per_token(m)` 为开源 5×base / 闭源 100×base (见 `研究系统设计.md §4.4`)。

### 5.4 营销 attract (加法绝对引流, 不进 rate)

营销活动直接为匹配 type 的 product 增加绝对 subscribers (线性 CAC, 不走 fame 中间件): 各 active campaign 的 `weekly_budget × marketing_conversion_rate × campaign_efficiency × fake_score_conversion_mult` 之和。`campaign_efficiency` 在 `marketing_lead` 在场时按其 ability 与 bonus 系数放大, 否则为 1.0。CAC ≈ $80/用户 (v12: 在 v3 的 $40 基础上 ×2, 订阅产品与 API 统一)。

`Campaign.fake_score_level` 表示营销文案对模型性能分数的夸大程度: `none/low/medium/high`。获客倍率分别为 `1.00/1.05/1.12/1.25`; 留存惩罚分别为 `0.00/-0.001/-0.004/-0.010`, 即 `0/-0.1%/-0.4%/-1.0%/周`。留存惩罚进入 `_resolve_per_product` 的 additive weekly_rate, 仅影响该 campaign 的 `target_product_id`; 多活动累加后夹到 `[-0.01, 0.0]`, 活动结束即停止影响。

### 5.5 base_attraction (加法绝对引流, 时间曲线 + 排名加权)

base 引流 = 时间曲线 `base_demand_curve(turn)` × 总榜排名折扣。时间曲线是一条**分段线性曲线, 4 个锚点** (`turn = 0 / 100 / 280 / 400`), 锚点之间线性插值, `turn ≥ 400` 后**截至最后一个锚点值保持饱和**。锚点坐标与排名折扣系数见 §6 与 `tuning.tres`。只有总榜前 3 才有自然引流, 其余靠营销和价格自救。

### 5.6 整体演化 (每周 action 相位)

对每个 product `p` (含 api 类型) 每周结算:

- 绑定模型缺失 / 未发布 → 视为孤儿产品, 每周硬扣 `orphan_product_churn` 个 subscriber。
- **API 价格悬崖特例**: API 产品价比 `≥ API_PRICE_CLIFF_RATIO` (= 2.5) → 当周 `subscribers` 强制清零, 跳过乘性演化。
- 其余情况: `subscribers` 先按 `(1 + weekly_rate)` 乘性演化, 再加上营销引流与 base 引流, 下限为 0, 上限按 `ProductTypeSpec.max_subscribers` 硬夹 (订阅类产品有上限, api 类型 `max_subscribers = 0` 表示不夹)。`weekly_rate` 包含排名、价格、能力门槛与虚假性能分数留存惩罚。夹住由 ProductSystem 在 `update_subscribers` 写回时实施, UserSystem 只产出 raw delta。

### 5.7 token_demand / api_token_demand 重算

每周对每个 published 模型重算需求快照: 订阅部分 = 绑定该模型的非 api 产品 `Σ subscribers × tokens_per_user_per_week`; api 部分 = 绑定该模型的 api 产品 `Σ subscribers × API_TOKENS_PER_SUB_PER_WEEK`。`token_demand` = 两者之和, `api_token_demand` = api 部分。

**`token_demand` 不累积**: 每周快照, 算力短缺一周丢一周 (短缺惩罚见 §7 未决)。**`subscribers` 累积**: 是 Product 上的持久态, 进存档, 反映「该产品的市场势能」。

### 5.8 删除的旧逻辑

- ❌ `_price_pressure(price) = 1 + max(0, price - 99) / 200` 删 (现在用 `price_rate` 分段曲线统一对待 churn + growth)。
- ❌ `attract = ATTRACT_K1 × quality_share × fame` 删 (fame 已删, 改 rank rate)。
- ❌ `api_part = TOKEN_BASE_PER_FAME × fame × cap_total / 100` 删 (api 走 product.subscribers 池)。
- ❌ `Model.demand_multiplier` 字段删 (合并到 `Product.subscribers` 同一池)。
- ❌ `_campaign_boost` 经 fame 转用户的路径删, 改直接打 subscribers。
