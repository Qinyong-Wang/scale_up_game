# 玩法设计 (总览)

> **目的**: 跨系统的"总图" — 资源/资产二分模型、相位与触发时机的总表、玩法 ↔ 系统对应表、各系统的入口链接。
> **读者**: 第一次读项目设计的人; 想知道"哪个玩法在哪个系统里实现"的人。

每个系统的**详设**已拆到独立文档, 见 §3。本文只讲"跨系统的契约", 不重复各系统内部细节。

---

## 0bis. 命名约定与单位 (全局)

> **所有数值用真实世界量纲, 所有专有名词用化名**。这条贯穿所有系统设计与 `.tres` 资源命名。

### 名称: 化名取代品牌

模型 / 架构 / 算法用动物族, 硬件 / GPU / 供电用植物族, NPC 公司同理。现实品牌与现实模型名 (GPT, Llama, A100, H100 …) **不出现**在代码 / 资源 / UI; 学术机制名 (attention / loss / 量化精度) 例外, 用英文缩写。具体化名族与枚举值见 `公共枚举表.md §16`。

### 数值: 真实量纲 + 周节奏

数值用真实量纲: 模型 M params、数据 B tokens、训练算力 TFLOPs、推理 tokens/s、训练 compute FLOPs、钱用元。时间以周为单位 (1 turn = 1 周), `turn = 0` 锚到 2017-06-12 (Transformer 论文 arXiv), 详见 `游戏基础架构设计 §3.4.1`。字段名统一周量纲 (`*_weekly_*` / `*_per_week` / `*_weeks`); `from_dict` 路径对少数旧名做兼容映射。

### 训练时长

训练耗时按 Chinchilla 思路: `compute ≈ 6 × params × tokens` (MoE 取 active params), 除以数据中心有效吞吐 (含集群效率、架构 / 注意力 / 损失系数、数据质量、lead / staff 乘数) 得周数。完整公式与系数以 `task_system` / `infra_system` 代码为准。

## 0. 资源与资产模型 (跨系统契约)

游戏内一切可变状态分两类:

- **资源 (Resources)**: 通用货币性数值, 跨系统消耗与产出。当前只有钱 (cash, 由 Economy 拥有, 可正可负, 负得太久 / 太深破产)、付费用户 (paid_users, 由 User 派生)、token 需求 (token_demand, 由 User 按 model 派生) 三项。**注意 fame 名声已在 v7 PR-F 删除** — 现在用户增长靠排行榜 rank 驱动, 不再有累积衰减的名声资源。
- **资产 (Assets)**: 系统特定状态实体, 各有 owner system 与生命周期 (leads / datacenters / models / products / campaigns / loans / events / npc_companies …)。

> 完整资源 / 资产切片与所有权表见 `系统耦合矩阵.md §1`, 本文不重复枚举。

## 1. 相位与触发时机总表

每回合 = 1 周, TurnManager 顺序触发 3 个相位:

| 相位 | 做什么 |
|---|---|
| `upkeep` | Economy (利息扣 + 滚动 burn/revenue 窗口) / Hiring (周度工资) / Infra (rent + GPU runtime + cloud GPU) / Task (`weekly_cost`) / Marketing (campaign 周度扣预算) |
| `action` | Task 进度推进 + error_rate 滚 → **信号链**: Product 重算 quality → User 算 subscribers + token_demand → emit `users_resolved` → Monetization 结算营收 → emit `revenue_resolved` ; Market 排榜 + emit `leaderboard_resolved` → Event 检查融资 + 抽事件 ; Hiring 候选池刷新 ((`turn-1) % 4 == 0`); Dataset 半年解锁信号 (`turn % 26 == 0`); Infra 推进自建队列 |
| `resolve` | Economy 破产判定 / 自动存档 |

> **顺序耦合**: action 内"产品 → 用户 → 营收"链通过 `users_resolved` 信号锁死。详见 `系统耦合矩阵.md §4.1`。

## 2. 玩法 ↔ 系统对应

| 玩法 | 主 | 辅 |
|---|---|---|
| ① 模型生命周期 (pretrain → posttrain → evaluate → publish → serve) | Task / Research | Infra / Hiring / TechTree / Dataset |
| ② 排行榜竞争 (模型上榜 → rank 驱动用户增长与估值溢价) | Market | Research |
| ③ 用户增长 (paid_users + token_demand) | User | Marketing / Market / Product |
| ④ API 营收 + 订阅营收 | Monetization | Product / Infra / User / Research |
| ⑤ 科技树研究 (6 棵: arch / attention / loss / engineering / application / **context** v7 PR-G) | TechTree | Task / Monetization (arch.inference_coef) / Research (attention.inference_coef 折 fpt) / Infra (engineering 乘数下沉) / Product (application 解锁 type) / Task (context 树解锁 D 轴档位 + evaluate agent_bonus) |
| ⑥ 招聘 (lead S/A/B/C + 6 specialty + founder) | Hiring | Economy (现金量影响 S/A 候选抽取) |
| ⑦ 资金 (营收 / 融资 / 贷款 / 破产) | Economy | Monetization / Market (排行榜 rank 影响估值) |
| ⑧ 营销 (campaign 周度预算) | Marketing | User (消费 boost) |
| ⑨ 数据集 (开源 / 购买 / 自采) | Dataset | Task (data_collection) |
| ⑩ 事件 (机会 / 危机 / flavor) | Event | 几乎所有系统 (effect dispatch) |
| ⑪ 数据中心 + GPU + 供电 (cypress/maple/bamboo × t0-t4; grid/solar/nuclear/wind/coal) | Infra | Hiring (chief_engineer 加 cluster_eff) |
| ⑫ 开源 Serving 路线 (公共开源模型直接部署, 自动物化为 downloaded_os published model + API 产品) | Infra / Research / Product | Monetization |

### 2.1 模型生命周期

```
       pretrain task     posttrain task    evaluate task
         (N 周)            (M 周)            (k 周)
   pretrained ────► posttrained ────► evaluated ────► published ────► serve
        │              ▲                  │ ▲              │            ▼
        │              │ posttrain        │ │ evaluate     │       token_demand
        │              │ (capability      │ │ (refresh)    │       & API revenue
        │              │  stale)          │ │              │
        ▼              │                  │ │              ▼
       evaluate ─────────────────────────┘ └────────────── unpublish (→ evaluated)
        (可选, 给未 evaluated 揭基线)

   公共开源模型也可以通过 `research.download_open_source` 实例化为 evaluated 模型 (provenance=downloaded_os), 玩家快速起步用。
   若玩家直接在基础设施 tab 部署公共开源 release, 系统会自动物化/发布对应 downloaded_os 模型并创建 API 产品。
```

**约束**:
- 只有 `evaluated` 状态可 `publish` (且 `capability_stale = false`)。
- `pretrained` / `posttrained` 不可部署。
- 已 `evaluated` 后再 `posttrain`: capability 由 posttrain 直接结算 (v2), 不再 stale 等下次 evaluate。
- `published` 不能 posttrain / evaluate / delete, 必须先 `unpublish` (回 `evaluated`)。
- 公共开源模型有两条入口: (a) `research.download_open_source` 进入 models (有 capability, 可手动 publish / serve); (b) `infra.deploy_open_source_model` 从基建 tab 一步部署, 自动物化/开源发布/建 API 产品 (零研发起步, 但仍走产品与价格弹性)。

### 2.2 三种盈利路线 (玩法多样性)

游戏不强制研究路线, 三条可单独走或混搭:

1. **研发路线**: pretrain → posttrain → evaluate → publish → serve → 冲排行榜 + 收 API 营收。需 dc 算力 + lead 阵容, 风险高上限高。
2. **开源 Serving 路线**: 在基础设施 tab 选公共开源 release 直接 serve; 零研发, 启动快利润薄。部署后它会成为一个 downloaded_os published model, 可在模型/产品页调 API 单价并参与营收结算。
3. **产品路线**: 自训 published 模型 + chatbot / agent 产品 → 订阅营收。受 capability 阈值门控, 阈值越高解锁越高级 product (api / chatbot / agent / coding_agent / multimodal_assistant)。

> 玩家可在不同阶段切换路线 (例: 前期开源 Serving 攒钱 → 后期自研冲榜)。

## 4. 跨系统未决项

各系统未决项分散在对应文档末尾。以下是**跨系统全局未决项**:

- [ ] **同相位内顺序显式化**: action 相位 "产品 → 用户 → 营收" 链当前靠 autoload 顺序 + 信号链; 应改为子相位或更细的信号链。见 `系统耦合矩阵 §4.1`。
- [ ] **持续效果 (BuffSystem)**: EventEffect 都是瞬时, 无法表达"行业泡沫期持续 6 月用户增长 ×1.5"这类。需 `active_buffs` 切片或 BuffSystem。
- [ ] **`infra.force_terminate_dc`**: 危机事件想强制关停 busy dc 当前不支持。
