# 办公室与收藏系统设计

**一句话目的**：给后期资金一个「带陈列与轻投资属性」的出口——一间「办公室」**房间**（背景+办公桌+电脑，点击电脑打开**收藏柜**=玩家持有的收藏品可卖；墙角陈列**已获得的奖杯**）+ 独立的**拍卖行** tab（按当前市价买入各类收藏品）；收藏品价值随游戏日历沿增值曲线上涨、2070 封顶，卖出按当前市价但收一笔抽成。

> 这是慈善三期规划里的「二期」（见 `慈善系统设计.md §10`）。奖杯**授予来源**（登顶总榜 / 慈善达全球档 / 宇宙「42」）均已接入（见 §4）。

## 1. 定位与玩法

- **收藏 = 增值资产 + sink**：拍卖行按当前市价买入收藏品（钱被占用，是 sink）；价值随时间沿曲线升值；想回收时在收藏柜卖出，按当前市价**扣一笔抽成**（手续费/税）后回款。既能停泊溢出资金、又不是无风险套利。
- **价值随游戏日历**：当前市价 = 该收藏品增值曲线在「当前游戏年份」的插值（游戏从 2017 起，见 `GameState.turn_to_date`）。曲线最后一个关键帧锚在 **2070**，即各类收藏的价格上限；2070 之后价格保持封顶不再涨。
- **拍卖行 = 目录按市价买**：上架若干具名 lot（第一台 AI 训练卡 / 炎兽闪卡 / 创世币 / 超跑 / 名画…），点击按当前市价直接买入，不做实时竞价。每件收藏是**唯一**的（一件就一件），买走后从拍卖目录消失、进收藏柜。
- **化名规范**：收藏品名一律用**虚构名**，不出现真实品牌（如用「炎兽闪卡」而非真实卡牌 IP，「极速幻影 GT」而非真实超跑），与 GPU/模型化名规范一致。

## 2. 收藏品数据模型

数值放 `resources/data/collectibles/*.tres`（`CollectibleSpec` 资源），`CollectionSystem` 在 `_ready` 加载。
发布包里 `res://` 目录枚举不可靠，且导出资源可能以 `.tres.remap` 形式出现；因此收藏品清单的权威来源是 `CollectionSystem.COLLECTIBLE_PATHS` 显式路径表，而不是运行时 `DirAccess` 扫目录。新增 / 删除收藏品 `.tres` 时必须同步该表，测试负责校验路径表与磁盘一致，避免编译后拍卖行库存为空。

`CollectibleSpec`（`scripts/resources/collectible_spec.gd`）：

- `id: StringName` — 唯一 id。
- `category: StringName` — 类别（`&"ai_hardware"` / `&"trading_card"` / `&"crypto"` / `&"supercar"` / `&"painting"`）。
- `display_name: String` / `description: String` — UI 文案（走 i18n）。
- `curve_years: Array[int]` — 升序的关键帧年份（首帧=面世年，末帧=`2070`）。
- `curve_prices: Array[int]` — 与 `curve_years` 平行；各关键帧年份对应的市价，**末帧即价格上限**。

**当前市价**：给定当前游戏年份 `y`，市价 = 在 `(curve_years, curve_prices)` 上做分段线性插值；`y ≤ 首帧年` 取首帧价，`y ≥ 2070` 取末帧价（封顶）。每类曲线形状不同（见 §9）：crypto/卡牌早期爆发、超跑/名画稳健高位、AI 训练卡随 AI 成熟而升值。

## 3. 买入 / 卖出

- **买入**：`collection.buy {collectible_id}`。市价 = `current_price(id)`。校验：存在、未被持有、`市价 ≤ GameState.cash`（不允许买到现金为负）。通过则 `economy.spend(reason=&"collectible_purchase")` 扣钱、记入 `owned_collectibles[id] = 买入价`、emit `collectible_bought`。
- **卖出**：`collection.sell {collectible_id}`。校验：当前持有。回款 = `round(current_price(id) × (1 − SELL_FEE))`（`SELL_FEE` 抽成，见 `collection_system.gd` 常量，参考 0.15）。`economy.award(reason=&"collectible_sale")`、从 `owned_collectibles` 移除、emit `collectible_sold`。
- **盈亏与抽成**：收藏柜显示买入价 / 当前市价 / 浮动盈亏（当前市价 − 买入价，毛口径）/ **抽成**（`SELL_FEE` 比例 + 折算额），「卖出」按钮直接写**到手回款**（市价扣抽成后），供玩家决定何时卖（见 §8.2）。
- 买卖都是**即时**命令（不像慈善是 task）：拍卖落槌即拥有。

## 4. 奖杯与陈列（房间）

`TrophySpec`（`scripts/resources/trophy_spec.gd`）：`id` / `display_name` / `description` / `unlock_hint` / `form` / `flavor`。放 `resources/data/trophies/*.tres`。

- `GameState.trophies: Array[StringName]` — 已获得的奖杯 id。
- **`form`**（`&"trophy"` / `&"medal"` / `&"answer_box"`）决定在房间里的陈列位置：`trophy` 摆**远处茶几**，`medal` 放**近处办公桌**，`answer_box` 是终局专用的**密封盒**，摆在近处桌面、显示器下方（§8.1）。默认 `leaderboard_first`=trophy，慈善三档=medal，`universe_answer`=answer_box。
- **`flavor`**：点击该奖章/奖杯时信息 dialog 里在「描述」之后再显示的一段叙事文案（走 content.csv i18n，源中文串当 key）。
- **办公室房间只陈列已获得的奖杯**（§8.1）：每枚一个可点精灵，点击 → 荣誉信息 dialog（名 + 描述 + flavor）。未获得的不在房间出现（`unlock_hint` 仅作 spec 备注，后续若要做「成就总览」再用）。
- **授予来源**均经 `CollectionSystem.award_trophy(id)`（幂等，首次授予才 append 到 `trophies` 并 emit `trophy_awarded`），已接入：
  - `leaderboard_first` — 玩家登顶 `total` 总榜（`MarketSystem._check_player_rank_changed`）。trophy。
  - `universe_answer` — 宇宙模拟终章揭示「42」（`SimulationSystem`，慈善三期 capstone）。answer_box。
  - **慈善三档奖章**（medal）`charity_bronze` / `charity_silver` / `charity_global` — `CharitySystem._on_credit` 在**每完成一档捐助**（任一公益方向，首档/二档/顶档）时点亮对应铜/银/金奖章（`TIER_MEDALS` 与 causes 的 3 档 `tier_amounts` 平行，顶档金牌=「全球慈善家」）。捐助是多周 task，奖章在**任务结算完成**时点亮，不是开捐当下。

## 5. GameState 字段

新增切片（owner：`CollectionSystem`）：

- `owned_collectibles: Dictionary` — `{collectible_id(StringName): 买入价(int)}`。持有即记买入价，卖出移除。
- `trophies: Array` — 已获得奖杯 id（`StringName`）。由 §4 的授予来源 append。

`reset()` 置空；`to_dict()`/`from_dict()` 用 `_dict_with_sn_keys_to_strings`/`_strings_to_sn_keys`（dict）与字符串数组序列化（trophies）。旧档无 key → 空，兼容。

| 命令 | 入参 | 出参 | 错误码 |
|---|---|---|---|
| `collection.buy` | `{collectible_id}` | `{ok, price}` | `&"unknown_collectible"` / `&"already_owned"` / `&"insufficient_cash"` |
| `collection.sell` | `{collectible_id}` | `{ok, proceeds}` | `&"not_owned"` |

## 7. 信号

新增 EventBus 信号（三处同步：`event_bus.gd` + `event_bus_test` + `事件总线信号表.md`）：

- `collectible_bought(collectible_id: StringName, price: int)`
- `collectible_sold(collectible_id: StringName, proceeds: int)`
- `trophy_awarded(trophy_id: StringName)` — 由 §4 的三个授予来源经 `award_trophy` 触发。

`main.gd._subscribe_events()` 订阅三者 → `_request_refresh()`。

## 8. UI（办公室 tab）

侧栏「其他」组（`NAV_GROUP_OTHER`）含三个 tab：**办公室 / 拍卖行 / 慈善**。

**货币符号**：拍卖行 / 收藏柜里所有金额（市价 / 买入价 / 浮动盈亏 / 买入按钮）一律用全局货币符号 **`$`**（与顶栏现金、经济面板一致），**不用 `¥`**——本作是一家全球 AI 公司，统一用 `$` 计价，化名规范同理不绑定任何现实币种。

### 8.1 办公室 = 第一人称房间场景

`scenes/ui/views/office/`（HUD「办公室」tab）不再是卡片列表，而是一间**第一人称视角的办公室**（`OfficeView`, `extends Control`，纵向铺满 tab 区，找祖先 ScrollContainer 同步视口高）：

- **背景**：一整张房间场景图 `IconRegistry.office_texture(&"room-bg")`——第一人称坐在办公桌前，近处桌面 + 电脑 + 远处茶几烤进画面。视觉目标是**干净、温暖、有 AI 公司氛围的管理游戏办公室**，不要做成收藏品陈列室：房间里不摆玩家收藏品，不做玻璃柜，不把拍卖资产塞进场景。引擎按**等比 FIT** 居中铺进 stage rect（整图必全显、不裁不变形，余下为中性 letterbox）→ 奖章/奖杯的归一化锚点能精确落到桌面/茶几上。缺图回退程序化占位（墙 + 地板 + 显示器，`_draw`）。
- **画面层次**：OfficeView 在背景上加一层很轻的氛围处理（stage 细描边 / 暗角 / 底部渐变 / 荣誉接触阴影），让它像游戏界面而不是单纯贴图；这些效果只增强观感，不承载业务信息。背景使用等比 FIT 时，顶部/底部 letterbox 必须是中性 matte，不出现突兀白条。
- **电脑（近处屏幕）**：room-bg 中央显示器上覆一个透明可点按钮，屏幕区域绘制简洁的抽象 dashboard 光效，且应覆盖显示器黑屏的大部分面积（不是只贴在上半块）；屏幕默认有很轻的角标/辉光提示可交互，hover 时再提高亮度；点击 emit `computer_pressed` → main 打开**收藏柜 dialog**（§8.2）。电脑是收藏柜唯一入口，避免多热点让玩家猜。
- **奖章 / 奖杯 / 终极答案盒**：只陈列**已获得**的荣誉（`GameState.trophies`，§4），作为办公室点缀而不是资产展示——`form=medal` 的散放在**近处办公桌**、`form=trophy` 的散放在**远处左侧茶几**、`form=answer_box` 的密封盒摆在**显示器下方的近处桌面中央**（锚点常量 `_DESK_ANCHORS` / `_TABLE_ANCHORS` / `_ANSWER_BOX_ANCHOR`，经 stage 变换贴到桌/几面）。锚点必须尊重透视：茶几奖杯底部落在图内 `y≈0.74–0.78` 的桌面带，尺寸约为 stage 高的 9–10%；近处桌面奖章落在 `y≈0.88–0.94`，尺寸约为 stage 高的 7–8%；终极答案盒落在 `y≈0.84–0.88`，尺寸约为 stage 高的 10%。每枚一个透明精灵 `office_texture("medal-<id>" / "trophy-<id>" / "answer_box-<id>")`（缺图回退程序化按钮 / 图标字形），点击 emit `honor_pressed(id)` → main 打开**荣誉信息 dialog**（§8.1bis）。

视图不读 GameState；main.gd `refresh({trophies})` 传**全部** TrophySpec（带 `earned` + `form`），房间只摆 earned 的、按 form 分桌/几。信号 `computer_pressed()` / `honor_pressed(trophy_id)`。

### 8.1bis 荣誉信息 dialog（点击奖章/奖杯打开）

`scenes/ui/honor_dialog/`（`HonorDialog`, `extends AcceptDialog`）：标题=荣誉名，正文=描述 + `flavor` 叙事文案（空 flavor 隐藏）。`refresh({name, description, flavor, answer?})` 接已翻译串；main 按 id 查 `TrophySpec` 装配。`form=answer_box` 的终局荣誉必须传 `answer="42"`，dialog 在正文上方用大号、居中的「42」做主视觉，确保玩家打开密封盒时第一眼看到答案。

### 8.2 收藏柜 dialog（点击电脑打开）

`scenes/ui/collectibles_dialog/`（`CollectiblesDialog`, `extends AcceptDialog`）：列出**持有**的收藏品卡（买入价 / 当前市价 / 浮动盈亏 / **抽成** + 「卖出」按钮），空则空状态提示。`refresh({cabinet, sell_fee})` 接 dict；「卖出」emit `sell_pressed(collectible_id)` → main `collection.sell` 成功后即时重渲染。**买入**仍走独立的拍卖行 tab（§8.3）。

**卖出抽成要在 UI 体现**（玩家看得到才不会觉得回款“缩水”莫名）：每张藏品卡显式展示卖出抽成——
- **「抽成」字段**：`SELL_FEE` 比例 + 该件按当前市价折算的抽成额，形如 `15% · -$<fee>`（`fee = 市价 − 回款`）。
- **「卖出」按钮**：标签直接写**到手回款**（`卖出 $<回款>`，`回款 = round(市价 × (1−SELL_FEE))`），与拍卖行「买入 $<价>」对称，玩家点之前就知道实际到手多少。

`SELL_FEE` 由 main 经 `refresh` 的 `sell_fee` 传入（取 `CollectionSystem.SELL_FEE`），dialog 不直接持有系统引用；缺省（旧 caller 不传）= 0 → 不显示抽成、按钮回款=市价。浮动盈亏仍是**毛**口径（市价 − 买入价，未扣抽成），抽成字段与到手按钮负责把真实现金口径讲清。

### 8.3 拍卖行 tab + 轮换上架

`scenes/ui/views/auction/`（HUD「拍卖行」tab）：可买 lot 卡（类别 / 描述 / 当前市价 + 「买入」按钮，现金不足则禁用）。空则空状态提示。

**轮换上架（rotation）**：拍卖行**不再一次性列出全部未持有收藏**，而是只展示一个**轮换的精选 lineup**：

- **数量不多不少**：lineup 固定 `AUCTION_SLOTS`（参考 8）个槽位。库存（未持有）足够时正好 8 件；库存不足 8 时就显示剩下的全部（晚期收得差不多了才会变少）。
- **多样性保证**：先给「当前有库存的每个类别」各保底抽 1 件（5 类齐全时占满 5 个槽），剩余槽位再从全库存按权重补齐 → lineup 必然横跨多个类别，不会整屏同一类。
- **刷新节奏**：每 `AUCTION_REFRESH_WEEKS`（参考 4）周在 `action` 相位重roll 一次 lineup；买走的 lot 立即从当前 lineup 消失（空槽留到下次刷新）。lineup 与「上次刷新回合」存进 `GameState`（owner `CollectionSystem`），读档恢复。
- **稀有度 = 出现概率**：每件 `CollectibleSpec` 带 `appear_weight`（出现权重，由生成器按价位反比设定：便宜款 ~1.0、中档 ~0.5、贵重 ~0.2、文物级 ~0.08）。抽 lineup 时按权重随机 → 文物旗舰（8B GPU、12B 零号区块）**可遇不可求**，便宜款常驻。这就是「刷新概率」的可调旋钮。
- **可调项**：`AUCTION_SLOTS`（槽位数）/ `AUCTION_REFRESH_WEEKS`（刷新周期）放 **`resources/data/collectibles/auction_tuning.tres`**（`AuctionTuning` 资源，`CollectionSystem._ready` 鸭子类型加载，缺文件回退默认 8 / 4）；单品出现率 `appear_weight` 在各收藏 `.tres`（生成器 `build_collectibles.py` 控制）。注意该 tuning 与收藏 spec 同放 `collectibles/`，生成器清空步骤与目录扫描都按文件名跳过它。
- **导出兼容**：拍卖库存从 `CollectionSystem.COLLECTIBLE_PATHS` 加载，不使用 `DirAccess` 扫 `collectibles/`；否则 macOS/iOS 等导出包中 `.tres.remap` 文件名会让 `.tres` 过滤失效，导致上架列表为空。
- 抽取用 `GameState.rng()`（可复现），完成后按 (类别, 封顶价) 排序展示，UI 不变（仍读 `available_lots()`）。

- 文案走 `tr()`（chrome 走 `strings.csv`；收藏/奖杯内容串走 `.tres → content.csv`）。
- **美术**：真实贴图经 flux 管线生成（`tools/art/generate.py` 批次 `office_room`=scene 风格宽幅场景、`office_props`=prop 风格去背精灵），落 `assets/sprites/ui/office/`：`room-bg.png`（落地窗+湖+极简办公室，**办公桌与电脑烤进画面**，不去背）+ `trophy.png`（透明金奖杯精灵）。缺图自动回退程序化占位，无需改代码。
- **收藏品卡缩略图**：拍卖行 lot 卡与收藏柜藏品卡的 160×160 头像走**逐件**贴图——每件收藏一张独立图标，落 `assets/sprites/ui/collectible/<id>.png`（`<id>` 即 `CollectibleSpec.id`），由 `IconRegistry.collectible_icon(id)` 读取，view 把它当 `avatar.texture` 传给卡片。由 `tools/art/generate.py` 批次 `collectibles`（icon 风格去背精灵）按各件英文名/描述出图：crypto=纪念金币 / NFT 牌、trading_card=全息卡牌、ai_hardware=加速板卡/芯片、supercar=超跑模型、painting=画框名画（内容抽象化，不复刻真实作品/品牌，符合化名规范）。缺图自动回退 Avatar 的 seed 配色 + 首字母，无需改代码。
- 验收：UI 改动按 TDD 第 6 步截图。

## 9. 内容与增值曲线（由生成器产出，权威在 `.tres`）

收藏品 `.tres` 由 **`tools/build_collectibles.py`** 生成：每类 **15-25 件**（当前 crypto/trading_card/ai_hardware 各 16、supercar/painting 各 15，共 78 件），名字遵循化名规范（虚构名）。价格跨度大、每类都有 **1B+ 文物级**：

- **价格区间**（2070 封顶价参考）：crypto 80M–12B、trading_card 40M–2B、ai_hardware 800K–8B、supercar 120M–600M、painting 300M–1.5B。
- **文物旗舰**：`first_ai_card`「第一台 AI 训练卡」2070 封顶 **8B**（2017:5k → 2035:50M → 2055:2B → 2070:8B），符合「第一块训练用 GPU 文物级、后期 1B+」。
- **曲线形状**（生成器 `SHAPES`，按各件 `peak` 缩放、debut→2070 分段线性、2070 封顶）：crypto/卡牌早期爆发，ai_hardware 后期成文物，supercar/painting 高位稳健。
- **测试夹具**：`genesis_coin_7`（1k→8M，便宜，买卖测试用）与 `first_ai_card`（文物旗舰）在生成器里用显式 `OVERRIDES` 固定曲线，保证测试确定性。
- 生成器同时把所有 display_name / description 的英文合并进 `content.csv`。`SELL_FEE` 参考 0.15。改内容/调价改 `build_collectibles.py` 后重跑 + `build_translations.gd`。
