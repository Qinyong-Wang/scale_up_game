# 招聘系统设计 (HiringSystem)

> **目的**: 管理两类人力资源 — **lead** (具名个体, 影响 task 与 product 的能力上限) 与 **staff** (按职能聚合, 提供吞吐量)。负责候选池刷新、签约、解雇、工资发放与"忙碌"标记。
> **读者**: 招聘面板 UI、需要锁定人员的 system (TaskSystem / ProductSystem)。

---

## 1. 数据切片

owner = `HiringSystem`。

| 字段 | 类型 | 语义 |
|---|---|---|
| `leads` | `Array[Lead]` | 已签约 lead |
| `lead_pool` | `Array[Lead]` | 当期候选池 (每月初刷新, 见 §4.1) |
| `staff_pool` | `Dict[role: StringName, count: int]` | 按职能聚合总数, 固定 5 个 role: `ml_eng / infra_eng / data_eng / marketing / ops` |
| `staff_busy` | `Dict[role, int]` | 当前被锁定数, 与 `staff_pool` 同 keyset |

`Lead` 字段: `id`, `display_name`, `specialty`, `level` (`S/A/B/C/founder`), `ability` (0..100), `signing_fee`, `weekly_salary`, `is_player_scientist`, `locked_by_task_id`, `assigned_to_product_id`。

> **2026-05 移除 `equity_grant`**: 该字段从未被任何系统消费 (招人只扣 `signing_fee` 现金, 不稀释股权), 留着只会误导玩家。已从 `Lead` / `LeadLevelSpec` / `lead_levels/*.tres` / UI 全部删除。融资稀释的创始人/投资人股权表 (`GameState.equity`) 是另一套, 保留不动。

**三态互斥**: 任何时刻 lead 要么 idle, 要么 `locked_by_task_id != ""` (训练独占, TaskSystem 持有), 要么 `assigned_to_product_id != ""` (常驻产品, ProductSystem 持有)。Lead 自带 `is_idle()` 方法。

### 1.1 specialty 与 bonus

6 个 specialty 按**职能分组**对应 6 类工作, 每个 specialty 在 1~2 个职能上提供 bonus。bonus 系数 × `ability/100` 即为该 lead 的实际加成。

| specialty | 主职能 |
|---|---|
| `chief_scientist` | Pretrain / Research |
| `ml_research_lead` | Posttrain |
| `eval_lead` | Evaluate |
| `chief_engineer` | Infra / Product |
| `data_scientist` | Data |
| `marketing_lead` | Marketing |

bonus key 命名规范:
- `*_speed`: 工期除数, 由 `lead_speedup_for()` 统一消费。
- `*_score_bonus` / `*_add` / `*_throughput` / `*_efficiency`: 数值或能力加成, 由各自系统单独消费。

每个 specialty 提供哪些 bonus key、系数取值如何, 权威源是 `resources/data/hiring/lead_bonus/<specialty>.tres`, 装配到 `HiringSystem.LEAD_BONUS_TABLE[specialty][bonus_key]`。

### 1.2 公开 API (跨系统读)

| API | 用途 |
|---|---|
| `lead_speedup_for(lead, task_subtype)` | 返回 task 工期乘子 `1 + (ability/100) × *_speed 系数`; specialty 不匹配返 1.0; player_scientist 总返 1.0 |
| `lead_bonus_coef(lead, key)` | 读 `LEAD_BONUS_TABLE[specialty][key]` 原始系数 (不乘 ability); player_scientist 总返 0.0; 调用方负责 ability 缩放 |
| `lead_matches_specialty(lead, needs)` | player_scientist → true; 否则严格相等 |
| `Lead.is_idle()` | 三态判定: 既未 locked 也未 assigned |

> **player_scientist 例外** (创始人 = 万能 lead): 任意 specialty 校验通行, 但**不提供任何 bonus** (`lead_speedup_for` 返 1.0, `lead_bonus_coef` 返 0.0)。一局只能创建一次, 不收签约费 / 不发工资 / 不能解雇。

### 1.3 真名生成: 名跟随肖像 (族裔 / 性别一致)

lead 的 `display_name` 在 `_gen_lead` (新生) 与 `_migrate_legacy_lead_names` (旧档补名) 两处生成, 规则: **名字必须和这个 lead 的头像一致** —— 头像里是谁, 名字就是哪个族裔 / 性别。

之前名 / 脸脱钩 (头像是多元肖像池, 名字一律中文随机), 一张白人 / 拉丁裔的脸配个 "王美丽" → 出戏。现在两者由同一个 `lead.id` 哈希耦合:

```
lead.id ──FNV-1a 哈希──▶ portrait-NN ──查表──▶ {region, gender} ──PersonName.generate──▶ display_name
         (IconRegistry.lead_portrait_index)   (IconRegistry.LEAD_PORTRAIT_DEMOGRAPHICS)
```

- 肖像池索引与族裔 / 性别表权威源在 `IconRegistry`: `lead_portrait_index(id)` 给出落到 12 张里的哪张, `lead_demographics(id)` 给出该张的 `{region, gender}`。头像本身的具体长相参考 `design/图片素材生成流程.md §8`。
- 名字按 `region` 走对应名库 (`PersonName`, 见 `scripts/systems/person_name.gd`):
  - `east_asian` → 中文姓 + 名 (与 `NameRomanizer._PINYIN` 同一批字, en locale 下转拼音)。
  - `western` / `south_asian` / `hispanic` / `middle_eastern` / `southeast_asian` → 拉丁字母 "Given Surname" (纯 ASCII 拼写, 避免字体缺字; en / zh locale 下都原样显示, 走 `NameRomanizer` passthrough)。
- `gender` 决定取男名池还是女名池; 姓氏池性别共用。
- 6 大区为**粗粒度**: `western` 一个池覆盖白人 / 黑人 / 等所有 Anglophone 头像 —— 头像已经定了脸, 名库只负责给一个英语习惯的名字, **不按肤色再细分名字** (那样反而落入刻板印象, 也无必要)。
- 取名用 `GameState.rng()`, 同 seed 下名字稳定 → 存读档不变、测试可重现。肖像索引是纯哈希 (无 rng)。

> player_scientist (创始人) 的名字是玩家自定义 / 默认 "创始人", 不走本规则, migration 也不覆盖。

## 2. 命令与信号

详见 `命令总线表.md §HiringSystem` 与 `事件总线信号表.md §HiringSystem`。要点:

- `hiring.hire_lead`: 立刻扣 `signing_fee` 走 `economy.spend`, lead 从 pool 移入 leads。
- `hiring.fire_lead`: 不退签约费; player_scientist 不可解雇; busy 中 lead 不可解雇。
- `hiring.adjust_staff`: 减员不会 fire busy 中 staff; 加员立即扣首周工资。
- `hiring.lock_lead` / `release_lead`: TaskSystem 训练独占 (写 `locked_by_task_id`)。
- `hiring.assign_lead` / `unassign_lead`: ProductSystem 常驻 (写 `assigned_to_product_id`)。
- `hiring.lock_staff` / `release_staff`: 增减 `staff_busy[role]`, 由 `holder_id` 标识占用方 (审计用, 不强校验)。
- `hiring.create_player_scientist`: 整局一次, 入参 `specialty` 已废弃只为序列化兼容; 创建后 `specialty = &"founder"`, `is_player_scientist = true`。主 HUD 进入游戏时会自动调用一次, 玩家不再需要在招聘页手动点击「成为创始研究员」。

## 3. 对外依赖

- **读切片** (v7 PR-F): `GameState.cash` (候选池刷新时 S/A 抽取概率, 取代旧的 `MarketSystem.fame`)。设计意图: 公司有钱 → 顶级 lead 主动投递; 公司穷 → 候选池里只有 B/C。
- **调命令**: `economy.spend` (签约费 / 周度工资 / 加员首周工资)。
- **订阅信号**:
  - `phase_started(upkeep)` — 发周度工资 (lead.weekly_salary 之和 + staff.count × SALARY_PER_ROLE), 一次性聚合 spend, reason = `&"salaries"`。
  - `phase_started(action)` — 每 `LEAD_POOL_REFRESH_INTERVAL_WEEKS` (= 4) 周刷新一次 lead_pool (见 §4.1)。

## 4. 内部算法

### 4.1 候选池刷新 (action 相位, 每 4 周)

每 `LEAD_POOL_REFRESH_INTERVAL_WEEKS` (= 4) 周刷新一次 (turn 1, 5, 9, ...; 玩家第 1 周即可看到首批候选, 月初节奏)。刷新时清空 `lead_pool`, 重新抽满 `pool_size` 个候选 — 每个候选 `level` 按 **cash 抽取** (见 4.2), `specialty` 均匀抽 — 再 emit `lead_pool_refreshed`。

候选池**不存档跨周**: 玩家本周内未签约的 lead 在下一次刷新时被替换。

### 4.2 cash → level 抽取 (v7 PR-F, 取代旧 fame brackets)

`pool_config.tres.cash_brackets` 按 `cash_min` 降序匹配, 第一个 `GameState.cash >= cash_min` 的 bracket 决定 S/A/B/C 抽样权重; cash 越高, S/A 权重越大。bracket 区间与权重数值见 `.tres`。

### 4.3 cash 联动设计意图

正反馈环: 营收高 → cash 多 → S/A 概率 ↑ → lead 能力 ↑ → 模型能力上限 ↑ → 排榜 ↑ → 营收 ↑。
反向制衡: S/A lead 的 `signing_fee` 和 `weekly_salary` 高数量级 (S 级 ~¥12M/年, 对标顶级 Chief Scientist)。即使有钱也未必招得起。

## 5. 玩家操作规则

### 5.1 招聘 / 解雇

- 候选池里点「签约」即扣签约费, lead 进 `GameState.leads`。
- 已签约 lead 只能在 idle (三态互斥任何状态都没占) 时解雇; player_scientist 永远不能解雇。

### 5.2 task / product 强制 specialty 校验

各 task 模板的 `input_schema.needs_lead_specialty` 字段声明强制 specialty。**校验由 TaskSystem 与 ProductSystem 各自完成**, 详见 `任务系统设计.md §6.1`。

**已决设计**: 各职能严格按 specialty 对应, 不允许跨职能调度 (chief_scientist 不能当 product lead 等)。**唯一例外**是 player_scientist — 任意 specialty 校验通行, 但无任何 bonus, 等于「占位顶岗」, 鼓励玩家把创始人填补暂时招不到的空位。

### 5.3 UI 输入门控

- 解雇 / 减员只对 idle 的人 / 不超 busy count 的减员生效。
- task 启动 UI 的 lead 下拉**只列匹配 specialty 的 idle lead**; 无匹配时禁用启动按钮。
- 候选池刷新前几周 UI 应给「即将刷新」提示 (具体由场景层实现)。

### 5.4 UI 布局: 招聘 / 员工 两个界面

招聘相关 UI 拆成**两个侧栏 tab**, 按「招新 / 在册」分工:

- **招聘** (`HiringView`): 候选 Lead 池 (按 specialty 分组 + 签约按钮)。这是「往公司里招人」的地方; 创始人本人由主 HUD 开局自动加入团队, 不再作为招聘页 CTA。
- **员工** (`StaffView`): 创始人卡片 + 普通员工 staff 按职能增减 + 本周总工资合计 + 已签约 Lead (按 specialty 分组 + 解雇按钮)。这是「管理在册人员」的地方。
  - 创始人顶部展示与 LeadCard 同规格的「玩家自己」卡片: 头像沿用新游戏选择, 标记 `founder`, 展示万能 lead / 无加成 / 不可解雇 / 当前状态。若旧档或测试数据暂时没有创始人, 员工页只显示自动加入说明, 不引导玩家回招聘页点击按钮。
  - 普通员工是最常用的人力水位调节入口, 放在已签约 Lead 列表之上; 每个 role 行需要同时显示总人数、忙碌数、空闲数、单人周薪与 +1/-1 操作, 操作按钮紧跟该 role 信息, 不推到整屏右侧。

两个 view 都不访问 GameState, 只接 `main.gd` 预算好的 data dict; 业务动作通过信号 (`lead_action` / `staff_adjust`) 反向通知。

## 6. 数据表 (resources/data/hiring/)

数值**表驱动**, `HiringSystem._load_tables()` 在 `_ready` 装配到 `LEAD_BONUS_TABLE` / `SALARY_PER_ROLE` / `LEAD_POOL_BASE_SIZE` / `_pool_config` 缓存。

| 表 | 路径 | Resource 类 |
|---|---|---|
| lead 等级 (×4) | `lead_levels/{s,a,b,c}.tres` | `LeadLevelSpec` |
| 员工周薪 (×5) | `staff_salaries/{ml_eng,infra_eng,data_eng,marketing,ops}.tres` | `StaffRoleSpec` |
| lead 加成 (×6) | `lead_bonus/<specialty>.tres` | `LeadBonusSpec` |
| 候选池配置 | `pool_config.tres` | `LeadPoolConfig` (`pool_size`, `cash_brackets`) |

`LeadPoolConfig.cash_brackets` 是一个 `Array`, 每个元素是普通 `Dictionary` (`{cash_min: int, weights: {level: prob}}`), **没有**单独的 `LeadCashBracket` Resource 类。**改数值改 .tres**。v7 PR-F: 字段从 `fame_min: float` 改为 `cash_min: int`, 旧 `fame_brackets` 字段已删。
