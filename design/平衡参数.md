**设计意图**:
- `per_card_inference_tflops` 按训练 TFLOPs 的约 30% 设, 对齐现实 vLLM / TRT-LLM 实测推理利用率 (含 KV cache + paged attention + 中等批量)。量化红利 (FP8/int8/FP4/int4) 不计入 baseline, 走 engineering 树下沉到 serving 吞吐。
- bamboo (TPU) 现实仅 GCP 租用, 游戏化保留 `purchase_price`, 典型玩法走 cloud DC 租用。
- **三生态定位** (v11): `ecosystem_score` 是训练吞吐乘子, 见 `基础设施系统设计.md §1.5`。cypress 纸面最强 + 生态满分 (1.0) + 最贵; maple 中庸, 生态 0.8 (训练 -20%); bamboo 纸面略低于同期 cypress、价格便宜约 20-30%, 但生态 0.6 (训练 -40%) — bamboo 推理不受惩罚, 适合「便宜铁跑推理 / serving」打法。
- Power (v11, 2 档): H100 700W + PUE 1.4 + $0.07/kWh ≈ $14/卡/周纯电 → `grid` 常规供电取 $14/卡/周, 无前期投入。`green` 绿色能源取 $4/卡/周 (自建光伏 / 风电 LCOE 远低于市电), 但有一次性安装 + 储能费 `install_cost_per_card` = $2600/卡 — 每卡每周省 $10, 回本点 = 2600 / 10 = 260 周 = 5 年。

## DatasetSystem 设计意图

### Pretrain 数据配比 (v9, 2026-05 重做)

权威实现: `TaskSystem._compute_capability_measured` (evaluate) 与 `_scaling_law`。

旧 (~v8) 用"来源 min 乘子"+ tag 按数据集**条数**统计, 问题: 混任一开源就拖到 0.9; 加专精集反而稀释命中比; 自采一份全覆盖集吊打一切; `ds.quality` 在 pretrain 路径完全没用。

v9 改为 **token×quality 加权**, `source` 字段降为审计 / 装饰:
- 按 token 加权 → 大通用集配小专精集时, 专精轴 share 极低, 想堆专精必须真砸 token。
- 按 quality 加权 → 低质量大集与高质量小集在分子里可打平, 玩家选高质量数据有明确收益。
- tag share → ratio 走 log 递减 → 5-20% 专精占比就能拿大部分 ratio, 配比探索有"甜区", 不必强求 100% 专精。
- **通用底盘覆盖** (v10, 2026-05): `raw` 额外乘 `data_breadth_factor`。按 `web/books/encyclopedia/edu/news/arxiv/multilingual/textbook/chat` 等通用知识 tag 的 token×quality share 算覆盖, 目标 share = `DATA_BREADTH_TARGET_SHARE = 0.45`; 达标后 factor=1.0, 0 覆盖时降到 `DATA_BREADTH_MIN_FACTOR = 0.65`, 中间线性插值。意图: code/reasoning 仍是高杠杆, 但纯专精预训练缺少现实模型需要的世界知识/语言底盘, 不再是无代价最优。
- `general` 轴自身仍取满 `raw`, 但 `raw` 已受通用底盘覆盖影响: 任何模型只要训练就有 general 分, 只是纯专精数据的底盘更弱。
- **隐藏商业分析标签**: `business_analysis` 不在 UI 里标注。evaluate 时按该标签 token×quality share 生成小幅轴向折扣: `code` 最多 ×0.96, `reasoning` 最多 ×0.97, `agent` 最多 ×0.95; 线性随 share 插值。`general` / `multimodal` 不变。用来表达“商业分析数据能训练出会开会的模型, 但会稍微伤害严肃 coding / reasoning / agent 表现”。
- `source` 不再进公式 → 自采 / 开源 / 商业差别完全由 size / quality / coverage_tags 表达, 无"来源歧视"。

`TAG_RATIO_LOG_K` / `DATA_QUALITY_FACTOR_FLOOR` / `DATA_QUALITY_FACTOR_CAP` / `DATA_BREADTH_*` / `BUSINESS_ANALYSIS_*` 等常量见代码。cap=1.5 是为防止纯 textbook 数据堆出离谱分数。

### Posttrain 自采人力分档 (2026-05) + 商业集定价

权威实现: `TaskSystem._data_collection_pricing` (posttrain 分支) + `_POSTTRAIN_LABOR_TIERS` 常量。

**为什么分档**: 旧版 posttrain 自采成本只按 token 量 (`30k + size_B × 1M`), **完全不看质量**——玩家把 `target_quality` 拉到 0.9 不要钱, 等于把现实里「质量 = 标注人力成本」的核心规律抹掉; 同样 0.03B@0.85 自采 84k vs 购买 1.5M (≈18× 便宜), 理性玩家根本不买、全自采。

**怎么分**: 后训练数据成本现实里由**标注人力**主导。按**有效质量** (= 玩家选档 `target_quality` + data_scientist lead 加成, 与 `_data_collection_quality` 同源) 分 4 档, 每条单价 = 锚定时薪 × 每条耗时:

| 档 | 有效质量 | 人力 | 时薪 | 每条 |
|---|---|---|---:|---:|
| T1 | ≤0.65 | 众包 | $10/hr | $1 |
| T2 | ≤0.80 | 领域标注员 | $60/hr | $10 |
| T3 | ≤0.90 | 资深专家 | $300/hr | $60 |
| T4 | >0.90 | PhD 专家 | $1000/hr×48min | $800 |

- 定档用**有效质量**而非选档值 → 堵住「T1 选档 + 强 lead 顶到 0.9」用低价拿高质数据的漏洞。
- T1 单价 $1 = 旧公式, 默认档 (q0.65) 成本不变, 向后兼容。
- T4 极贵 (0.03B PhD ≈ 24M, 每 1B token ≈ $800M) → 顶级质量数据无法靠自采便宜地刷, 贴现实 (PhD 标注 $100–300/hr, 游戏取 $1000/hr 强化稀缺)。
- 现实对照: RLHF 偏好标注 $0.5–5/条 (通用) → ~$100/条 (专家级); 领域专家 $40–100/hr, 顶级专才 $100–300/hr; PhD 级标注已成 reasoning/code 数据常态。

**商业集购买价 (2026-05 ×3 上调)**: 购买 = 自采同质量档 × ~2.4-2.7 的便利溢价 (即时拿到 + 不占 lead + 质量不受 lead 波动)。`reasoning_chains_v1` 1.5M→4.5M, `code_review_pairs_v1` 2M→6M, `agent_traces_v1` 3.5M→10M。守护测试 `dataset_collection_dynamic_test.test_commercial_set_price_is_sane_premium_over_self_collect` 钉死买/自采比落在 2-4×。

> **agent_traces size 0.015B→0.06B (2026-05)**: 自采成本按 size 线性, 原 0.015B 太小 → 自采仅 95 万, 10M 买价相对自采崩成 10.5× (没人会买)。加大体量到 0.06B 后自采 ≈ 365 万, 买/自采回到 ~2.7×。即"小而贵"的商业集与"按量计价"的自采天然冲突, 高买价必须配相称的体量。价/体量以 `.tres` 为权威。

### Posttrain 能力增减 (设计意图)

权威实现: `ResearchSystem.simulate_posttrain` (能力增减) + `TaskSystem._posttrain_fixed_tier` (时长) + `resources/data/tasks/posttrain/*.tres` 档位。

**v12 (2026-05) 防刷分重写**: 旧版「逐份相加 gain + 目标轴无上限」可被两招刷爆 —— 堆份数纯线性刷分、且把同样 token 拆成多小份反而收益翻几十倍 (单份 `log2(1+size×1000)` 凹且地板高)。改为「按轴聚合 + 朝软天花板饱和」, 详见 `研究系统设计 §4.2`。

- **按轴聚合**: 同一目标轴的数据集先合并成 `T=Σsize`、`q̄=Σ(qᵢ·sizeᵢ)/T`。→ 拆碎不再有收益 (1×1B ≡ 10×0.1B); 掺低质大份会拉低 `q̄`。
- gain 随 `q̄²` 急剧塌缩、forget 随 `(1−q̄)` 线性 → 低质数据净亏, breakeven 约 q̄≈0.5-0.6。意图: 逼玩家用高质量后训练数据。
- **软天花板**: `ceiling = base_power × POSTTRAIN_CEILING_MULT`, 目标轴 `realized = (ceiling−c0)·(1−exp(−raw_gain/POSTTRAIN_SATURATION_SCALE))`, 永不超 ceiling 且边际递减。`base_power` 绑 pretrain 规模/质量 (减掉 posttrain_delta 防 ratchet), 想刷更高只能训更大/更好基座, 不是堆数据。
- 档位 (s/m/l/xl) 按 size 切, 决定 `min_gpu` 与档位基础周数; **总周数再加 `floor(Σsize / POSTTRAIN_TOKENS_PER_WEEK_B)`** (token 量附加周, 正常单跑 +0)。`POSTTRAIN_ERROR_RATE` 默认 0.02/周, `ml_research_lead` 在场可显著降低。

| 常量 | 值 | 含义 |
|---|---|---|
| `POSTTRAIN_GAIN_K` | 8.0 | 目标轴 raw_gain 系数 |
| `POSTTRAIN_FORGET_K` | 8.0 | 非目标轴遗忘系数 (× max(0,1−q̄)) |
| `POSTTRAIN_CEILING_MULT` | 1.4 | 软天花板 = base_power × 此值 (单轴可专精到整体实力的 1.4 倍) |
| `POSTTRAIN_SATURATION_SCALE` | 35.0 | 饱和速度; raw_gain≈35 时关闭约 63% 的天花板缺口 |
| `POSTTRAIN_TOKENS_PER_WEEK_B` | 1.0 | 每 1B 后训练 token 多占 1 周集群 |

> `POSTTRAIN_RATIO` (v1 比例式) 已废弃, v2 `posttrain_fixed_tier` 取代; 见 `任务系统设计 §6.2`。

## TechTreeSystem 设计意图

权威数据: `resources/data/tech/<tree>/*.tres`; 公式实现在 `TaskSystem` / `model.gd`。

- **context 子树** (D 轴, v7 PR-G): v6 前硬编码 4 档, 现拆成 `ctx_4k..ctx_10m` 5 节点线性树, PretrainDialog 只列已解锁档位。长上下文有 `train_penalty` (训练变慢) 但给 evaluate `agent` 维度一个**加法** `agent_bonus` (仅当 agent gate 通过)；另带 `serving_penalty` (部署后推理吞吐除数, 见上「算力瓶颈回归」一节)。详见 `任务系统设计 §6.7`。
- **Arch capability_cap** (v7 PR-G 陷阱节点): `bert_encoder` / `t5_enc_dec` 等 encoder 世系节点带 `capability_cap` > 0, 把 evaluate `base` 钳死。v11 后 BERT 线是**有限 scale**而不是完全不 scale: `bert_encoder → ... → bert_giant_encoder` 把 cap 从 30 抬到 64, 让玩家投入后确实涨分, 但 dense / MoE 主线 cap=0 无上限, 中后期仍能越过 encoder 天花板。
- **Multimodal method** (E 轴, v7 PR-G): `cross_train` / `pixel_ar` / `diffusion_ar` / `native_ar` 不同方法给 multimodal 轴不同 `capability_coef` 与 `train_penalty`, 由对应 arch 节点解锁; 单模态模型隐式选 `none`, 不影响公式。
- **Engineering 量化累乘**: `flops_per_token_reduction` 多节点累乘, 无显式 clamp, 由 prerequisites DAG 软引导 (int4 须先 int8, fp4 须先 fp8)。**量化不再降质** (v4 PR-B 删 `quality_penalty`)。
- **MoE `active_param_ratio`**: 权威源 `model.gd::ACTIVE_PARAM_RATIO_BY_ARCH`; 预训练完成时 `research.add_model` 一次性写入 `model.active_param_ratio`。dense=1.0, MoE 越稀疏越低 (`octopus_*` 0.25→0.025)。一致性要求: 训练时长、推理 fpt、agent gate 都用此比例, 但 Chinchilla optimal tokens **按总参数算** (MoE 数据更饥渴是合理代价)。

## TaskSystem 设计意图

权威常量在代码 (`TaskSystem`), 模板在 `resources/data/tasks/<subtype>/*.tres`。

- **scaling_law**: Chinchilla `compute = SCALING_LAW_FLOPS_C × params × tokens`, 是全局平衡支柱 (见下文调参注意)。
- **退款**: 取消 task 按已用进度比例退款, `REFUND_COEF` 偏惩罚玩家; 模型生命周期 task base_cost=0 故无退款。
- **staff 边际**: `staff_multiplier = 1 + STAFF_MARGINAL × log2(1 + ml_eng_count)`, 边际递减。`STAFF_MARGINAL` 2026-05 由 `0.15` 调到 `0.03` —— 训练速度主要受 GPU 算力上限约束, 工程师只做效率优化, "多招人"不该是压缩工期的主力杠杆 (旧值 10 人 +52% 过强, 现约 +10%)。
- **Pretrain 模板**: 统一用 `&"pretrain_model"`, 架构由 `arch` 选, 大小由玩家在 PretrainDialog 自由输入 (ModelSizeSpec 已撤回); `flops_per_token = 2 × size_params_M × 1e6` 在 `research.add_model` 时写入。代码 `TEMPLATES` 还有几个 legacy 固定 size 模板仅作单测 fixture, 玩法不暴露。

### LEAD_BONUS 设计意图

权威源 `resources/data/hiring/lead_bonus/<specialty>.tres` (6 个 specialty), 运行时缓存在 `HiringSystem.LEAD_BONUS_TABLE`。

- 加成统一形式 `1 + ability/100 × coef`, lead tier 决定 ability (S/A/B/C ≈ 92/78/60/40)。
- `lead_speedup_for(lead, task_subtype)` 把 task subtype 映射到对应 `*_speed` key (pretrain/posttrain/evaluate/data_collection/tech_research), specialty 不匹配返 1.0。
- 意图: 每个 specialty 主攻一类 task, 强 lead 给可观但非碾压的加速 (顶配 S lead `research_speed` ≈ +50%)。
- **`*_score_bonus` 是分数倍率 (非时长), 同样 `1 + ability/100 × coef`**, 但作用在能力分而非速度上, 让顶级科学家训出的模型**更强**而不只是更快:
  - `pretrain_score_bonus` (chief_scientist): 预训练完成时烘焙到 `model.pretrain_score_mult`, evaluate 放大 raw (`任务系统设计 §6.7`)。
  - `posttrain_score_bonus` (ml_research_lead): 后训练时放大目标轴增益 (`研究系统设计 §4.2`)。
  - `evaluate_score_bonus` (ml_research_lead): evaluate 时的 `lead_eval_acc`。
  - 三者都只对真正 lead 生效, player_scientist (万能 lead) coef 视为 0。

### evaluate 产出 (设计意图)

权威实现: `TaskSystem._compute_capability_measured`。raw = `base × arch_bonus × data_quality_factor × data_efficiency × lead_eval_acc`, 再按 5 轴 (general / code / reasoning / multimodal / agent) 算 `tag_ratio`。

- `base` 随 `log10(size_params)` 增长。`general` 轴恒 1.0; `code` / `reasoning` 走 token×quality 加权 tag share; `multimodal` 由输入模态决定 0/1; `agent` 额外乘工具解锁系数并过规模硬门槛。
- `business_analysis` 隐藏标签不增加任何轴的 ratio, 只按 share 小幅降低 `code` / `reasoning` / `agent` 最终分。
- **capability 无上限**: 各轴只保证 ≥ 0, 不再 clamp 100。100 = "当前 SOTA" 基准, 中后期可达 150~200+。NPC step 增长亦无 cap。
- **Agent 规模门槛**: `AGENT_MIN_ACTIVE_B` / `AGENT_MIN_TOTAL_B` 是激活 / 总参数硬下限, 不达 agent=0。意图: agent 能力需要真正的大模型, 小模型刷不出来。
- **Chinchilla data_efficiency**: 以 `CHINCHILLA_OPTIMAL_RATIO=20` tokens/param (Hoffmann et al. 2022) 为最优点, 欠训按幂指数缩放、过训按对数缓增并 cap。意图: 诚实反映欠训 / 过训的数据效率, 让"砸数据"有正确回报曲线。详见 `任务系统设计 §6.7.1`。

## UserSystem 设计意图 (v7 PR-F, 2026-05 重做)

详见 `用户系统设计.md §5-6`, 数值在 `resources/data/user/tuning.tres` 与 `resources/data/products/types/*.tres`。

**v7 PR-F 整体废 fame**, 改成「排行榜 rank + 时间曲线 + 价格弹性 + 营销」的加法分量模型。

- **排行榜 rank 加成**: 总榜 / 子榜名次给 subscribers 池正负加成 (上榜涨、跌出榜掉), 适用于模型对应的 api 产品 + 所有绑该模型的订阅产品 (统一管道)。
- **基础需求曲线** `base_demand_curve(turn)`: 分段线性, 从 2017 的 0 涨到 2024 后饱和 (#1 模型每周吸收 subscribers 量纲), 只有总榜 top-3 享有, 按名次递减系数分配。意图: 市场总量随时代变大, 早期再强也吸不到多少用户。
- **关键校准**: `marketing_conversion_rate` 取约 $80/用户 CAC (v12: 在 v3 $40 基础上 ×2, 转化率 0.025→0.0125; 订阅产品与 API 共用此率), 营销线性无规模溢价。`api_tokens_per_sub_per_week` 校准到 turn 280 (ChatGPT 时刻) #1 模型月 token 量与真实早期 ChatGPT API 同量级。`orphan_product_churn` 让模型 unpublish 后绑定产品每周硬掉用户, 防僵尸产品。
- **Product token / 价格指导**: 各 product type 的 `tokens_per_user_per_week` / `subscription_price_guidance` / `unlock_thresholds` / 对应 `sub_board` 在 `.tres`。v7 PR-F 把 `*_per_month` 字段显式重命名为 `*_per_week` (旧版已隐式按周解读); 新增 `subscription_price_guidance` (价格弹性公式分母)。chatbot 指导价由旧 99 大幅下调到 20、coding_agent 由 199 调到 100 (用户指定, 鼓励低价倾销策略)。
- **人均 token 需求写实化** (2026-05, 用户指定): 旧版各档 `tokens_per_user_per_week` (chatbot 5000万 / multimodal 1亿 / agent 2亿 / coding 5亿) 比真实人均高 ~50–1000×, 且四档跨度被压扁到不足 1 个数量级。对照真实锚点 (ChatGPT ~7亿周活 / 25亿条·天 ⇒ 人均 ~4万 token/周; 重度 agentic 编程 ~1–3亿/周; Google 全线 980万亿/月) 改为 **chatbot 5万 / multimodal_assistant 25万 / agent 200万 / coding_agent 2亿**, 把跨度拉回 ~4 个数量级。`API_TOKENS_PER_SUB_PER_WEEK` 维持 200万 (小开发者档 ≈ $20–60/月; 现在自然高于休闲聊天用户、低于重度编程, 排序合理)。⚠️ 本次只改需求量纲: 订阅需求降后 `capacity_ratio≈1`, 基建对订阅几乎不再是瓶颈, GPU 服务算力 / per_token 定价的配套再平衡留待后续单独一轮。所有消费方统一优先读 `tokens_per_user_per_week`; `tokens_per_user_per_month` 只是旧别名 fallback, 不再要求 UI/结算直接读取旧字段。
- **算力瓶颈回归 — ×5 token + 长上下文吞吐惩罚 + GPU 提价 30%** (2026-05, 用户指定; 即上一条 ⚠️ 预告的"后续单独一轮"): 写实化后订阅几乎不吃算力 (`capacity_ratio≈1`), "一张桌面卡服务所有人"重现, 失去基建核心循环。本轮把算力拉回瓶颈位:
  - **人均 token ×5**: chatbot 5万→25万 / multimodal_assistant 25万→125万 / agent 200万→1000万 / coding_agent 2亿→10亿; `API_TOKENS_PER_SUB_PER_WEEK` 200万→1000万。`.tres` 的 `_per_week` 是权威值, `_per_month` 旧别名同步同值但只作 fallback; fallback 常量 (`user_system` / `monetization_system`) 同步同值。理由: 写实锚点偏休闲单轮对话, 实际 engaged 用户 + 长上下文 + agentic 多轮使人均显著更高 — 这里偏"算力要紧"的玩法而非严格写实。
  - **长上下文吞吐惩罚** (新机制): context 子树每档新增 `serving_penalty` (推理吞吐除数)。部署模型时按其 `context_length_tokens` 给 `dc.serving_tokens_per_sec` 除以该档惩罚 (4k=1.0 无惩罚)。对照现实: 长上下文 serving 因 KV cache 显存带宽瓶颈急剧掉吞吐。档位 **4k 1.0 / 32k 1.5 / 200k 3.0 / 1M 6.0 / 10M 12.0** (中等偏陡, 比 `train_penalty` 明显更狠)。只作用于玩家自训模型 (OS/NPC release 无 ctx 字段, 视作 1.0)。与人均 ×5 在长上下文产品 (agent/coding) 上叠加 → 长上下文模型 serving 成本显著上升, 鼓励"够用就好"的 ctx 选择, 而非无脑拉满。
  - **GPU 提价 30%**: 所有 GPU `purchase_price` 与 `rent_weekly_cost` ×1.30 (两字段均得整数)。`maintenance_per_week` 为极小 OpEx (3–45, 无法整数化 +30%) 本轮不动。`base_price` 指导价随购卡价摊销项上涨而自然上抬, 与"算力更贵"主题一致。

**已删调参** (v7 PR-F): ❌ `attract_k1` / `attract_k2` / `churn_k` / `base_churn_rate` / `quality_base` / `token_base_per_fame` / `token_per_user_per_month` / `_price_pressure` 公式。

## ResearchSystem — Serving 定价 (v6 PR-E, 2026-05)

权威实现: `ResearchSystem` (`weekly_growth_rate` 等)。

**已删除** (2026-05 PR-E): `MARKET_MARKUP` / `OS_PRICE_HEADROOM` / `max_os_api_price()` 硬钳位。旧逻辑用静态上限强制 clamp 玩家定价, 闭源不受约束、开源被一刀切, 不体现市场弹性。

**新逻辑: 基准价 / 指导价 / 周需求增长率**:
- **基准价 `base_price`** 每年重算一次, 锚定上一年内最新发布的 GPU 的服务成本 (该 GPU 的 **(维护 + 电费 + 购卡价/156 周摊销)** / 推理 TFLOPs × **有效 fpt**)。意图: 定价天花板随硬件进步自然下降, 不再硬编码。CapEx 摊销固定 3 年 = 156 周 (`GPU_AMORTIZATION_WEEKS`), 2026-05 加入 — 旧版只算 OpEx 导致指导价比真实云价低 ~6×。
- **MoE 成本稀疏度 8× 上限** (2026-05): 算 cost 时 active_param_ratio clamp 到 ≥ `MOE_COST_SPARSITY_FLOOR = 0.125` (8×)。octopus_sparse (1/20) / super_sparse (1/40) 因 serving 路由 / KV / batch 浪费, 实际成本下降远不及论文 — 不 cap 的话玩家研究到 super_sparse 后 base 直接塌一个数量级, API 永远是白菜价。其他口径 (训练 FLOPs / serving capacity) 仍用原始 active_param_ratio, **不**受这条 cap 影响。
- 量化自动便宜: engineering 量化下沉到 serving 吞吐, 等价于单 token 成本下降。
- **指导价 `guidance_price`**: 开源 = 5× base_price, 闭源 = 100× base_price (2026-05 调整, 旧版 2× / 40× 太薄, 玩家定个合理价就触 cliff)。
- **周需求增长率** `weekly_growth_rate(r)` (`r = per_token_price / guidance_price`): 倾销 (r≤0.6) 封顶 +4%/周, 指导价 (r=1.0) 持平, 偏高线性掉量, r≥2.5 触发 demand=0 cliff (当周需求归零, 跌价后恢复)。意图: 价格越接近 / 低于指导价越涨用户, 离谱定价即时失去市场。
- **需求池** (v7 PR-F 重写): 价格弹性已并入 `Product.subscribers` 同一池, 由 UserSystem 统一管理 (见 `用户系统设计.md §5`); `Model.demand_multiplier` 字段已删除。每个 published 模型自动绑一个 type=`api` 产品作 API 需求池。

## 调参注意事项

- **scaling_law 是平衡支柱**: `SCALING_LAW_FLOPS_C` 调高 → 训练更慢, 牵动全局节奏。
- **rank / revenue / cash 分工**: 用户基数主要由排行榜 rank + 价格弹性 + 营销驱动; 融资估值 / 信用评级看 revenue + rank; lead S/A 概率看 cash 档。`fame` 已废弃, 不再作为全局上限变量。
- **退款系数**: `REFUND_COEF` 对玩家偏惩罚。若加 task_struggling 强制取消, 应给更高比例。
- **破产深度阈值**: `BANKRUPTCY_DEPTH_FLOOR` 兜底, 可加 `K×weekly_burn + L×quarterly_revenue` 让公司越大容忍越深。
