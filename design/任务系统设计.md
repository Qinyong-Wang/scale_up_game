# 任务系统设计 (TaskSystem)

> **目的**: 提供"长周期 project"的通用框架, 承载 7 个子类型 (`pretrain` / `posttrain` / `evaluate` / `data_collection` / `tech_research` / `charity` / `simulation`)。统一处理资源占用、进度推进、完成派发。
> **读者**: TaskSystem 实现者, 以及任何"训练/后训练/评估/采集/tech 研究"按钮背后的 system。

任务系统是全局最重要的耦合枢纽。它对外只暴露 `task.start` / `task.cancel` / `task.preview` 三个动词, 子类型差异通过 **TaskTemplate** 静态资源 + **完成回调** 处理, 让 ResearchSystem / DatasetSystem / TechTreeSystem 不需要互相知道。

**核心设计点**:
- 模型生命周期 task (pretrain / posttrain / evaluate) **base_cost = 0**, 资源占用 (lead/staff/dc 周费) 即成本。
- pretrain 完成不直接产 capability; 由 evaluate task 产出 (见 §6.7)。
- posttrain 完成时直接修改 `model.capability`, 不再依赖 evaluate 重读 (见 `研究系统设计 §6.2`)。
- 时间单位**周** (1 turn = 1 周); 字段名 `total_weeks / elapsed_weeks / weekly_cost / error_rate_per_week` (2026-05 全 rename, 旧 `*_months / monthly_*` 在 from_dict 兼容)。

## 1. 数据切片

owner = `TaskSystem`。`GameState.active_tasks: Array[TaskInstance]`。

### TaskInstance 字段

| 字段 | 类型 | 语义 |
|---|---|---|
| `id`, `template_id`, `subtype` | StringName | 标识与子类型分发 |
| `started_at_turn` | int | 启动回合 |
| `total_weeks`, `elapsed_weeks` | int | 时长 / 已过周数; `elapsed >= total` 触发完成 |
| `locked_lead_ids`, `locked_staff`, `locked_datacenter_id`, `locked_dataset_ids` | | 启动锁定, 完成/取消释放 |
| `base_model_id` | StringName | posttrain / evaluate 用 |
| `completion_command`, `completion_payload` | | 完成时调用的 CommandBus 命令 + 入参 |
| `base_cost_override`, `weekly_cost_override` | int | 动态费用模板 (data_collection_law) 在 start 时填入, 覆盖模板静态值; 0 = 用模板 |

### TaskTemplate 字段 (静态 .tres, `resources/data/tasks/<subtype>/*.tres`)

| 字段 | 含义 |
|---|---|
| `id`, `subtype`, `display_name` | 标识 |
| `base_cost: int` | 启动一次性费用; 模型生命周期 task 必须为 0 |
| `weekly_cost: int` | 每周持续费用 |
| `duration_func: StringName` | 时长公式名: `scaling_law / posttrain_fixed_tier / evaluate_law / data_collection_law / node_defined / fixed` |
| `base_duration: int` | `fixed` 用; 单位周 |
| `input_schema: Dictionary` | 启动校验 (`needs_lead / needs_lead_specialty / needs_dc / needs_dataset / needs_base_model / needs_target_node`) |
| `output_*` (pretrain 用) | `output_arch`, `output_size_params`, `output_flops_per_token`, `output_input_modalities`, `output_output_modalities`; 在玩家自由输入 size 后这些字段多为占位 |
| `error_rate_per_week: float` | 每周 action 触发 +1 周延期的伯努利概率; 0 = 确定性 |

## 2. 命令与信号

详见 `命令总线表.md §TaskSystem` 与 `事件总线信号表.md §TaskSystem`。要点:

### 命令

- `task.start {template_id, lead_ids, staff, datacenter_id?, dataset_ids?, arch_id?, size_params?, base_model_id?, target_node_id?, ...}` → `{ok, task_id, total_weeks, total_cost, weekly_cost}`
- `task.cancel {task_id}` → `{ok, refund}`: 按完成度部分退款 `base_cost × (1 - elapsed/total) × REFUND_COEF`; 模型 task base_cost = 0, 退款 = 0 (但占用即时释放是真正的退款)。
- `task.preview` (同 start 入参) → `{ok, total_weeks, total_cost, weekly_cost, size_params, arch, display_name, input_modalities, output_modalities, modifier_breakdown, predicted_capability, flops_per_token, warnings}`: 幂等纯读, 不锁资源不扣费。`flops_per_token` 仅 pretrain 分支非零, 与 §6 完成路径同公式 (`Model.infer_flops_per_token(size, active_ratio) / attn_inference_coef`); PretrainDialog 把它喂给 `research.preview_pricing` 算推理成本 / 指导价。

### `task.start` 错误码

| 错误码 | 含义 |
|---|---|
| `unknown_template` | template_id 找不到 |
| `unknown_arch` | pretrain 传入的 arch 找不到或未解锁 |
| `duplicate_model_name` / `invalid_model_name` / `invalid_size_params` | pretrain display_name / size 校验 |
| `insufficient_cash` | 透传 economy.spend (仅 base_cost > 0 模板) |
| `missing_lead` / `lead_busy` / `lead_specialty_mismatch` | lead 校验 |
| `missing_staff` | staff 数量不足 |
| `datacenter_unavailable` | dc 不存在 / 已 training / 已 serving |
| `dataset_required` / `dataset_locked` / `dataset_kind_mismatch` | dataset 校验 |
| `target_capability_required` / `target_size_required` | data_collection_law 路径 (posttrain kind 必须给 target_capability; size 必须为正) |
| `posttrain_datacenter_too_small` | posttrain 选的 dc.gpu_count < 模型 size 档要求 |
| `tech_researchers_too_few` / `tech_engineers_too_few` / `tech_datacenter_too_small` | v6 PR-D tech_research 资源校验 |
| `prerequisite_unlock_missing` | 架构 / tech 节点解锁不足 |
| `base_model_unevaluable` / `base_model_already_published` | posttrain / evaluate base_model 校验 |

### 信号

`task_started(id, subtype)` · `task_progress(id, elapsed, total)` · `task_completed(id, subtype, payload)` · `task_cancelled(id, refund)` · `task_resources_locked(id, locked)` · `task_resources_released(id, released)` · `task_delayed(id, new_total)` (error_rate 命中)。

> `task_completed` 不直接产出资产, 而是通过 `completion_command` 调对应所有者 system 的命令。新增 subtype 不改本系统。

| `data_collection` | "开始采集..." → DatasetCollectionDialog | 数据 | `task.start({template_id: data_collection_dynamic, kind, size, target_capability?, ...})` |
| `tech_research` | tech 节点"研究"按钮 → ResearchDialog | 科技 | `tech.start_research` → `task.start({template_id: tech_research_default, target_node_id, ...})` |

任务卡标题必须按 subtype 走 `strings.csv` 的 `TASK_SUBTYPE_*` 翻译, 覆盖上表全部 7 个 subtype。新增 subtype 时若没有登记翻译, 不允许在 UI 回退显示 raw id (例如 `charity` / `simulation`)。

> **测试 fixture 模板**: 代码 `TEMPLATES` 还注册了 `train_sparrow_s / train_otter_m / train_orca_l / train_elephant_mm / posttrain_general` 几个 legacy 固定大小模板。PretrainDialog / PosttrainDialog **不再使用**; 仅为单测中"固定 size + 固定 dataset"的 fixture 提供方便, 玩法不暴露。

### 4.1 PretrainDialog 关键契约

任意控件变化时 UI 调 `task.preview` (同 `task.start` 入参), 返回值的 `modifier_breakdown` 是 buff / debuff 权威, **必须显示在对话框**, 不只在 tooltip。

`display_name` (玩家输入) 同时作为 `model.id` 与 `model.display_name`。校验:
- trim 后长度 1..40
- 仅允许中英文 / 数字 / `-` / `_` / `.` / 空格
- 不与 `GameState.models[*].id` 或 `active_tasks` 中 pretrain 的 `display_name` 重复

`size` 由 SpinBox + 单位下拉 (`M` / `B`) 自由输入, 不再用 ModelSizeSpec 档位。

`predicted_capability`: preview 用与 §6.7 evaluate 公式**完全一致**的计算 (假设 `posttrain_count=0`, `eval_lead=null`), 返回 5 轴 dict; 让玩家在按"启动训练"前看到 evaluate 后的预估分数。预估值是确定性的, 不掺评估噪声。

`flops_per_token` (pretrain 分支): preview 必须与完成路径同公式产出, 让 PretrainDialog 把它喂给 `research.preview_pricing` → 即时显示 `推理成本 / 指导价 (开源 / 闭源)`, 玩家调整 attention / 大小 / MoE active_ratio 时这两条价格随动 (`研究系统设计 §4.8`)。

**对话框布局**: 左右两栏 — 左 = 训练配置表单, 右 = 实时预览, 各自独立 `ScrollContainer` 自滚。由 `popup_centered_ratio(0.82)` 撑到视口 ~82%, 充分利用宽屏, 玩家调参时同屏看到预览不必上下滚。`min_size` 不超出最小窗口 (1280×720) 兜底, 不再用 `max_size` 截断。

### 4.1.1 PretrainDialog 5 轴控件 (v7 PR-G)

| 轴 | 控件 | 数据源 | 影响 |
|---|---|---|---|
| A `arch` | OptionButton | 已解锁 arch 树节点 | `train_coef` / `inference_coef` / `capability_coef` / `capability_cap` (v7 PR-G) / `active_param_ratio` / `multimodal_method_unlock` |
| B `attention_id` | OptionButton | 已解锁 attention 树节点 (含 baseline `mha_baseline`) | `train_coef` (训练) / `inference_coef` (fpt 分母) |
| C `loss_id` | OptionButton | 已解锁 loss 树节点 (含 baseline `ce_baseline`) | `train_coef` (训练) / `capability_coef` (evaluate) |
| D `context_length_tokens` | OptionButton | `tech.get_context_tiers` 返回的已解锁档位 (含 baseline `ctx_4k`) | `train_penalty` (训练时长乘子) + `agent_bonus` (evaluate agent 加成) |
| E `multimodal_method` (v7 PR-G) | OptionButton | `tech.list_multimodal_methods`; 仅在 `input_modalities` 含非 `text` 模态时启用, 否则 disabled 且固定为 `&"none"`. 选项含 `&"cross_train"` (默认) + 已解锁的 `dit_v1`→`diffusion_ar` / `pixel_ar`→`pixel_ar` / `native_multimodal`→`native_ar`。 | multimodal 维度的 `capability_coef` 乘子 (evaluate) 与 train_penalty (训练时长), 各方法的系数见 `平衡参数.md` |

上下文档位显示文案必须只使用 Godot GDScript 支持的格式化占位符; 不使用 `%g` 等未支持 specifier。带 bonus 的档位显示为 `32k  · agent +2.00 · 训练 ×1.10` 这种固定小数格式, 避免打开 PretrainDialog 时因字符串格式化报错。

`predicted_capability` 5 轴 dict 现在包含 v7 PR-G 的 capability_cap 钳位 + context agent_bonus + multimodal_method 乘子。preview 必须与 evaluate 一致。

## 5. 内部算法

### 5.1 启动 / 推进 / 完成 / 取消

1. `task.start`: 校验 `input_schema` → 锁资源 (任一失败 rollback) → 扣 `base_cost` (若 > 0) → 算 `total_weeks` → 实例化 TaskInstance, emit `task_started + task_resources_locked`。
2. `phase_started(action)`: 滚 `error_rate_per_week` (命中 → `total_weeks += 1` emit `task_delayed`), `elapsed_weeks += 1` emit `task_progress`; `elapsed >= total` 触发 `_complete`。
3. `_complete`: 释放占用 → CommandBus 调 `completion_command` 把产出交给所有者 system → emit `task_completed` → 从 `active_tasks` 移除。失败时**不回滚** task (产出权由所有者负责)。
4. `phase_started(upkeep)`: 对每个 active task 扣 `weekly_cost` (用 `weekly_cost_override` 优先), reason = `task_weekly`。
5. `task.cancel`: 释放占用 + 按完成度退款 (`base_cost × (1 - elapsed/total) × REFUND_COEF`), emit `task_cancelled`。

### 5.2 completion_command 分发

| subtype | command | payload |
|---|---|---|
| `pretrain` | `research.add_model` | `{arch, size_params, attention_id, loss_id, context_length_tokens, multimodal_method (v7 PR-G), dataset_ids, display_name?, input_modalities, output_modalities}` (**不传 capability**) |
| `posttrain` | `research.posttrain_apply` | `{model_id, dataset_ids, datacenter_id, lead_ids}` |
| `evaluate` | `research.evaluate_apply` | `{model_id, capability_measured}` (capability_measured 由 §6.7 公式算出) |
| `data_collection` | `dataset.add` | `{kind, size, quality, coverage_tags, target_capability?, display_name?, source: collected}` |
| `tech_research` | `tech.unlock_node` | `{tree, node_id}` |
| `charity` | `charity.credit` | `{cause_id, amount}` |
| `simulation` | `simulation.complete_stage` | `{stage_id}` |

## 6. 核心公式

### 6.1 scaling_law (pretrain)

训练时长 = 训练总算力 / 有效吞吐, 向上取整为周数。设计意图与各因子的来源:

- **训练总算力**: 由参数量、激活参数比例 (MoE)、数据集 token 量, 再乘 D 轴上下文长度惩罚得出。`active_param_ratio` 权威表见 `model.gd::ACTIVE_PARAM_RATIO_BY_ARCH`; MoE 训练吞吐随激活参数比例缩放, 但 dataset_tokens 不变 (Chinchilla optimal token 仍按总参数算 — MoE 数据饥渴)。
- **有效吞吐**: 以 `dc.train_tflops` 为基础, 乘 arch / attention / loss 三轴的训练系数, 再乘 lead 加速与 staff 乘子。
- `context_length_penalty` (D 轴) 与 `multimodal_method_train_penalty` (v7 PR-G, 仅非 text 输入模态生效) 均由 tech 树节点驱动, 档位数值见 `平衡参数.md` 与 tech 节点 `.tres`。
- `cluster_efficiency` 已在 `dc.train_tflops` 里算过 (基础设施系统设计 §4.1), 训练时长公式直接用 `train_tflops`, **不再二次乘 `cluster_efficiency`** — 否则大集群衰减会被平方, 抵消多卡优势。
- `dataset_quality` 不再进训练时长公式 (v9): 数据质量只影响 evaluate 分数, 见 §6.7。
- 缺失字段全部取 baseline 1.0 (向后兼容)。

### 6.2 posttrain_fixed_tier (posttrain, v2)

按 `base_model.size_params` 查档位表 (`平衡参数.md §POSTTRAIN_TIER_TABLE`) 得档位基础周数。GPU 数是**下限校验**, 高于阈值不缩短时间。lead / staff **不影响时长**, 仅作风险缓冲。

**v12 (2026-05) token 量附加周**: 总周数 = `档位基础周数 + floor(Σ posttrain_dataset.size / POSTTRAIN_TOKENS_PER_WEEK_B)`。即每堆满 `POSTTRAIN_TOKENS_PER_WEEK_B` (B tokens) 后训练数据多占 1 周集群。正常 SFT 单跑 (0.05–0.5B) 附加 0 周, 时长不变; 只有大体量堆数据才吃额外时间。与 `研究系统设计 §4.2` 的「聚合 + 饱和」配合: 既然堆数据不再线性刷分, 时长附加进一步给「想多喂就得多占算力」一个机会成本。

设计意图: posttrain = "个性化高频小循环" (固定档位玩家可规划), pretrain = "大投入低频长循环"。

### 6.3 evaluate_law

评估时长不与模型大小线性相关 (基准集固定): 由常量 `EVAL_BASE_WEEKS` (= 1) 除以 lead 评估加速得出, 仅小幅受 lead bonus 影响, 最少 1 周。常量见 `task_system.gd`。

### 6.4 lead_speedup + staff_multiplier (跨 duration 公式共用)

lead 按其 ability 与 specialty 对应任务子类型的加成系数提供训练加速; staff 按 ml_eng 人数的对数边际收益提供吞吐乘子。两者均为 ≥1 的乘子, 共同进入 §6.1 的有效吞吐。系数表 (`LEAD_BONUS_TABLE`) 与常量见 `平衡参数.md` 与代码。

### 6.5 cluster_efficiency

`dc.cluster_efficiency` 由 InfraSystem 计算 (`gpu.native_cluster_eff × power.efficiency_modifier × big_cluster_decay + chief_engineer.cluster_eff_add`)。见 `基础设施系统设计 §4.1`。

### 6.6 data_collection_law (v2.1)

`base_cost / weekly_cost / weeks / quality / coverage_tags` 在 `task.start` 时由 `_data_collection_pricing(kind, size_b)` 动态计算, 写入 TaskInstance 的 override 字段; `task.preview` 同样调用此函数, 让 DatasetCollectionDialog 预览与启动账单一致。公式表见 `数据集系统设计 §5`。

### 6.7 evaluate 产出公式 (`capability_measured`)

evaluate 任务完成时由 `_compute_capability_measured` 产出 5 轴 `capability_measured`。设计意图概述 (详细公式与常量见 `平衡参数.md §evaluate产出` / `§DatasetSystem 公式` 与 `task_system.gd`):

- **裸基线 `raw`**: 由模型规模决定的 base 分 (受 v7 PR-G 陷阱架构 `capability_cap` 封顶), 再乘 arch 能力系数、loss 能力系数 (C 轴, `ce_baseline=1.0`)、数据质量因子、**通用底盘覆盖因子 `data_breadth_factor`**、Chinchilla 数据效率、lead 评估加成 `lead_eval_acc`、**预训练科学家加分 `pretrain_score_mult`**。
- **预训练科学家加分 (`pretrain_score_mult`)**: chief_scientist 的 `pretrain_score_bonus` 不只让训练更快, 还让训出来的模型**更强**。预训练完成时按当时锁定的 lead `1 + ability/100 × pretrain_score_bonus` 烘焙到 `model.pretrain_score_mult` (无 lead / player_scientist → 1.0), evaluate 时整体放大 `raw`。冻结在预训练那一刻 (后续换 lead 不改已训模型)。PretrainDialog 在"分数"区显示"科学家训练加分"。Per 招聘系统设计 §5.1。
- **按轴分配**: general 轴固定取满 `raw`; code / reasoning / agent 轴按对应数据标签的占比折算; multimodal 轴仅在含 image 模态时按 multimodal 训练方法系数给分。
- **v10 数据配比**: 标签占比与数据质量因子都按 `token × quality` 加权 (不再按数据集条数, source 字段不再进公式), 标签占比再过 log 递减。意图: 加专精集不再稀释分母, 要堆某轴必须真砸 token; 来源不再歧视, 高质量小集在加权平均里能显形; log 递减让 5-20% 占比就拿到大部分增益, 给配比探索甜区。`general` 轴固定取满 `raw`, 但 `raw` 会被通用知识覆盖不足惩罚: 通用 tag (`web/books/encyclopedia/edu/news/arxiv/multilingual/textbook/chat`) 的 token×quality share 达到约 45% 才视为底盘完整; 0 覆盖时整体 raw 约 ×0.65。这样 code/reasoning 专精仍有效, 但不能完全替代世界知识与语言底盘。
- **v7 PR-G 加成**: agent 轴在过门槛后额外叠加 context 树的加法 `agent_bonus`; multimodal 轴乘 multimodal 训练方法系数。

**v7 PR-G 三处修改 (保留)**:
- `base` 钳到 `arch.capability_cap` 让 BERT/encoder-decoder 等架构再大也上不去 — 模拟历史范式淘汰。`capability_cap = 0` (默认) 表示无上限。
- `context_agent_bonus` 是**加法**而非乘法, 因为乘法会与小模型相互作用过大; 加法只在过 agent gate 时生效 — 大模型 + 长上下文 → agent 分上一个台阶, 小模型解锁 ctx 拿不到 agent bonus (因为 agent=0)。
- `multimodal_method_coef` 只乘到 multimodal 轴 — 训练时间的 train_penalty 在 `_scaling_law` 里另结, 这里只管评估分数。

**capability 无上限**: 各轴只保证 ≥ 0, 不再 clamp 到 100。100 = 当前 SOTA 基准, 中后期可达 150~200+。NPC step 增长亦无 cap。

**Bug B fix (2026-05)**: 不再有 `posttrain_lift = (1 + 0.10 × posttrain_count)` 乘子。posttrain 的轴向 +target / -forget 通过 `model.posttrain_delta` 累加, 在 `research.evaluate_apply` 时叠加 (见 `研究系统设计 §6.3`)。

### 6.7.1 Chinchilla data_efficiency

按"实际 token / Chinchilla 最优 token"的比值给数据效率系数: 欠训时按幂函数塌缩 (大模型不凑足 token 总分会被腰斩), 过训时只有微小对数加成 (数据满了就该开新方向, 不要无意义复用同一份数据), 整体有上限。公式与常量见 `平衡参数.md` 与代码。

### 6.7.2 agent 轴硬门槛

agent 轴有一道由总参数量与激活参数量 (`Model.active_param_ratio_for(arch)`) 共同把守的硬门槛: 任一项低于阈值则 agent 直接归零。阈值常量见 `平衡参数.md` 与代码。

设计意图: 现实里小模型无论怎么蒸馏 tool-use 数据都做不出像样 agent, 因为缺乏「持有上下文 + 多步规划」所需的容量。只有足够大的 dense 或 MoE-routing 模型才能跨过门槛。其他 4 轴不受影响 — 小模型可以是不错的聊天 / 代码工具, 但不会成为合格 agent。
