# 图片素材生成流程

> **目的**: 给出一条可重复执行的「**写 prompt → 出 raw 图 → harness 后处理 → agent 读图 inspection → 接受接入**」操作流程, 让任何一批 UI 图标都能以一致的风格产出、被客观验收。
> **读者**: 负责生成图片素材的人 + 帮忙写 prompt / 跑 harness / 读图验收的 agent。
> **关系**: 本文是**操作手册 (how)**; 目录契约、harness 行为与机械 QA 标准见 [图片素材管线设计.md](图片素材管线设计.md) (what / 契约), 不在此重复。UI 接入路径见该文 §5。
> **状态**: 🟡 v0 — 首批落地「18 档机房建筑图标」, 流程通用于后续 GPU / 供电 / 模型 / 产品 / 事件批次。

---

## 1. 角色分工

| 角色 | 负责 |
| --- | --- |
| **人 (你)** | 提供 `REPLICATE_API_TOKEN`、对 agent 的验收结论做最终接受决定。**不手动喂模型** (生成已自动化)。 |
| **agent (Claude)** | 写 prompt/subject、跑 `tools/art/generate.py` (调 Replicate 出图) + `process_asset.py` (后处理)、**用 Read 工具读图做 inspection**、写验收结论、接受后接入代码 + 补测试。 |
| **harness** | 确定性后处理 (去洋红背景 / 裁切 / 居中) + 机械 QA (四角 alpha=0 / 不触边 / 写 `pipeline-meta.json`)。**不生成美术。** |

一句话流程 (全自动化, 见 §5):

```
prompt (内置 BATCHES + 统一 STYLE 前缀)
   │
   ▼
generate.py 调 Replicate 出 raw 图 (洋红底)  ──►  tools/art/runs/<label>/raw.png
   │
   ▼
process_asset.py  ──►  <label>.png + pipeline-meta.json    (--process 可链式)
   │
   ▼
agent 用 Read 读图 inspection  ──►  机械 QA(harness) + 语义/审美 QA(agent 肉眼)
   │            │
   │            └─ 不过 → 调 subject / 换 seed / 换模型 / 调 harness 参数 → 重跑 (不手 P)
   ▼
人接受  ──►  PNG 进 assets/sprites/ui/<类>/  ──►  agent 接入 .tres icon_path + Avatar + 测试
```

## 2. 风格一致性规范

一致性是「一套图标看起来像一套」的硬指标, 由三层共同保证:

1. **同一 style 前缀** — 所有 prompt 共用下面这段固定前缀 (§3.1), 任何批次不改它的视角 / 光照 / 色板措辞。
2. **同批一次出图** — 优先把同批图标放进**一张 sheet 一次生成** (§4.2), 一次扩散内部天然一致, 远胜 N 次独立生成再拼。
3. **同一生成设置** — 同批固定同一模型 / 同一 seed 家族 / 同一尺寸, 记进 `prompt.txt` 顶部注释 + `pipeline-meta.json`, 便于复现与回归。

风格 token (沿用 `图片素材管线设计.md §2.1` 的 `clean_hd_ui`):

- 现代扁平 2D 游戏 UI 图标 / editorial tech 插画, 不是像素风、不是写实照片。
- **统一 isometric 3/4 俯视**, 柔和棚拍光, 单一主体居中, 大留白。
- 冷色科技色板 (石板蓝 / 青 / 浅灰, 少量 cyan 高光), 边缘清爽如矢量。
- 背景**纯 `#FF00FF` 洋红**、无渐变 (harness 据此去背); 若生成器支持直接导出透明 PNG, 可省略洋红, harness 仍负责裁切居中。
- 无文字 / 数字 / logo / 真实品牌 / 真人 / 噪点。图标缩到 40–48px (Avatar) 仍可辨认。
- **化名规范** (CLAUDE.md): 文件名、prompt 存档、manifest、UI 文案不出现真实品牌 / 真实型号。建筑是通用结构, 不画任何真实公司大楼 / logo。

## 3. Prompt 写法

每条 prompt = **固定 style 前缀** + **该资产 subject** + **构图约束** + **negative**。subject 是唯一逐图变化的部分。

### 3.1 固定 style 前缀 (复制即用, 不改措辞)

```
clean HD 2D game UI icon, modern flat editorial tech illustration, soft studio
lighting, isometric 3/4 view, single centered subject with generous margin,
cohesive cool palette (slate blue, teal, soft cyan accents, light grey),
crisp vector-like edges, subtle soft shadow under the subject only,
no text, no numbers, no logos, no real-world brands, no people,
solid flat #FF00FF magenta background.
```

### 3.2 negative (复制即用)

```
photo, realistic photography, busy or gradient background, scenery, sky, planet,
starfield, watermark, text, letters, numbers, logo, brand mark, people, blurry,
noisy, low-res, multiple disconnected objects, subject touching the border.
```

### 3.3 sheet 模式追加约束

整批一张图时, 在 subject 之后追加:

```
arranged as a clean grid of separate icons, each icon centered inside its own
cell with clear margin, icons do NOT touch each other or the cell borders,
identical style / lighting / scale logic across all cells.
```

### 3.4 命名约定

- 文件 / sheet label 用**英文小写连字符**, 由 `.tres` 的 `id` 推导: `facility_solo` → `facility-solo`, `facility_rack_16` → `facility-rack-16`。harness `sheet --labels` 按此顺序产出 `<label>.png`。
- 反推 `id` 只需 `-` 换 `_`; 但接入时**以 `.tres` 显式 `icon_path` 字段为准**, 不靠文件名硬推 (见 §7)。

整批一张 **6 列 × 3 行 = 18** 的 sheet, 按 tier 行优先排列:


## 5. 去哪生成 (Replicate, 全自动化)

生成走 **Replicate API**, 由 `tools/art/generate.py` 全自动驱动: 内置批次的 subject + §3.1 统一 STYLE 前缀 → 调模型 → 下载 raw PNG (洋红底) → 写 `prompt-used.txt` + `gen-meta.json`。真实模型名只出现在 `generate.py` 里 (化名规范)。

```bash
# 依赖 (本目录 venv, gitignore): python3 -m venv tools/art/.venv && tools/art/.venv/bin/pip install replicate pillow requests
export REPLICATE_API_TOKEN=r8_xxx        # 或写进 tools/art/.env (gitignore, 不提交)

tools/art/.venv/bin/python tools/art/generate.py --list                        # 看批次与每档 prompt
tools/art/.venv/bin/python tools/art/generate.py --probe                        # 自检所选模型真实入参
tools/art/.venv/bin/python tools/art/generate.py --batch infra_buildings --process   # 18 档, 生成+后处理
```

模型 preset (在 `generate.py` 的 `MODELS` 表里, 真实名仅此处):

- `flux` (默认) — 提示词跟随强, 对"纯洋红底扁平图标"听话, 单图约几美分。
- `flux-dev` — 更省, 质量略低。
- `recraft` — 扁平 / vector 图标专长, 用 `style` 控一致性 (无 seed)。

一致性: 同批走**同一 seed** (批次 `seed` 字段) + 同一 §3.1 前缀, 让 18 档风格统一。换批 / 重出某档用 `--only <label> --seed <n>`。`generate.py` 把模型 / seed / prompt 全记进 `gen-meta.json` + `prompt-used.txt`, 可复现可追溯。

> 自动化下**首选逐档 single** (每图高清、harness `single` 稳、reject-edge-touch 可靠); §4.2 的 sheet 布局保留为人工或一致性兜底方案。

raw 图落盘约定 (见 `图片素材管线设计.md §3`): `tools/art/runs/<label>/raw.png`。

## 6. 后处理 (harness)

机械处理与 QA 全部交给 `tools/art/process_asset.py` (行为契约见 `图片素材管线设计.md §4`)。建筑批推荐命令:

**sheet (首选, 18 格一次切):**

```bash
python3 tools/art/process_asset.py sheet \
  --input tools/art/runs/infra-buildings/raw-sheet.png \
  --output-dir tools/art/runs/infra-buildings \
  --rows 3 --cols 6 \
  --labels facility-solo,facility-pod,facility-rack-16,facility-rack-32,facility-rack,facility-room,facility-hall,facility-floor,facility-building-s,facility-building-m,facility-building-l,facility-campus-s,facility-campus-m,facility-campus-l,facility-metropolis,facility-space-s,facility-space-m,facility-space-l \
  --size 128 --fit-scale 0.82 --component-mode largest --trim-border 6 \
  --reject-edge-touch \
  --prompt-file tools/art/runs/infra-buildings/prompt.txt
```

**single (补单格 / 重出某一档):**

```bash
python3 tools/art/process_asset.py single \
  --input tools/art/runs/facility-solo/raw.png \
  --output-dir tools/art/runs/facility-solo \
  --name facility-solo \
  --size 128 --fit-scale 0.82 --component-mode largest \
  --reject-edge-touch \
  --prompt-file tools/art/runs/facility-solo/prompt.txt
```

参数取舍:

- `--size 128`: Avatar 最大 96px, 128 留余量且偶数好缩放。
- `--component-mode largest`: 只保留最大连通主体, 滤掉零散噪点 / 漂浮残块。
- `--fit-scale 0.82`: 主体占 82%, 四周留白, 在 40–48px 头像里不顶边。
- `--trim-border 6` + 默认 `--edge-clean-depth 2`: 吃掉格子边缘的洋红残边。
- `--reject-edge-touch`: 主体触边直接判失败, 逼着回到 §3 调构图而不是凑合。
- `--background-mode sampled --background-tolerance 45`: 对非纯洋红 / 棚拍底 / 渐变底优先使用。它只移除边缘连通背景, 不全局高阈值抠色, 避免头像脸部、奖杯把手、收藏品边框被吃掉。不要把容差常态化拉到 `75–85`; 这会把浅色气泡 / 纸张 / 高光也当成背景吃掉。
- `--defringe`: 头像 / 奖杯 / 奖章默认 0; 标志、公益、模拟图标可用 1。大于 1 容易削断细结构。

harness 退出码非 0 或 `pipeline-meta.json` 里 `ok=false` ⇒ 该批/该格不通过, 进入 §6/§3 重出, 不接入。

## 7. agent 读图 inspection

harness 只做**机械 QA**; 语义 / 审美由 **agent 用 Read 工具读图肉眼判**。两层都过才算可接受。

**第一层 — 机械 QA (harness, 客观)**: 先读 `pipeline-meta.json`, 确认 `ok=true`、`edge_touch=false`、四角 alpha=0、每格都有输出。透明度以此为准 (Read 渲染时可能把透明区合成到底色上, 不作为透明判据)。

**第二层 — 语义/审美 QA (agent, 肉眼)**: 用 `Read` 逐张看 `<label>.png`, 再 `Read` 一次 `sheet-transparent.png` 看整批一致性。逐项核对:

1. **主体正确** — 画的就是该档对应的建筑 / 规模, 且**尺度逐档递进**合理 (单卡 → 机架 → 机房 → 建筑 → 园区 → 都市 → 太空)。
2. **居中留白** — 主体居中、四周留白, 不贴边、不被裁掉一角。
3. **小尺寸可读** — 设想缩到 40–48px 仍能认出是哪一档, 细节不糊成一坨。
4. **风格一致** — 与同批其它图同一视角 / 光照 / 色板, 像出自一套。
5. **背景干净** — 无洋红残边、无灰边、阴影只在主体下方不投到背景。
6. **无违规** — 无文字 / 数字 / logo / 真实品牌 / 真人。
7. **alpha 完整** — 头像、奖杯、奖章、收藏品不能有透明裂缝或缺块; 细柄 / 把手 / 绶带 / 画框边缘不能被抠断。
8. **浅色主体保全** — 聊天气泡、纸张、画布、人物衣服、奖章绶带这类大面积浅色主体必须是实心连续的; 若只剩描边或出现大片 checkerboard 透空, 判为去背失败。

**判定与回路**: 任一项不过 ⇒ 记下是哪一格 + 哪条不过, 回到 §3 调 prompt / 换 seed, 或回 §6 调 harness 参数 (`--trim-border` / `--fit-scale` / `--component-mode`) 重出该格——**不手动修图**。全过 ⇒ 写一句验收结论 (例: "18/18 通过" 或 "16/18 通过, space-m/space-l 主体偏小待重出"), 把不过项记进提交说明。

## 8. 接受与接入

人确认验收结论后:

1. **落盘**: 通过的 PNG 从 run 目录拷进 `assets/sprites/ui/infra/` (建筑批; GPU/供电后续同目录, 靠 `id` 前缀区分)。运行时只依赖这里, 不读 `tools/art/runs/`。
2. **接入** (按 TDD, 详见 `图片素材管线设计.md §5`):
   - 给 `FacilitySpec` (`scripts/resources/facility_spec.gd`) 加可选 `@export var icon_path: String`, 各 `.tres` 填 `res://assets/sprites/ui/infra/facility-<id>.png`。
   - `dc_card.gd` 把 facility 的 `icon_path` `load()` 后传给 `avatar.texture` (现在传的是 `null`); 缺图 / 加载失败 → 继续走 seed 配色 + glyph 回退, 不影响老存档。
   - 补 UI / view 测试, **覆盖 有 texture / 无 texture 两条路径** (回退路径已有 `avatar` 测试可参照)。
3. 先单元测试 → 再集成测试 (含 infra_view 渲染), 全绿才算这批接入完成。

机房建筑用 `FacilitySpec.icon_path` (按档位 id 取图); 其余按**类型**复用的图标改走统一 `IconRegistry` (见 §8bis)。

- 生成: `generate.py` 内置同名批次 (`datasets`/`products`/`tech`/`tasks`/`gpu`/`power`/`leads`/`charity`/`simulation`)。label = `<category>-<key>`, 接受时落到 `<category>/<key>.png`。营销活动图标可按单图 label `marketing-campaign` 生成, 接受后落 `assets/sprites/ui/marketing/campaign.png`。**`simulation` 批用专属 `sim` 风格** (STYLE_SIM): 5 阶主题 (天气/海洋/地球/太阳系/宇宙) 天然撞 icon 的 "no scenery/sky/planet/starfield", 故画成**孤立全息数据可视化球体** (内容收在球内、背景不撒星点), 仍纯洋红去背; `charity` 走默认 `icon` 风格。两批都 `flood_bg + component_mode all`。
- 例外: `brand` 与 `task` 两类是新游戏 / 任务卡里的小尺寸功能图标, 不再走 AI raw 图 + 抠图。它们由 `tools/art/build_deterministic_ui_icons.py` 确定性绘制为 128×128 透明 PNG, 必须满足 `图片素材管线设计.md §4.6` 的实心剪影与 alpha 填充率门槛。若重新尝试 AI 图, 也必须先过同一 QA。
- **批次级 harness 旗标** (见 `generate.py` BATCHES):
  - `reject_edge: False` — 主体本就铺满画面的批次 (复杂图标 / 人物半身像); 不按触边判废, harness 仍裁内容 bbox + 居中 + 缩 82%。
  - `component_mode: "all"` — 人物肖像必须用 `all`; 用 `largest` 时头与身躯断开会丢头。
  - `background_mode: "sampled"` — 写实头像、标志、公益 / 模拟图标、办公室道具、收藏品等非纯洋红底批次使用边缘采样去背; 禁止再靠高 `threshold` / `edge_threshold` 全局吃背景。默认 `background_tolerance` 控制在 `45` 以内; 更高值必须逐图验收, 否则会把浅色主体从边缘连通抠掉。
  - `style: "portrait"` — lead 用 §3 之外的肖像风格 (`STYLE_PORTRAIT`)。**坑**: "avatar/bust/die-cut sticker/soft shadow" 这类词会让 flux 在人物背后画一圈浅色圆形徽章 / 棚拍底 / 光晕, 那圈不是洋红、去背吃不掉, 在头像里留个浅色方块 (用户会看到"空白背景")。`STYLE_PORTRAIT` 已显式禁止 circle/badge/frame/border/sticker outline/backdrop/shadow 并要求洋红 full-bleed 铺满到人物轮廓。即便如此偶有个别 seed 仍画彩色底 (如 portrait-07 的橙/粉盘); 这类用 `--edge-threshold 200` 让 flood-fill 从洋红边吃掉彩色底 (肤色距洋红 ~210 不受影响), 或换 seed。
  - lead 是**多元肖像池** (`portrait-01..12`, 性别均衡 + 族裔/年龄多样), 按 `lead.id` 哈希分配, 而非按 specialty (否则同 specialty 多人同脸, 且整体性别/族裔单一)。
- **坑**: `process_asset.py` 的 `--name` 会被 `sanitize_slug` 把 `_`→`-` (如 `product-coding_agent` 落成 `product-coding-agent.png`); 接受时按**原 label** 切出 `category`/`key` (key 保留下划线) 再拷, 别用 sanitize 后的文件名当 key。
