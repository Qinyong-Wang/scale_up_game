# 事件总线信号表

> **目的**: 集中列出所有 EventBus 信号、载荷、发出方、已知订阅方。是 fan-out 通知契约的总览。
> **读者**: 加新订阅前确认载荷; 加新信号必须先在本表登记。

约定:
- 信号名格式 `<domain>_<past_tense_verb>` (过去时), 表示 "X 发生了"。
- emit 完不关心订阅者是否处理成功 — 通知是单向的。
- 多个订阅者同时订阅; 顺序未指定 (除非通过信号链显式控制, 见 `系统耦合矩阵.md §4.1`)。
- `event_bus.gd` 是跨系统信号声明表, 信号通常由其他 system / UI emit 或 connect, 不在该类内直接使用; 声明区统一抑制 Godot 的 `unused_signal` warning。

## 关键信号链

### `users_resolved → revenue_resolved`

唯一显式信号链, 用来锁死 UserSystem → MonetizationSystem 的执行顺序 (UserSystem 必须先把 paid_users / token_demand 算完, 营收结算才有数据可用)。订阅在 MonetizationSystem._ready 里固定。

### `model_published / model_unpublished → 即时重排`

MarketSystem 订阅这两个信号, 收到后**只重排榜单** — 玩家 publish 完立即看到名次。(v7 PR-F: fame 字段废弃, 不再有"按 rank 发 fame"步骤; UserSystem 直接读 rank 驱动需求池。)

ProductSystem 也订阅 `model_published`: 自动重绑 `auto_track_latest = true` 的同类产品 + 自动创建 API 产品 (model 的对外 API 开关)。`model_unpublished` 时 ProductSystem 静默删 API 产品。

### `task_cancelled → tech_research_cancelled`

TechTreeSystem 订阅 task_cancelled, 把 researching_nodes 里对应的占位清掉。取消 tech_research 类型 task 通过这条链走, 不需要专门 cancel 命令。

### `tech_unlocked(engineering) → dc_compute_recomputed` (v4 PR-B)

InfraSystem 订阅 tech_unlocked, 当 tree == `engineering` 时遍历所有 `status == serving` 的 dc 重算 `serving_tokens_per_sec` (新解锁的 `throughput_multiplier` / `flops_per_token_reduction` 立即反映到 dc 卡片 + MonetizationSystem 算力池)。

## 命名风格

- ✅ 过去时: `*_added` / `*_resolved` / `*_completed` / `*_changed`。
- ✅ domain 前缀: `lead_*`, `dataset_*`, `model_*`, ...
- ❌ 不要现在时 (`add_lead`, `resolve_users`)。
- ❌ 不要带返回值 — 信号是 fire-and-forget; 要返回值用 CommandBus。
