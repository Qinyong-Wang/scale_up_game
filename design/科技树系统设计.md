# 科技树系统设计 (TechTreeSystem)

> **目的**: 管理 6 条独立的技能树 DAG, 每条树由不同系统消费。提供"启动研究"入口 (转发 TaskSystem), 研究完成时记录解锁。
> **读者**: 科技树面板 UI、TaskSystem (`tech_research` 子类型 + 训练读取 arch / attention / loss / context coefs)、ResearchSystem、InfraSystem (engineering 乘子)、MonetizationSystem、ProductSystem (application 解锁)。

---

## 0. 六棵树职责 (v7 PR-G 扩张)

| 树 id | 消费者 | 节点举例 |
|---|---|---|
| `arch` | 训练时长 (train_coef) + 评估能力 (capability_coef + **capability_cap** 上限封顶) + active_param_ratio (MoE) + multimodal_method 解锁 | dense (ant_v*) / MoE 系 (octopus_v*/sparse/super_sparse) / **有限 scale 陷阱**(bert_encoder 系 / t5_enc_dec 系) / **多模态生成** (dit_v1/pixel_ar/native_multimodal/cross_train) / **未来** (world_model_v1/embodied_v1) |
| `attention` (PretrainDialog B 轴) | 训练 train_coef + 推理 inference_coef (pretrain 完成时折进 model.flops_per_token) | mha_baseline / gqa / mqa / mla / hybrid_attention / **infinite_attn** (10M+) |
| `loss` (PretrainDialog C 轴) | 训练 train_coef + capability_coef (轻度) | ce_baseline / zloss / mtp / **dpo / rlvr / o1_rl** (reasoning RL 链) |
| `engineering` | 推理 (throughput_multiplier + flops_per_token_reduction, 下沉到 `dc.serving_tokens_per_sec`) | KV cache / paged attn / continuous batching / spec decoding / fp8 / fp4 / int8 / int4 / **bit2_quant / analog_compute** (2030+) |
| `application` | 解锁 product type / 触发 agent 轴工具使用加成 | tool_use / fox_code_specialist |
| `context` (新, PretrainDialog D 轴解锁) | PretrainDialog D 轴 (4k/32k/200k/1M/10M) 可选档位 + evaluate `agent` 维度加成 (context_agent_bonus, 加在 raw 之后) | ctx_4k (baseline) → ctx_32k → ctx_200k → ctx_1m → ctx_10m |

**legacy alias** (代码内保留, 不在设计文档使用): `inference → engineering`, `agent → application`, `transformer_v0/v1/v2 → ant_v1/v1/v2` (映射见 `tech_tree_system.gd` 顶部)。

**节点命名 vs 玩家可见效果分离**: 内部 id 走化名规范 (动物族); UI 只显示 `effects_summary` ("训练 +20% · 推理 +10% · 代码能力 +5"), 玩家凭数字决策, 不要求懂技术原理。

## 1. 数据切片

owner = `TechTreeSystem`。

| 字段 | 类型 | 语义 |
|---|---|---|
| `unlocks` | `Dict[tree: StringName, Dict[node_id, bool]]` | `unlocks[tree][node_id] == true` 表示已解锁 |
| `researching_nodes` | `Dict[tree, Dict[node_id, task_id]]` | 防同节点并发研究 |

`unlocks` 初值包含 baseline 节点默认解锁 (`ant_v1` / `mha_baseline` / `ce_baseline`), 由 `_ensure_baseline_unlocks()` 兜底。

### TechNode 字段 (静态 .tres 模板, `resources/data/tech/<tree>/*.tres`)

| 字段 | 含义 |
|---|---|
| `id`, `tree`, `display_name`, `description` | 标识 + flavor |
| `prerequisites: Array[StringName]` | 同树前置节点 (跨树前置不允许) |
| `research_cost: int` | 历史字段, v6 后保留 0 (锁 dc 即成本) |
| `research_months: int` | 时长 (历史名, 单位**周**); v6 范围 [24, 48], baseline 节点 = 0 |
| `min_researchers, min_engineers, min_gpu_count` | v6 准入门槛 (ml_eng / infra_eng / dc.gpu_count) |
| `effects: Dictionary` | 数值效果 (各树不同 key, 见下) |
| `effects_summary: String` | 玩家可读效果文案 (设计师手写或自动生成) |
| `capability_cap: float` (v7 PR-G) | arch 树专用; > 0 时把 `_compute_capability_measured` 的 `base` (size scaling 项) 钳到该上限, 用于 BERT/encoder-decoder 等不可 scale 的陷阱架构。0 = 无上限 (默认)。|

### `effects` 字段按树分类

**arch**: `train_coef`, `inference_coef`, `capability_coef`, `multimodal_method_unlock` (v7 PR-G: StringName, 解锁 PretrainDialog E 轴的某种 multimodal_method)。`capability_cap` 是 TechNode 顶层字段, 不放在 `effects`。已删除未消费的 `capability_dim_bonus` / `modality_unlock` 节点; 多模态训练入口由 PretrainDialog 的 `input_modalities` 与 E 轴方法承担。

**attention / loss**: `train_coef`, `inference_coef` (attention 用作 fpt 分母, 与 engineering 的 `flops_per_token_reduction` 区别: attention 在 pretrain 完成时一次性写进 model.flops_per_token; engineering 运行时累乘到 dc), `capability_coef` (loss)。

**engineering** (v4 PR-B): `throughput_multiplier` (≥1, 推理算力乘子; KV cache / batching / spec decoding), `flops_per_token_reduction` (≤1, 单 token FLOPs 缩减; FP8 / FP4 / INT8 / INT4, 多节点累乘)。**量化不再降质** (v4 删 `quality_penalty`)。2026-05 清理: `train_efficiency` 没有任何训练公式消费, 对应纯训练效率节点删除。

**application**: 不再使用 `effects` 作为权威消费面。产品解锁读取 `ProductTypeSpec.application_node_required` 指向的节点 id; TaskSystem 额外读取 `application/tool_use` 解锁状态给 agent 轴工具使用倍率。已删除未消费的 `product_quality_bonus` / `capability_threshold_lower` 路线节点。

**context** (v7 PR-G 新): `max_tokens` (int, 解锁 PretrainDialog D 轴该档位), `train_penalty` (float, 该档位训练时长乘子, 与 `TaskSystem._CONTEXT_LENGTH_PENALTY` 表对齐), `agent_bonus` (float, evaluate 时 `agent_axis += agent_bonus × passes_gate`, agent gate 见 §6.7.2), `serving_penalty` (float, 2026-05 新增: 部署后推理吞吐除数, `InfraSystem._refresh_serving_capacity` 按部署模型 `context_length_tokens` 查表对 `dc.serving_tokens_per_sec` 缩减; 4k=1.0)。

## 2. 命令与信号

详见 `命令总线表.md §TechTreeSystem` 与 `事件总线信号表.md §TechTreeSystem`。要点:

### 命令

- `tech.start_research {tree, node_id, lead_ids, staff, datacenter_id}`: v6 必填 dc; 内部转发 `task.start({template_id = tech_research_default, target_node_id, target_tree})`; 错误码: `unknown_node` / `already_unlocked` / `already_researching` / `prerequisites_missing` / `tech_researchers_too_few` / `tech_engineers_too_few` / `tech_datacenter_too_small` (后三个由 TaskSystem 校验透传)。
- `tech.unlock_node {tree, node_id}`: TaskSystem 完成回调, UI 不调。
- `tech.is_unlocked {tree, node_id}` / `tech.list_available {tree}`: 纯读。
- `tech.get_arch_coefs {arch_id}` → `{train_coef, inference_coef, capability_coef}`。
- `tech.get_attention_coefs {attention_id}` → `{train_coef, inference_coef}` (PretrainDialog B 轴)。
- `tech.get_loss_coefs {loss_id}` → `{train_coef, capability_coef}` (PretrainDialog C 轴)。
- `tech.get_engineering_coefs {node_ids?}` → `{throughput_multiplier, flops_per_token_reduction}`; 缺省累乘当前已解锁。
- `tech.get_application_unlocks` → `{nodes: Array[StringName]}`。
- `tech.get_context_tiers` (v7 PR-G) → `{tiers: Array[{node_id, max_tokens, train_penalty, agent_bonus, serving_penalty}]}`; 按 `max_tokens` 升序, 仅含已解锁节点 (含 baseline `ctx_4k`)。PretrainDialog D 轴 + evaluate agent bonus + serving 吞吐惩罚兜底查询都读它。
- `tech.list_multimodal_methods` (v7 PR-G) → `{methods: Array[StringName]}`; 已解锁的 `multimodal_method_unlock` ∪ `&"cross_train"` (cross_train 始终默认解锁)。

### 信号

`tech_research_started(tree, node_id, task_id)`, `tech_unlocked(tree, node_id)`, `tech_research_cancelled(tree, node_id)`。

### 公开 API (跨系统调用)

- `TechTreeSystem.is_unlocked(tree, node_id) -> bool`
- `TechTreeSystem.get_node_template(node_id) -> TechNode` (供 TaskSystem 校验 min_resources 等)

## 3bis. 科技树面板 UI (树状 DAG 布局)

`scenes/ui/views/tech_view/` — 科技 tab 视图。**抛弃卡片墙**, 改为**树状 DAG 图**: 直观呈现节点的前置依赖链。

- 每棵树一段: `section_header` 标题条 + 一块 **TreeCanvas** 画布。
- **分列**: 节点按"同树前置链最长深度"分到不同的列 (column), 根节点 (无前置) 在最左列, 越靠右越深。同一列内的节点纵向堆叠。
- **连线**: 画布 `_draw` 在节点之间画前置连线 (正交三段折线); 前置已解锁的连线高亮 (绿), 否则浅灰。
- **树级状态摘要**: 每棵树标题下方显示一行紧凑状态胶囊: 已解锁 / 可研究 / 研究中 / 锁定。玩家不需要逐个扫节点就能知道这棵树现在有没有可点的下一步。
- **四态着色** (每个节点一个小面板, 取代旧 `card`):
  - `unlocked` — 绿底, 标题前缀 `✓`, 顶部状态胶囊"已解锁", 无按钮。
  - `researching` — 蓝灰底, 顶部状态胶囊"研究中", 底部显示"研究中…", 无按钮。
  - `available` — 白底 + 炭黑边, 顶部状态胶囊"可研究", 主按钮使用 primary 变体, 文案"研究 (N 周)" → 发 `research_requested(tree, node_id)`。
  - `locked` — 灰底 + 文字灰显, 标题前缀 `锁定 ·`, 顶部状态胶囊"锁定"; 底部显示缺失前置 (例如"需要: Ant v2 / GQA"), 无按钮。锁定状态不用 emoji, 避免字体缺字。
- `_build_tech_view_data` (main.gd) 现在枚举**每棵树的全部节点** (经 `TechTreeSystem.nodes_in_tree`), 附带 `prerequisites` 与上述四态之一, 交由 TreeCanvas 布局连线。

## 4. ResearchDialog (启动研究对话框)

`scenes/ui/research_dialog/research_dialog.gd`, 参照 `pretrain_dialog` 模式 (ConfirmationDialog 子类), 由 `_render_tech_tab` 在玩家点"研究"按钮时构造。入参 `setup(tree, node_id)`。

**控件**:
- `_lead_dropdown`: 第 1 项 "(无 Lead, 1.0×)"。**只列对研究有意义的 idle lead** — 即带 `research_speed` bonus 的 specialty (当前仅 `chief_scientist`) 外加创始人 (player_scientist 万能但无加成)。其他方向 (ml_research_lead / eval / 工程 / 数据 / 营销) 对 `tech_research` 无加速, 不列出, 避免玩家选错人。判定走 `HiringSystem.lead_bonus_coef(lead, &"research_speed") > 0`, 数据驱动。
- `_ml_eng_spin`: 范围 `[min_researchers, idle_ml_eng]`。
- `_infra_eng_spin`: 范围 `[min_engineers, idle_infra_eng]`。
- `_dc_dropdown`: 仅列 `status == idle && gpu_count >= min_gpu_count` 的 DC。研究期间 dc.status 切到 `training`, 不可 serving。
- `_duration_label`: 任意控件变化 → `task.preview` 拿权威 total_months, 显示 "原 X 周 → 实际 Y 周 (lead ×Z)"。

**确认调用**: 收齐 lead / staff / datacenter 选择后发 `tech.start_research` 命令 (参数见 §2)。

**启动按钮禁用条件 (警告区显示原因)**:
- 可用 ml_eng / infra_eng 不足 `min_*`
- 无 idle DC 满足 `gpu_count >= min_gpu_count`
- 前置链断 (理论上 list_available 已过滤, 兜底)

## 5. 核心公式与触发时机

### 5.1 研究时长

研究周数由 `node.research_months` (单位周) 经 lead 加速折算 — 只有带 `research_speed` bonus 的 lead (当前 chief_scientist) 能缩短时长, 无 lead 时按原值。权威值由 `task.preview` 返回, 公式与系数以 TaskSystem 代码实现为准。

### 5.2 engineering 乘子下沉到 serving t/s (v4 PR-B)

`tech_unlocked(tree=engineering)` 触发 InfraSystem 遍历所有 serving dc 重算 `serving_tokens_per_sec`: 把已解锁 engineering 节点的 `throughput_multiplier` 累乘到 dc 推理算力、`flops_per_token_reduction` 累乘到单 token FLOPs, 二者相除得出每秒可服务 token 数。MonetizationSystem 直接读 `dc.serving_tokens_per_sec`, **不再二次乘**。

### 5.3 算力 = 锁 DC

研究期间 `infra.assign_to_task` 锁 dc, status=training, 不可 serving。**不另收现金算力费** — 周费已隐式承担。`research_cost` 字段保留为 0。

## 6. 六棵树设计意图

> DAG 结构 (节点 `prerequisites`) 与全部数值见 `resources/data/tech/<tree>/*.tres`。本节只记录每棵树的关键设计意图, 不镜像节点表与连线。

### arch (主干线性 + 多模态/长上下文支线 + 陷阱节点 + 未来扩展)

**设计意图**:
- 主干从 baseline `ant_v1` 出发, dense 线 (`ant_v2` → `bee_v1`) 接 MoE 子链 (`octopus_v1/v2/sparse/super_sparse`)。MoE 子链强制线性 (玩家想拿 sparse 必须先把 dense → MoE 全跑完); 多模态 / 长上下文是同树支线, 不阻塞 MoE 链。`active_param_ratio` 在 MoE 节点上让训练 / 推理 FLOPs 按比例缩减, 但 Chinchilla 仍按总参数计算所需 token (MoE 数据饥渴是合理代价)。
- **陷阱节点 / 编码器有限 scale** (v7 PR-G; v10/v11 扩链): `bert_encoder` 与 `t5_enc_dec` 是 2018-2019 现实主流方向。玩家在早期解锁数值并不差 (capability_coef ≥ 1.0), 且 BERT 系列现在可以继续 scale 一段: `bert_encoder → roberta_encoder → electra_encoder → deberta_encoder → bert_scale_encoder → bert_giant_encoder`, cap 从 30 逐步抬到 64。这个分支不是“完全不能 scale”, 而是**有限 scale**: 投入更多研究与更大模型确实会涨分, 但 `capability_cap` 始终钳住 `base`, dense decoder / MoE 主线无 cap, 中后期仍能靠规模超过它。T5/UL2 是另一条 encoder-decoder 陷阱短线 (cap 45→58)。设计目的: 让 BERT 路线看起来像一条有纵深、可继续投入的正路, 但玩家如果长期押注会撞到范式天花板。effects_summary 直接写出 "评分上限 X", 愿意读数值的玩家能识破。
- **multimodal_method 支链** (v7 PR-G): `cross_train` 是 2021 (~CLIP) 入门; 其下分出 `dit_v1` / `pixel_ar` 两条 2023-2024 路线; **`native_multimodal` 是一个菱形依赖节点 — 同时以 `dit_v1` 与 `pixel_ar` 为前置** (两条路线都研究完才能解锁), 是 2025 最强路线 (multimodal 维度 capability_coef 最高, 但 train_coef 最低 / penalty 最大)。每个节点 `multimodal_method_unlock` 字段对应 PretrainDialog E 轴选项。
- **未来节点** (v7 PR-G, 2030+): `world_model_v1` / `embodied_v1` 接在 `native_multimodal` 之后, 给玩家长尾目标; 当前只通过 arch 评估与训练系数生效, 不再挂未消费的 application 产品节点。

### attention (B 轴)

**设计意图**: 从 baseline `mha_baseline` 出发, GQA → MQA/MLA 分化路径 (MQA 偏推理极致, MLA 偏内存效率); hybrid_attention 是终极合并。`inference_coef` 是 fpt 分母 (≥ 1.0) — pretrain 完成时 `model.flops_per_token /= inference_coef`, 模型架构定死后推理 t/s 立即上升。`infinite_attn` 是 2030+ 投机节点, 仅在 ctx_10m 同时解锁时给加成。

### loss (C 轴)

**设计意图**: 单链结构, baseline `ce_baseline` → `zloss` → `mtp`。轻量改动, cost / time 也低; `zloss` 与 `mtp` 在现实里是 DeepSeek-V3 同期出现, 游戏里独立解锁不阻塞。**v7 PR-G 加 reasoning RL 子链** (`dpo` / `rlvr` / `o1_rl`): 主要影响 `capability_coef` 在 reasoning 维度的乘数, 对应 2023-2025 现实推理范式。

### engineering

**设计意图**: 4 条独立链 (低精度训练/推理、并行训练、推理 batch/KV、整数量化), 玩家按需点。量化节点 (FP8 / FP4 / INT8 / INT4) 用 `flops_per_token_reduction` (≤ 1, 多节点累乘), 全部解锁可累积大幅推理收益。**量化不降质** (v4 PR-B 决策: 删 quality_penalty, 现实里 FP8/INT8 PTQ+QAT 已无可察觉降质; 玩家直觉上不该回避收益节点)。

### application

**设计意图**: application 树只保留被系统直接消费的门控节点。`tool_use` 是 agent 类产品的解锁起点, 同时让 evaluate 的 agent 轴获得工具使用倍率; `fox_code_specialist` 是 coding_agent 产品门。旧的 RAG / planning / memory / image_gen / world_simulation / embodied_agent 节点只声明 `product_quality_bonus` 或 `capability_threshold_lower`, 但没有 ProductSystem / UserSystem 消费路径, 已删除以避免假收益。

### context (v7 PR-G 新, D 轴 + agent bonus)

**设计意图**: 单链结构, baseline `ctx_4k` → `ctx_32k` → `ctx_200k` → `ctx_1m` → `ctx_10m`。
- 拆出独立的 context 树是因为 PretrainDialog D 轴 (4k/32k/200k/1M) 原本硬编码 4 档对所有玩家始终可见, 没有研究决策空间。改成科技节点后, 32k 以上必须研究才能在 PretrainDialog 选中, 给中后期玩家提供持续研究目标。
- `agent_bonus` 在 evaluate 时**直接加到 raw 之后** (`agent_score = max(0, raw × agent_ratio × tool_use_bonus) + context_agent_bonus × passes_gate`), 不是乘子, 这样小模型即使过 gate 也只能拿少量 agent 分, 大模型受益更明显。bonus 数值与 ctx 档位严格挂钩, 鼓励"训练支持 1M ctx 的大模型, agent 分上一个台阶"的玩法。
- baseline `ctx_4k` 是 default unlock (与 `mha_baseline` / `ce_baseline` 同等地位), 由 `_ensure_baseline_unlocks` 兜底。
- D 轴 PretrainDialog 不再硬编码 4 档, 而是从 `tech.get_context_tiers` 拉已解锁档位; 训练 train_penalty 用对应节点的 `effects.train_penalty` (与旧 `_CONTEXT_LENGTH_PENALTY` 表同值, 兼容期内代码兜底)。
