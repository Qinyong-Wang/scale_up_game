# 基础设施系统设计 (InfraSystem)

> **目的**: 管理「算力」物理资产。**机房 (Facility) 与 GPU 是两种独立资产**: 玩家先建/租机房 (容量上限 + 周度场地费), 再单独购买 GPU 装进去 (折旧资产, 可二手卖出)。同一机房只能装一种 GPU。提供 idle / training / serving 三态。
> **读者**: 数据中心 UI、TaskSystem (锁定训练)、MonetizationSystem (推理算力池)、EconomySystem (GPU 二手回款)。

---

## 0. 关键概念

- **机房 (Facility)**: 物理建筑, 决定能装多少卡 + 用什么电。19 个档位 (`facility_solo` 1 卡 → `facility_planet` 100M 卡 太空算力微型星球)。玩家从单卡桌面起步, 大档位需要 **cash 阈值**解锁 (v7 PR-F: 取代旧 `unlock_fame_required`)。最高四档为「太空算力」(`facility_space_s/m/l` = 6M/12M/20M 卡 太空数据中心 + `facility_planet` = 100M 卡微型星球): `land_build_cost` 按 metropolis 每卡基线翻倍 ($8K → $16K/卡), 周地费/周租按基线线性外推; 微型星球只是把同一太空基线线性顶到 100M 卡 (建造 $1.6T / 周地费 $1.5B / 周租 $7.5B / 解锁现金门槛 $175B), 不改每卡经济, 是纯粹的终局体量目标。
- **GPU**: 折旧资产, 单独买入装机房, 可单独卖出 (按购买后年份做 10%/年**复利**折旧)。同一机房只允许一种 GPU 型号 (避免微观调度地狱)。加卡 / 卖卡瞬时生效。
- **GPU 不周度记账折旧**: 购入时一次性扣 cash, 之后只扣周度维护 + 电费; 折旧只在二手卖出时一次结算 (`refund = purchase_price × 0.9^years`)。
- **三态机**: `idle` / `training` (TaskSystem 锁) / `serving` (绑模型推理) 互斥。GPU 数 0 也能 idle。
- **`serving_tokens_per_sec` 是缓存**: 部署模型时算并缓存到 dc 上; `MonetizationSystem` 直接读, 不再二次缩放。

## 1. 数据切片

owner = `InfraSystem`。

| 字段 | 类型 | 语义 |
|---|---|---|
| `datacenters` | `Array[Datacenter]` | 已上线 dc |
| `construction_queue` | `Array[DatacenterConstruction]` | 自建中 |

### 1.1 Datacenter 字段 (非显而易见的语义)

完整字段以 `datacenter.gd` 的 `@export` 为准, 本节只记不直观的点:

- `ownership` 是三态 `&"rented"` / `&"owned"` / `&"cloud"` (见 §1.4); cloud DC 的 `facility_spec_id` / `power_supply` 为空, `max_gpu_count` 取创建时的 `count`。
- `gpu_id` + `gpu_count` 受单品牌约束 (同一 dc 只能一种 GPU); `gpu_purchase_history` 是 `Array[GPUBatch]`, 二手卖出按 FIFO 走。
- `serving_tokens_per_sec` 是**缓存值** — 部署时算一次, idle 时为 0; MonetizationSystem 直接读, 不二次缩放 (公式见 §4.1)。
- `serving_target_kind` 取 `&"owned_model"` 为主。旧存档可能仍有 `&"open_source_model"` (OS NPC release 直连部署, v9 PR-I 旧语义), 读档时按 §3.1 迁移 / 修复。
- ~~`fame_modifier`~~ 已于 v7 PR-F 删除 (fame 整体废弃)。

### 1.2 GPUBatch 字段 (内嵌在 Datacenter.gpu_purchase_history)

`count`, `unit_price`, `bought_at_turn`。

### 1.3 DatacenterConstruction 字段 (自建工程)

`id`, `facility_spec_id`, `power_supply`, `weeks_remaining`, `total_weeks`, `gpu_id` (可选, 完工自动装满)。完工后 id 转为 Datacenter.id。

`construction_queue` 是完整持久化状态: `GameState.to_dict()` 必须写出每个 `DatacenterConstruction`, `Save.read()` 后必须恢复队列与剩余周数。玩家在建设开始后的同一周直接关闭应用时, 主场景退出兜底 autosave 也必须保留该队列, 否则重新打开会丢失“自建中”的数据中心。

**基建 UI 展示** (2026-05): `construction_queue` 不再用纯文本列表。基建页应把每个自建工程渲染成紧凑的在建卡片: 左侧使用对应 `FacilitySpec.icon_path` 的建筑缩略图 (缺图回退 datacenter glyph), 标题为机房档位, 副标题为工程 id, 状态徽章为「自建中」; 字段区显示供电、预装 GPU (如有)、剩余/总周数, 并在字段下方给出进度条。进度条按 `done = total_weeks - weeks_remaining` / `total_weeks` 计算, `total_weeks <= 0` 时视为 0%, 避免坏档造成除零。旧字段名 `months_remaining` / `total_months` 只作为兼容输入, UI 文案统一显示「周」。

### 1.4 三种 ownership

| ownership | upfront | 周费组成 | 关停 |
|---|---|---|---|
| `rented` | 0 | `facility.rent_weekly_cost` + GPU maint + 电费 | terminate 自动按二手价卖光 GPU |
| `owned` | `facility.land_build_cost × (1 + power.build_cost_modifier)` + 可选 GPU 建设期预付 + `power.install_cost_per_card × facility.max_gpu_count` (绿色能源安装储能费) | `facility.land_weekly_cost` + GPU maint + 电费 | 同上 |
| `cloud` | 0 | `gpu.rent_weekly_cost × gpu_count` (全包: 电力 + 维护 + 设施) | 不退款; 不允许 buy_gpus / sell_gpus |

**云 DC vs 自有 DC 成本对比** (2026-05 重定价): `gpu.rent_weekly_cost` 统一为 `purchase_price / 40`, 即**云租 40 周 = 一张卡的购置价** (此前为 20 周, 砍半)。短期算力突发选 cloud, 长期持有选自建。

**闲置 GPU 出租 (算力平台)**: idle 且 `ownership != cloud` (owned / rented, GPU 都是自购) 的 dc, 玩家可 **opt-in** (卡片「出租到算力平台」按钮 / `infra.set_dc_rent_out`) 把已装 GPU 出租到第三方算力平台, 每周产生租金; 出租价 = `rent_weekly_cost × 0.5` (砍半云租的 50%), 平台抽 `22%` 管理费。默认关。详见 §4.4。

**玩家可选项 (新建数据中心对话框)**: 只提供 **云租用 (`cloud`)** 与 **自建 (`owned`)** 两种。`rented` (租机房 + 自购 GPU) 因与「云租用」体验难以区分 (见 ux_playtest_report §D-8) 已从对话框移除; `infra.rent_facility` 命令与 `rented` ownership 仅作内部 / 测试脚手架保留, 玩家无法在游戏内触发。

### 1.5 静态模板 (.tres)

三类只读模板, 字段以各 Resource 类脚本为准, 数值见 `.tres`:

- `FacilitySpec` — `resources/data/infra/facilities/*.tres`, 19 档机房规模 (solo 1 → planet 100M; 含 `rack_16` / `rack_32` 两档过渡机架, 方便初期玩家; 最高四档 `space_s/m/l` 6M/12M/20M 卡为太空数据中心 + `planet` 100M 卡微型星球, 部署 (build) 成本按太空基线翻倍, 周费线性)。大档位用 `unlock_cash_required` 做现金门槛 (v7 PR-F 取代旧 `unlock_fame_required`)。每档带可选 `icon_path` 建筑图标 (2026-05), 显示在 infra 卡片头像 + 新建数据中心档位预览; 缺图回退 seed 配色 + datacenter glyph (`▣`)。**云租用 DC** 无 facility (`facility_spec_id` 为空), 用专属图标 `assets/sprites/ui/infra/facility-cloud.png` (云 + 机架, 同批 isometric 风格), 由 `main._facility_icon_path("")` 提供。生成与接入流程见 [图片素材生成流程.md](图片素材生成流程.md)。
- `GPUSpec` — `resources/data/infra/gpus/*.tres`, 11 卡 3 品牌。推理算力 `per_card_inference_tflops` 约为训练算力的 30% (含 KV cache + 中等批量的真实利用率折损)。`release_turn` 按 2017-06-12 锚到真实 GPU 上市日期。
- **`power_factor` 功耗系数** (2026-05): GPU 的相对功耗乘子, `cypress_t1` (≈A100, 312 TFLOPs) 锚为 `1.0`。每卡周电费 = `power.weekly_cost_per_card × power_factor` (取整)。系数按 **功耗 ∝ 算力^0.45** 设定 (现实里制程进步 → perf/watt 每代提升 → 功耗增长远慢于算力, 如 A100→H100 算力 ×3.2 但功耗仅 ×1.75)。范围约 0.65 (bamboo_t1) → 2.55 (cypress_t3≈B200)。`maintenance_per_week` 不受系数影响。
- **`ecosystem_score` 训练惩罚** (v11): GPU 的 `ecosystem_score` 是训练吞吐乘子 (软件 / 框架成熟度), 烘焙进 `dc.train_tflops` (`= per_card_tflops × 卡数 × cluster_efficiency × ecosystem_score × (1 + facility.train_speed_bonus)`); **只惩罚训练, 不影响推理** (`inference_tflops` 不乘)。cypress 1.0 / maple 0.8 / bamboo 0.6 — 即 bamboo 训练慢 40%, 但推理满速且卡更便宜, 形成「便宜铁、训练慢、适合推理」的差异化生态。`cluster_efficiency` 字段本身不含 ecosystem。
- **太空数据中心训练加速** (`FacilitySpec.train_speed_bonus`, 2026-05): 太空档真空辐射散热 → 无热降频, 训练吞吐乘子加成。**只加训练 (`train_tflops`), 不加推理**。按档递增: `space_s` +10% / `space_m` +15% / `space_l` +20% / `planet` +20%; 地面档全部 0。值在 `.tres` 的 `train_speed_bonus` 字段。烘焙进 `_recompute_compute` 的 `train_tflops` (上面公式的 `(1 + train_speed_bonus)` 因子), 因此预训练时长 (`task.preview` 读 `train_tflops`) 自动随之缩短。UI 经 `InfraSystem.facility_train_bonus(facility_spec_id)` 单独取该加成, 显示在 infra 卡片 (「太空加成 +N%」字段) 与预训练 DC 下拉 (选项后缀「太空 +N%」)。
- `PowerSupplySpec` — `resources/data/infra/power/*.tres`, **2 种供电方式** (v11): `grid` 常规供电 (基准 $24/卡/周, 无前期投入); `green` 绿色能源 (基准 $4/卡/周, 一次性 `install_cost_per_card` 安装 + 储能费 $5200/卡, 建造时按机房满配卡数收取)。`weekly_cost_per_card` 是 `power_factor = 1.0` 的基准卡电费, 实际每卡电费再乘 GPU 的 `power_factor`。green 安装费**不**随 power_factor scale (一次性、按满配卡数), 故功耗越大的卡上绿电回本越快 (回本周数 = 5200 / (20×factor), cypress_t1 满配约 260 周/5 年, 高功耗卡更短)。`carbon_kg_per_card_week` / `build_cost_modifier` / `efficiency_modifier` 字段保留但 grid/green 均取中性值。

## 2. 命令与信号

详见 `命令总线表.md §InfraSystem` 与 `事件总线信号表.md §InfraSystem`。要点:

### 玩法命令

- `infra.rent_facility {facility_spec_id, power_supply_id}`: 零 upfront, idle 上线; 按 **cash** 校验大档位 (v7 PR-F: `unlock_cash_required`, 错误码 `facility_unlock_required` 保留兼容)。`NewDatacenterDialog` 预览也只展示现金门槛, 不读取旧 `GameState.fame` / `unlock_fame_required`。
- `infra.build_facility {facility_spec_id, power_supply_id, gpu_id?}`: 立即扣建设费 + 可选 GPU 预付费; 进 construction_queue, 完工后转 Datacenter, 自动装满 GPU (若 build 时声明)。同样按 `unlock_cash_required` 做 UI 预警与系统校验。
- `infra.create_cloud_dc {gpu_id, count}`: 零 upfront, idle 上线, ownership = cloud。`gpu_not_released` 错误码: GPU `release_turn > GameState.turn` 拒。
- `infra.terminate_dc {dc_id}`: 仅 idle 可执行; 自动按二手价卖光所有 GPU (cloud DC 例外, 不退款)。
- `infra.buy_gpus {dc_id, gpu_id, count}`: 任意 status 都允许 (弹性扩容); 错误码含 `unknown_gpu` / `gpu_not_released` / `invalid_count` / `mixed_brand` / `capacity_exceeded` / `cloud_dc_no_purchase`; 立即扣 cash, 追加 GPUBatch, 重算派生算力 + serving t/s。
- `infra.sell_gpus {dc_id, count}`: dc 必须 idle; FIFO 卖最早批次, 折旧公式 `unit_price × 0.9^years`, 走 `economy.award`。
- `infra.deploy_model {dc_id, model_id}`: idle → serving, 校验 model `published`, GPU 数 > 0。
- `infra.deploy_open_source_model {dc_id, release_id}`: 目标是某个 NPC 的开源 pretrain release。部署前先调用 `research.ensure_open_source_release_published` 把 release 物化 / 复用为 `provenance = downloaded_os` 的 published 模型, 自动创建 API 产品, 再把 DC 绑定到该 model。返回 `{ok, tokens_per_sec, model_id}`。release 解析错误码: `unknown_release` / `not_open_source` / `not_pretrain` / `not_released_yet`。
- `infra.undeploy_model {dc_id}`: serving → idle, 清 `serving_tokens_per_sec`。
- `infra.set_dc_rent_out {dc_id, enabled}`: 开/关「出租闲置 GPU 到算力平台」(opt-in, 见 §4.4)。错误码 `unknown_dc` / `cloud_dc_no_rental` (cloud 卡非自有不可出租)。
- `infra.preview_deploy_capacity {dc_id, model_id? | release_id?}`: 纯读, 不改 dc 状态; 用与 deploy 相同的 engineering 乘子路径算 preview t/s, 避免 UI 与实际不符。
- `infra.assign_to_task {dc_id, task_id}` / `infra.release_from_task {dc_id, task_id}`: TaskSystem 用; assign 时 GPU 数 > 0 才允许。

### 调试命令

- `infra.debug_instant_owned_dc {facility_spec_id, gpu_id}`: 跳过建设流程瞬时创建一个满配 owned dc, 不扣 cash。仅用于开发调试 / 集成测试 setup, 不暴露给玩家 UI。

### 信号

`datacenter_added` / `datacenter_removed` / `datacenter_status_changed` / `gpus_bought` / `gpus_sold` / `dc_compute_recomputed` / `model_deployed` / `open_source_model_deployed` / `model_undeployed` / `construction_progress` / `construction_completed`。

### 3.1 公共开源部署的物化语义 (2026-05 bugfix)

玩家在基建 tab 选择「公共开源模型」部署时, 不再创建一条只存在于 Datacenter 上的匿名 serving target。原因: 匿名 target 没有 `Model.per_token_price`, 没有 `model_published` 信号, ProductSystem 无法自动建 API 产品, 产品页也看不到该 API。

新流程:

1. `infra.deploy_open_source_model` 先校验 release 是已发布的开源 pretrain release。
2. 调 `research.ensure_open_source_release_published {release_id}`:
   - 若该 release 已物化过 (`Model.source_release_id == release_id`), 复用原 model。
   - 若没有, 按 `research.download_open_source` 同一公式创建 `provenance = downloaded_os` 模型。
   - 若 model 尚未 published, 自动以 `is_open_source = true` 发布, 默认价取开源 `guidance_price_per_token`。
3. ProductSystem 通过 `model_published` 自动创建 type=`api` 产品。
4. DC 实际写 `serving_target_kind = &"owned_model"`, `serving_target_id = model_id`, `deployed_model_id = model_id`, 因此产品页/营收页/定价弹窗都复用 published model 管道。

仍保留 `open_source_model_deployed(dc_id, release_id)` 信号用于 UI 刷新与兼容审计; 可额外通过返回值 `model_id` 找到实际绑定模型。

读旧档时, 若发现 DC 仍在 serving `serving_target_kind = &"open_source_model"` 的 release id, `InfraSystem._on_save_loaded` 必须执行同一物化 / 发布流程并把 DC 改绑到 `owned_model`。如果 release 已不存在或已不满足开源 pretrain 条件, 则按旧兼容逻辑退回 idle, 避免 0 t/s 的幽灵 serving 状态。

## 4. 核心公式

> 公式实现以 `infra_system.gd` (`_recompute_compute` / `_refresh_serving_capacity` / `_sell_gpus_internal`) 为准, 本节只记设计意图。

### 4.1 派生算力

派生算力 = 单卡算力 × 卡数 × `cluster_efficiency`; `cluster_efficiency` 由 GPU 原生集群效率、供电效率、大集群衰减 (卡越多边际效率越低) 与 chief_engineer 加成共同决定。serving t/s 在部署 / 买卖卡 / engineering 解锁时重算: 把 engineering 乘子 (throughput 提升、flops/token 降低) 下沉到这里, MonetizationSystem 直接读缓存不再二次乘。idle / 无目标时 `serving_tokens_per_sec = 0`; `preview_deploy_capacity` 走同一路径, 与实际 deploy 数值一致。

**长上下文吞吐惩罚** (2026-05): serving t/s 还要除以**部署模型的 context 档惩罚** `serving_penalty` —— 长上下文 serving 因 KV cache 显存带宽瓶颈急剧掉吞吐。即 `serving_tokens_per_sec = effective_inf × 1e12 / (model.flops_per_token × flops_per_token_reduction × ctx_serving_penalty)`。`ctx_serving_penalty` 按 owned 模型 `context_length_tokens` 查 context 子树该档 `serving_penalty` (4k=1.0 无惩罚 / 32k 1.5 / 200k 3.0 / 1M 6.0 / 10M 12.0; 档值与 `平衡参数.md` 一致, 兼容期内 `infra_system` 有同值常量兜底)。OS / NPC release 部署无 ctx 字段, 视作 1.0。

### 4.2 周度费用

upkeep 阶段拆三桶分别扣 cash: 场地费 (自有 land / 租用 rent)、GPU 运行费 (维护 + 电费, 非 cloud)、云 GPU 全包费 (cloud)。**电费按 GPU 功耗系数 scale**: 每卡电费 = `int(round(power.weekly_cost_per_card × gpu.power_factor))`, 由 `InfraSystem.electricity_per_card(gpu, pwr)` 集中计算 (upkeep 扣费、缓存 `facility_weekly_cost`、`ResearchSystem` 定价基线三处共用)。

### 4.3 二手卖卡

`sell_gpus` 按购买批次 FIFO 折旧 (`unit_price × 0.9^years`, years 自购入回合算起); 卖空批次出队, 卡数归零时清 `gpu_id` 与历史, 下次买可换品牌。

### 4.4 出租闲置算力到算力平台 (2026-05)

玩家 opt-in (`Datacenter.rent_out_enabled`, 默认 `false`, 经 `infra.set_dc_rent_out {dc_id, enabled}` 切换; cloud dc 拒, 错误码 `cloud_dc_no_rental`)。开启后, 该 dc 在 idle 且 `ownership != cloud` (owned / rented) 且 `gpu_count > 0` 时, 每周 upkeep (在 `_charge_weekly_costs` 之后) 把已装 GPU 出租到第三方算力平台 (cloud 卡非自有, 不出租):

- **毛租金** = `int(gpu.rent_weekly_cost × RENT_OUT_FRACTION) × gpu_count`, `RENT_OUT_FRACTION = 0.5` → 走 `economy.award` reason `gpu_rental_income` (账本类目「GPU 出租」, 计入应税收入)。
- **平台管理费** = `int(毛租金 × PLATFORM_FEE_RATE)`, `PLATFORM_FEE_RATE = 0.22` → 走 `economy.spend` reason `gpu_rental_platform_fee` (账本类目「出租平台管理费」, 可抵税)。
- 净到手 = 毛租金 × 78%。training / serving 的 dc 不出租 (算力已被自用); cloud dc 不出租 (卡非自有)。
- **训练资源占用**: `rent_out_enabled == true` 表示玩家仍把 idle GPU 挂在第三方平台, 不能同时作为自用训练资源。`PretrainDialog` / `PosttrainDialog` 的数据中心下拉只列 `status == idle && rent_out_enabled == false` 的 dc; 底层 `infra.assign_to_task` 同样拒绝出租中的 dc, 返回 `dc_rented_out`。玩家要先在基建卡片关闭出租, 再启动预训练 / 后训练。

开关持久化在 dc 上 (存档随之), 无新 EventBus 信号 (UI 点按钮后 main 直接 `_request_refresh`)。出租是已沉没的购卡/运维成本之上的边际增收, 设计意图: 让超配 GPU 与回合间空窗期的算力变现, 综合年化 ROI 约 25–41% (随卡价/功耗不同), 详见 `平衡参数.md`。

对外出租中的 idle dc 不能被「宇宙模拟工程」永久捐出: `SimulationSystem.eligible_datacenters` 与 `simulation.start_stage` 都必须要求 `rent_out_enabled == false`。玩家需要先在基建卡片点「停止出租」, 再到慈善 tab 捐建, 避免同一座机房既履行算力平台出租承诺又被永久移除。
