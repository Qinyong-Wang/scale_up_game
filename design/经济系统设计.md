# 经济系统设计 (EconomySystem)

> **目的**: 管理「钱」这一最基础资源, 提供消费/收入接口, 处理周期性扣费 (利息 / 工资 / 租金), 以及贷款/融资/破产判定。同时维护**周度财务报表**, 玩家可以直接看上周收支明细 + 12 周历史。
> **读者**: 经济面板 UI、所有需要花钱或收钱的 system 实现者。

经济系统是最**被动**的核心系统: 不主动驱动玩法, 只为其他系统提供「花钱」「收钱」两扇门。所有 cash 变动**必须**走它的命令, 是审计与防作弊的唯一手段, **同时也是财务报表的唯一数据源**。

---

## 1. 数据切片

owner = `EconomySystem`, 写权限独占。字段:

| 字段 | 类型 | 语义 |
|---|---|---|
| `cash` | int | 现金, **允许为负** (持续过久或过深会触发破产, 见 §4.2) |
| `debt` | int | 未偿还本金总额 = `Σ loans[*].principal_remaining` |
| `equity` | `{founder, investors}` | 股权结构, 用于融资稀释; `founder + investors == 1.0` |
| `loans` | `Array[Loan]` | 在借合同列表 |
| `bankruptcy_streak` | int | 连续 cash < 0 的**周**数 |
| `weekly_burn_rate` | int | 最近 N 周支出滑动平均 (`N = burn_window_weeks`) |
| `quarterly_revenue` | int | 最近 N 周营收滚动累计 (`N = revenue_window_weeks`; 不足时按已知均值外推) |
| `funding_rounds_accepted` | Dict | 已接受过的轮次, 同一轮不重接 (无顺序约束: 满足条件即可) |
| `weekly_ledger` | Dictionary | 当周 (in-progress) 收支账本: `{income: {category → amt}, expense: {category → amt}, gross_in, gross_out}`; 顶栏周净流可读它 |
| `ledger_history` | Array | 最近 `LEDGER_HISTORY_WEEKS` 周的已完成账本快照 (newest first), 每条额外带 `turn` + `ending_cash`; 经济页“上周收支明细”读 `ledger_history[0]` |

**`cash` 允许为负**: 任何 spend 都不会被「现金不足」拒绝, 简化所有调用方的错误处理。压力来自破产判定 (§4.2), 不来自单次 spend 失败。

`Loan` 字段: `id`, `principal_initial`, `principal_remaining`, `weekly_interest_rate` (签约时锁定), `weeks_remaining`, `taken_at_turn`。

> v9 (2026-05): 融资模型重构: 8 轮 (pre_seed→seed→a→b→c→d→e→f), **玩家自发触发** (不再走事件)。删除 `funding_rounds_offered` 切片、`economy.start_funding` (自定义金额) 命令、`funding_round_*.tres` 事件卡。
> v9.1 (2026-05): 删除轮次顺序锁: 只要 conditions 满足即可融该轮 (允许跳过前置轮, 例如直接接 a 轮无需先接 pre_seed/seed)。同轮防重接 (`already_accepted`) 保留。

## 2. 命令与信号

详见 `命令总线表.md §EconomySystem` 与 `事件总线信号表.md §EconomySystem`。要点:

- `economy.spend` / `economy.award` **永远成功** (cash 可变负); 所有花钱/收钱都走它, 同时**自动写入 weekly_ledger**。
- `economy.take_loan` 受信用上限约束, 不足返 `&"credit_limit_exceeded"`。
- `economy.repay_loan` 部分/全额还贷。
- `economy.preview_credit` 返回 `{max_loan, rate, rating}`, 供贷款对话框使用。
- `economy.preview_funding_rounds` 返回 8 轮状态数组, 每条 `{round, display_name, amount_min, amount_max, dilution_min, dilution_max, status, unlock_summary}`, status ∈ `{accepted, available, locked}`。供融资 UI 使用, 不修改任何状态。
- `economy.start_funding_round` 玩家主动接受某轮 (无顺序约束, conditions 满足即可)。金额/稀释当场 roll, 也可显式传入 (供测试)。同轮重复接受返 `&"already_accepted"`; conditions 不满足返 `&"conditions_not_met"`。
- 任何 cash 变动 emit `cash_changed(delta, reason)` + `resources_changed`; 取贷/还贷/月度本金扣还都 emit `debt_changed`; 融资完成 emit `equity_changed` + `funding_completed`; **每个 resolve phase 结束时滚动账本 emit `ledger_rolled(turn, snapshot)`**。

## 3. 对外依赖

- **读切片**: 累积/周度 revenue (本系统 `_revenue_history` 内部环形) 用于估值与信用评级。`market.get_rank` 调用 (信用 S/A 高档与 c-f 轮融资条件需要排名)。
- **调命令**: 无 (纯被动)。
- **订阅信号**:
  - `phase_started(upkeep)` — 扣 loan 利息+本金, 滚动 burn/revenue 缓冲。**其他系统**的工资/租金/营销月费在同相位**各自**调 `economy.spend`, 本系统不聚合 (新增经常性支出不需要改本系统代码)。
  - `phase_started(resolve)` — 破产判定 + **账本滚动**。

## 4. 内部算法

### 4.1 贷款

- 取贷: `_max_loan = min(β × quarterly_revenue + γ × max(cash − debt, 0) − debt,  MAX_LOAN_ABSOLUTE_CAP − debt)`, 两路再各自 `max(., 0)`。这里用**净现金** `max(cash − debt, 0)` 而非裸 `cash`: 借来的现金被等额新增的 debt 抵消, 所以**取贷本身不会抬高信用上限** —— 每借 1 元, 剩余可贷额正好减 1 元。旧公式直接用 `cash` 会让每次借款把额度推高约 2×, 形成「借一次翻一倍」的无限借钱漏洞。
- **绝对上限 `MAX_LOAN_ABSOLUTE_CAP = $20B`** (`max_loan_absolute_cap` 字段, 见 `resources/data/economy/tuning.tres`) 是**银行对该公司的总敞口**, 不是单笔贷款上限: 已有 debt 占掉敞口, 剩余可贷 = cap − debt。后期 $10B/周 营收 + 高净现金时, β/γ 公式会算出 $500B+ 的可贷额, 偏离"单一银行愿意借多少"的现实直觉。
- **期限上限 `MAX_LOAN_TERM_WEEKS = 156`** (`max_loan_term_weeks` 字段, 见 `resources/data/economy/tuning.tres`) 是银行愿意给的最长合同期, 即 3 年期 (游戏 1 年 = 52 周)。`economy.take_loan` 对 `term_weeks > 156` 返回 `&"loan_term_exceeded"`; 0 或负数期限仍按无效贷款请求处理。UI 可提供更短档位, 但最长选项必须包含 156 周。
- 利率: `rate = BASE_INTEREST_RATE × rating_multiplier(credit_rating)`, 签约时锁定到 `loan.weekly_interest_rate`, 利率档位与信用评级见 §4.3。
- 周扣 (upkeep): 每周扣 `interest + principal_due`, 其中 `interest = principal_remaining × weekly_interest_rate`, `principal_due = principal_remaining / weeks_remaining` (等额本金简化)。**利息与本金分两次 emit `cash_changed`** (reason 分别为 `loan_payment_interest` / `loan_payment_principal`) 以便账本分类显示。`weeks_remaining` 归零后从 `loans` 移除并 emit `loan_repaid(fully=true)`。

### 4.2 破产判定 (resolve 相位)

两条独立判定线, 任一触发立刻 `bankruptcy_triggered`:

- **线 A (负太久)**: `cash < 0` 累计 `bankruptcy_streak` 达 `BANKRUPTCY_STREAK_LIMIT`, 触发 `cash_negative_too_long`。`cash >= 0` 时归零。
- **线 B (负太深)**: `cash` 低于一个由 `weekly_burn_rate / quarterly_revenue` 推出的负阈值, 触发 `cash_too_deep`。公司极小时该阈值会退化到一个兜底下限 (`BANKRUPTCY_DEPTH_FLOOR`), 防止任何赤字立刻 game over。

两条线都先 emit `bankruptcy_warning` 再 trigger。

**预警阈值 `BANKRUPTCY_WARN_STREAK`** (默认 8 < `BANKRUPTCY_STREAK_LIMIT` 12): 线 A 的 streak 跨过它时, 玩家应被明确提醒「再不扭转, 游戏将结束」。streak 仍随每周 cash<0 累加, `bankruptcy_warning` 照常每周 emit; 预警阈值只是 UI 用来区分「轻度赤字」与「濒临破产」的分界, 不改变信号契约 (仍是 `(reason, streak, threshold)`)。

**终局 (game over)**: `bankruptcy_triggered` 是**终局信号**——公司资金链断裂、本局结束。UI 层 (`main.gd`) 收到后弹出 Game Over 结算弹窗 (显示存活周数 / 最终资金 / 破产原因), 玩家确认后**删除 autosave** (本局已输, 不可「继续游戏」恢复死局) 并切回起始页。系统层只负责发信号, 不直接切场景, 以便 headless 测试可独立验证两侧。

### 4.3 信用评级 / 利率

同时看 `debt / quarterly_revenue` 比率 **与** `quarterly_revenue 绝对值`, 两者都过阈值才升档 (S/A/B/C/D 五档)。设计意图: revenue 是「经济实力」, 比 fame 更直观、更可控。新业务起步阶段 (revenue 低) 即使排榜靠前也只能拿 C/D 评级。利率 = `BASE_INTEREST_RATE × 档位倍率`, 高档便宜、低档贵, S 与 D 之间约 10:1 区间。C 档是新公司常见信用档, 周利率目标为 **0.25%** (`BASE_INTEREST_RATE 0.20% × 1.25`), 避免早期贷款压力过重; 具体阈值与其他倍率见代码常量。

### 4.4 估值 (融资轮)

估值由 `quarterly_revenue` 与 `cash` 折算的基数, 乘以「总榜排名溢价」(排名越高溢价越大) 与「在售模型数量加成」得出, 并以一个**估值下限**兜底保护新档免于零估值。每轮 .tres 写定 `amount_min/max + dilution_min/max`, 玩家接受时在区间内 roll 出具体值。具体系数见代码常量。

### 4.5 融资门槛: founder 持股 ≤ 50% 拒绝

`economy.start_funding_round` 中, 若 `new_founder ≤ FOUNDER_MIN_STAKE` (= 0.5), 返回 `&"founder_stake_below_50"`。不允许「正好 50%」过关。这是兜底防止后期重融资榨干创始人股权。

### 4.6 8 轮融资 (玩家自发, 可跳轮)

游戏融资改为 **8 轮**, 完全由玩家在 UI 选择「接受」触发, **不再走事件**。每轮有独立的 conditions, **不要求按顺序接受** — 只要某轮 conditions 满足, 即可直接接 (例如开局先专注产品做到细分榜 Top3 + 有产品后, 可以跳过 pre_seed/seed 直接接 A 轮)。

| round | display | 触发意图 |
|---|---|---|
| `pre_seed` | 前种子轮 | 无门槛, 开局即可接 |
| `seed` | 种子轮 | ≥1 evaluated 模型; **entrepreneur 出身开局即满足** (见下) |
| `a` | A 轮 | 任一细分榜 Top3 + ≥1 published 模型 + ≥1 在线产品 |
| `b` | B 轮 | 任一细分榜 Top1 + 周营收过门槛 |
| `c` | C 轮 | 任一总榜 Top3 + 周营收过门槛 |
| `d` | D 轮 | 任一总榜 Top3 + 周营收过更高门槛 |
| `e` | E 轮 | 任一总榜 Top1 + 周营收过门槛 |
| `f` | F 轮 | 任一总榜 Top1 + 周营收过更高门槛 |

各轮金额区间 / 稀释区间 / 周营收门槛数值在 `resources/data/economy/funding_rounds/*.tres`, 此表只描述设计意图。**门槛口径为周营收** (游戏 1 回合 = 1 周, 订阅按周计): `周营收 = 最近 3 周营收均值` (`_avg_recent_weekly_revenue(3)`)。`unlock_summary` 文案与 UI 已统一写「周营收」(2026-05 修正: 之前误写「月营收」, 实际判定一直按周)。

**entrepreneur 出身**: `_funding_round_conditions_met(&"seed")` 在 `FounderSystem.seed_round_unlocked()` 为真时直接通过, 跳过「≥1 evaluated 模型」门槛 (见 `economy_system.gd`)。

**状态**: 接受过的轮记为 `accepted`, conditions 满足且未接受的轮为 `available`, 否则 `locked` (= conditions 未满足)。同轮重复接受返 `&"already_accepted"`; conditions 未满足返 `&"conditions_not_met"`。**无前置轮要求**, 可跳轮。

**预览 vs 执行**:
- `economy.preview_funding_rounds` 给 UI 用, 返回 8 轮全部状态 (`accepted / available / locked`)。
- `economy.start_funding_round` 真正执行, 内部再次校验 conditions (防 UI 跳过)。

### 4.7 burn / revenue 滚动窗口

`phase_started(upkeep)` 末尾:
- 把当周累计 `_current_spend / _current_revenue` push 进 `_spend_history / _revenue_history`, 截到 `burn_window_weeks / revenue_window_weeks`。
- `weekly_burn_rate = avg(_spend_history)`。
- `quarterly_revenue = sum(_revenue_history)`; 若历史不足窗口宽度, 按已知均值 × 窗口宽度外推。

环形缓冲**不写存档** (重启后从当前回合开始重建)。

### 4.8 周度账本 (财务报表)

`weekly_ledger` 在 `_on_spend / _on_award / _charge_loans` 中**同步累积**。每条 `cash_changed(delta, reason)` 落到一个**显示类目** (`category`), 由 `_categorize(reason)` 给出。映射表见代码 `REASON_CATEGORY` (UPPER 引用)。不区分经常性 / 一次性现金流: 一次性融资、贷款进账、招聘签约费、机房建设、GPU 采购、数据集购买、任务启动费等都必须入账, 经济页按类目展示收入与支出。

前缀匹配规则:
- `event:*` → 「事件」
- `campaign:*` → 「营销」
- 其他按字典直接查; 找不到归「其他」。

落账时按符号分流到 `income` / `expense` 两个类目桶, 各桶累计 `gross_in` / `gross_out`, 滚动时再补 `turn` 与 `ending_cash`。

**`phase_started(resolve)`** 触发滚动:
1. 把当周 `_current_ledger` 加上 `turn = GameState.turn` + `ending_cash = GameState.cash`, push 入 `ledger_history` (newest first)。
2. `ledger_history.size()` 截到 `LEDGER_HISTORY_WEEKS` (12)。
3. 重置当周账本为空。
4. emit `ledger_rolled(turn, snapshot)` 通知 UI。

`weekly_ledger` 与 `ledger_history` **写存档** (与 `_spend_history` 等内部环形不同 — 它们是 UI 数据, 必须跨存档保留)。经济页的“上周收支明细”只读最近完成周 `ledger_history[0]`; 若还没有完成过任何一周, 显示空态。进行中当周仍由顶栏“周净流”读取, 不在经济页明细里混入未结周数据。

**经济页展示契约 (2026-05)**: “上周收支明细”必须用表格而非自由文本列展示。收入表与支出表各自包含表头、类目列、右对齐金额列、底部小计行; 收入金额使用正向语义色, 支出金额使用危险语义色。金额符号必须使用跨平台字体都稳定覆盖的 ASCII `+` / `-`, 不用数学负号 `−`, 避免 Windows / macOS 字体 fallback 时出现方块或乱码。两张表在同一行并排, 窄宽度下仍保持各自固定最小宽, 避免类目名与金额被拉到屏幕两端。最近 12 周历史同样使用带表头的 5 列表格 (周 / 总收入 / 总支出 / 净流 / 期末现金), 净流按正负着色。顶部现金 / 负债 / 股权 / 破产压力概览使用 StatChip 风格 KPI 卡, 不再用裸 Label 堆叠。

类目清单 (raw reason → 中文类目) 的权威源是代码 `REASON_CATEGORY` 字典; 新增类目改代码、不回填本文档。

### 4.9 税务结算 (周度)

每个 `phase_started(resolve)` 在**滚动账本之前**结算一次税 (`_settle_taxes`), 扣税额记入**当周**账本 (作为支出类目) 后再滚动。resolve 内顺序: **结税 → 破产判定 → 滚动账本**, 因此 `ending_cash`、顶栏净流、破产判定都基于税后现金。

**税基 = 当周经营性净利润**, 直接从 `weekly_ledger` 计算 (读已落账数据 → 写存档安全, 中途读档不丢本周已发生利润):

```
经营性净利润 = Σ(应税收入类目) − Σ(可抵扣支出类目)
```

排除项 (融资 / 债务本金属资产负债表科目, 非损益), 由 `NON_TAXABLE_REASONS` 经 `_categorize` 映射成类目集合界定:
- 收入侧排除: 融资 (`funding_round`)、贷款进账 (`loan_taken`) — 是融资不是利润, 不计税
- 支出侧排除: 贷款还本 (`loan_payment_principal`)、主动还贷 (`loan_repaid`) — 是还债不是费用, 不可抵
- 贷款**利息** (`loan_payment_interest`) **算费用、可抵扣**
- 一次性 / 经常性**不区分**, 当周现金制: 当周发生的经营性现金进出全额计入当周 (买 GPU / 建机房当周全额抵扣, GPU 转售 / 任务退款当周全额计税)

**亏损结转** (`GameState.tax_loss_carryforward`, 写存档): 经营性净利润为负的周把亏损累加进亏损池、当周不交税; 盈利周先用亏损池抵减应税利润 (`offset = min(池, 利润)`、池随之扣减), 再按下面分档计税。让 capex 的抵扣总能用上, 削弱按周结算的运气成分 (≈ 现实年度 NOL 结转)。

**两档税** (税率 / 门槛见 `tuning.tres` §5, 默认值):
- 企业所得税: 抵减后应税利润超过免征额 (默认 100w) 的部分 × 25%
- AI 税 (UBI): 抵减后应税利润超过门槛 (默认 1B) 的部分**再** × 20% (故 1B 以上边际税率 = 45%)

两档分别记 `corporate_tax` →「企业所得税」、`ai_ubi_tax` →「AI 税(UBI)」两条支出类目, 各自 emit `cash_changed` + `resources_changed`; 财务页「上周收支明细」支出栏与 12 周历史表 (gross_out / 净额) 自动展示。AI 税名义用于发放 UBI (仅文案, 暂无再分配系统)。

## 5. 数值表 (resources/data/economy/)

数值**表驱动**, 权威源 `.tres`, `EconomySystem._load_tables()` 在 `_ready` 装配为运行时缓存 (同名 UPPERCASE 字段, 供测试与日志读取)。

| 表 | 路径 | Resource 类 |
|---|---|---|
| 全局 tuning | `tuning.tres` | `EconomyTuning` |
| 融资轮 (×8) | `funding_rounds/{pre_seed,seed,a,b,c,d,e,f}.tres` | `FundingRoundSpec` |

`EconomyTuning` 字段含义见 `economy_tuning.gd`; `starting_cash` 是 `GameState.STARTING_CASH` 的权威源, `GameState.reset()` 从此读取。**改数值改 .tres, 不要回填代码**。

`FundingRoundSpec` 新增字段:
- `display_name`: 中文展示名 (如「种子轮」)
- `unlock_summary`: UI 锁定卡上展示的解锁条件文案 (如「≥1 evaluated 模型」)
